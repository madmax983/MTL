## 1. Summary / motivation

v0.4 is the **effects release**: it turns MTL from a closed pure calculator into a language that can *do things* through a host — read input, call tools, emit output — without giving up the unconditional core theorems (P1 determinism, P2 refinement, P3 progress). Three headline decisions, each forced by the evidence below, not by taste:

- **(a) Add `Invoke` as a fourth `SpecStep` constructor** (§2, §4). The pure core suspends at every `Call(name)` and hands `(name, stack)` to an unverified host runner (spec §8.2's two-machine split). `Invoke` is **terminal-within-a-run**: the core never threads `host_state`; resumption is a fresh `spec_run`. This costs a **projected ~5 of 76** verified items and reopens *none* of the file's hard proofs.
- **(b) Strings stay host-side as opaque handles — no `Value::Str`, no string primitives in the core** (§5). The measurements are decisive: **0 of 8** agentic tasks need in-core strings, `reverse` is *cheaper* host-side (4 vs 5 tokens, zero new primitives), and `rle` is token-neutral (28 vs 31, within the ±4 band) so its decision falls to proof cost — which one-directionally favors *not* adding the constructor.
- **(c) Metering = a host-side bytes/budget meter alongside fuel, not step-weighted fuel** (§6). Fuel stays a pure in-core step counter; a second meter, debited only at `Invoke` yield points, bounds host work (calls, output bytes, time). This keeps `spec_run`'s `decreases fuel` byte-untouched.

The through-line: **the effect boundary is the trust boundary**, and the design keeps every impure concern — host state, resource caps, cancellation cleanup — strictly on the host side of a single narrow channel (the `Invoke` value), so the verified core remains a closed, total, deterministic step relation.

## 2. The `Invoke` outcome & the host-runner contract

### 2.1 What exists today

The pure core is a single closed spec function over `SpecState = { stack: Seq<SpecValue>, cont: Seq<SpecWord> }`. It closes over nothing — no dictionary, no host state, no effect trace — and its step relation has **exactly three** outcomes; `Call(name)` is a dead arm that always faults:

```rust
// mtl_core.rs — CURRENT
pub enum SpecStep { Next(SpecState), Halt(Seq<SpecValue>), Fault(Error) }

SpecWord::Call(_) => SpecStep::Fault(Error::UnknownWord),   // no host binding in core
```

There is **no `Invoke`, no `CoreStep`, no `HostResult`** in the code — they are spec-prose only (§8.2, adopted verbatim from review §3).

### 2.2 The fourth outcome

Add `Invoke` as a **fourth constructor of `SpecStep`** (not a parallel `CoreStep` enum — see §2.4). The `Call` arm stops faulting and instead **yields**:

```rust
pub enum SpecStep {
    Next(SpecState),
    Halt(Seq<SpecValue>),
    Fault(Error),
    Invoke(Seq<char>, Seq<SpecValue>, Seq<SpecWord>),   // (name, stack_snapshot, continuation)
}

SpecWord::Call(name) =>
    SpecStep::Invoke(name, s.stack, rest),   // rest == cont.subrange(1, len): the tail AFTER the Call
```

Two deliberate choices: (i) `Invoke` carries an immutable **snapshot** of the whole stack (a `Seq<SpecValue>`, not a delta) and the continuation `rest` — the tail *after* the `Call` word, so the `Call` is already consumed; (ii) the core does **not** decide whether `name` is granted. Every `Call` yields; the host runner accepts or rejects (§3). This keeps `spec_step` closed over nothing — the single property the whole proof rests on — and `Error::UnknownWord` no longer arises in-core from `Call` (grant/deny becomes a `HostFault`).

### 2.3 The two-machine loop

The exec side gains a matching driver outcome. `StepResult` and the thin driver `Outcome` each gain an `Invoke` variant, and the impure loop lives *above* the verified `run`:

```rust
// PURE — verified core (mtl_core.rs), deterministic + total up to fuel:
spec_step : SpecState -> SpecStep                  // + Invoke arm
spec_run  : (SpecState, fuel) -> SpecOutcome       // + Invoke(name, stack, cont) suspend arm
run       : (&mut Vm, fuel)  -> Outcome            // exec twin, refines spec_run

// IMPURE — host runner, UNVERIFIED, in the TCB:
enum HostResult { Resume(Vec<Value>), HostFault(HostCode) }   // host_state stays host-local

fn drive(mut vm: Vm, fuel: u64, host: &mut dyn Host) -> RunResult {
    loop {
        match run(&mut vm, fuel) {                       // pure core runs to a boundary
            Outcome::Halt(stk)      => return Done(stk),
            Outcome::Fault(e)       => return Faulted(e),
            Outcome::FuelExhausted  => return Cancelled,          // clean, BETWEEN steps
            Outcome::Invoke(name, snapshot, cont) => {            // core suspended, no effect yet
                match host.service(&name, snapshot) {            // THE EFFECT happens here (impure)
                    HostResult::Resume(result_stack) =>
                        vm = Vm { stack: result_stack, cont },   // fresh re-entry from cont
                    HostResult::HostFault(code) => return HostFaulted(code),
                }
            }
        }
    }
}
```

Semantics (spec §8.2): the core steps deterministically until it reaches `Invoke(name, stack, cont)`, suspends, and hands `(name, stack)` to the host runner; the host executes the named capability and returns a `HostResult`; the core **resumes from `cont`** with the returned stack.

### 2.4 The trust boundary

| | verified pure core | host runner |
|---|---|---|
| Artifact | `spec_step` / `spec_run` / `run` | `host.service` (arbitrary Rust) |
| State | `SpecState = (stack, cont)` only | `host_state` (files, sockets, tools, RNG…) |
| Guarantees | P1, P2, P3, total up to fuel | **none proved** — assumed to conform (§3) |
| May diverge / panic / lie? | no | yes |

The **only** channel between them is the `Invoke` value: it carries `(name, stack_snapshot, cont)` *out*, and the host returns `Resume(result_stack)` or `HostFault(code)` *in*. `host_state` **never enters the core**; `cont` is opaque to the host, which must hand it back untouched at resume. This is why the core theorems remain *unconditional statements about the core* while end-to-end statements about a running program require the host contract (review §3: "trusted or external components remain part of the trusted computing base rather than disappearing by declaration").

**Recommendation.** Extend `SpecStep` to four constructors rather than introduce a separate `CoreStep`/`CoreState` pair as §8.2's pseudo-syntax suggests. `SpecState` (stack + cont, closing over nothing) *is* the CoreState the review wants; a parallel enum would duplicate the entire `spec_step → exec_step → run` spine for no gain. **Do not thread `host_state` through the core.** This is the load-bearing choice that keeps P1/P3/P2 intact (§4).

## 3. Capability signature declarations & host conformance

A **capability** is a named host word with (a) a declared stack effect (values consumed → produced) and (b) a fault contract (the `HostFault` codes it may raise). Declarations live host-side (they are not core artifacts); the checker may later carry them as effect rows (review §19: `emit : Str -> Unit ! {output}`).

### 3.1 Declaration syntax

```
cap <name> : ( <in…> -- <out…> ) ! { <HostCode…> }
```

Four concrete examples (drawn from the agentic suite, §8):

```
cap read-state : ( -- s )                      ! { InputClosed }        -- returns a state handle
cap emit       : ( v -- )                      ! { OutputCapExceeded }  -- effect {output}
cap done?      : ( s -- s 0|1 )                ! { }                    -- total predicate, never faults
cap try-op     : ( -- r )                      ! { BudgetExhausted, ToolError, Timeout }
```

`in`/`out` are arities with (optionally) declared value shapes (`Int`, `Quote`; strings are opaque handles carried as host-owned values — §5). Linear/affine capabilities (host handles) declare their multiplicity per spec §14.2 ("at most once in all executions; exactly once on every normal terminating path").

### 3.2 Host conformance — formal statement (proof = future work)

This is the **impure-boundary analogue** of what P1/P2/P3 give the pure core. The core proofs establish nothing about the host; conformance is an *assumed* contract that *constrains* conforming hosts. Spec §14.5 names it as the **P9 host-conformance sub-judgment**; review §20 places it in **Layer F**. No formal statement exists in the sources yet — this is the statement v0.4 adopts:

> **Host conformance.** Let capability `c` have declared effect `( in_c -- out_c )` and fault set `F_c`. A host runner `H` **conforms at `c`** iff for every reachable `Invoke(c, stack, cont)`:
> 1. **(shape)** if `H` returns `Resume(stack')` then `stack'.len() == stack.len() - |in_c| + |out_c|`, the untouched prefix `stack[.. stack.len()-|in_c|]` is preserved verbatim, and each pushed value matches the declared shape of `out_c`;
> 2. **(fault containment)** if `H` returns `HostFault(e)` then `e ∈ F_c`;
> 3. **(termination)** `H` returns a `HostResult` or signals `HostFault` — it does not silently diverge (a host timeout resolves to `HostFault(Timeout)`, which must be in `F_c` if `c` may time out);
> 4. **(no leak)** `H` leaves no linear resource on `stack'` beyond its declared budget; granted linear resources are consumed or returned per the cleanup protocol (§7);
> 5. **(at-most-once service)** `H` services each yielded `Invoke` **at most once** and resumes the core with exactly the carried `cont` (§7).
>
> `H` **conforms** iff it conforms at every declared `c`. **End-to-end soundness = (P1 ∧ P2 ∧ P3, proved) ∧ (H conforms, assumed).**

Because `H` is unverified Rust, clauses 1–5 cannot be discharged in Verus; the *statement* is nonetheless the specification a host must meet to compose with the core theorems. Where a conforming host is itself written in Verus, clauses 1–2 become provable exec-side postconditions — but that is out of v0.4 scope.

## 4. Proof impact on the verified core

Baseline: **76 verified, 0 errors**, 2 intended `Clone` stubs. The key finding: **`Invoke` touches the step-relation spine, not the value model.** This is the opposite of the `Str` proposal (§5), which touches `SpecValue`/`Value`, both `Clone` impls, and every `view_*`/`spec_step_prim` arm. `Invoke` touches none of those.

### 4.1 Deep-view / termination + Clone stubs — untouched

`Invoke(name, stack, cont)` introduces **no new `Value` and no new `Word` constructor**. Its payload is a `Seq<char>` (names — already used by `SpecWord::Call`), a `Seq<SpecValue>` (a stack — already `deep_view`'d by `view_stack`), and a `Seq<SpecWord>` (a continuation — already `deep_view`'d by `view_words`). Therefore:

- **The deep-view lexicographic termination measure is not reopened.** No new mutual-recursion spine, no `decreases` obligation added. `Invoke` is a *terminal outcome*, not a recursive value — it introduces no recursion, so it decreases nothing.
- **Both `Clone` stubs are untouched** — no new value/word constructor to clone.
- **`spec_run`'s `decreases fuel` is unaffected**: `Invoke` is a base case of the recursion (terminates the run like `Halt`/`Fault`), so the fuel measure still decreases on the only recursive arm (`Next`).

This is the sharp contrast with §5: a `Str` leaf would reopen the deep-view measure and both Clone `ensures`; `Invoke` does not, because it lives entirely on the outcome/step axis.

### 4.2 The touched set (projected)

Of the **76 verified items**, the touched set is the *step-relation spine and its outcome enums*:

| touched (projected) | why |
|---|---|
| `spec_step` (`Call` arm) | yields `Invoke` instead of `Fault(UnknownWord)` |
| `p3_progress` | +1 disjunct in `ensures` (`… || is Invoke`) |
| `exec_step` | +1 `ensures` arm, new refinement obligation |
| `p2_refinement` | +1 `requires`/`ensures` clause |
| `spec_run` / `run` | +1 suspend arm (return, don't recurse) |

Plus enum edits (`SpecStep`, `StepResult`, `Outcome`, a new `HostCode`) that are not "proofs" but ripple into the above. **Projected ~5 verified items touched; ~71 untouched.** The untouched majority — all 23 `exec_prim` arm helpers, `exec_arith`/`exec_divmod`/`exec_cmp`, every `view_*`/`trunc_*` lemma, and all ~11 `smoke_*` theorems — never mention `Call` or the value model, so they are inert to this change.

- **P3 — Progress** survives: the match is still exhaustive, `spec_step` still total; adding `|| is Invoke` keeps the trivial totality discharge.
- **P1 — Determinism** survives by construction: the new `Call(name) => Invoke(name, s.stack, rest)` arm is deterministic and still non-overlapping (no `arbitrary()`/wildcard).
- **P2 — Refinement**: `exec_step` must emit `Invoke` in exactly the cases the spec does (on `Call`), with snapshot/`cont` images equal under `deep_view`. This reuses the *existing* `view_stack`/`view_words` homomorphism lemmas already proven for the `Next` splicing arms — no new lemma machinery, **projected** mechanical.

The reason all three survive is the §2.4 choice: **`Invoke` is terminal-within-a-run.** The core *stops* at `Invoke` — it does not thread `host_state`, does not loop on host results, never resumes mid-recursion. Resumption is a **fresh `spec_run`** seeded by the host's `result_stack` with continuation `cont`.

> **"5" is a statement-inspection projection, not a Verus VC count** (Verus was not run in this design round). Net: **extend `SpecStep` (4 constructors) rather than introduce a parallel `CoreStep`** — the latter duplicates the whole spec→exec→run spine, multiplying the touched set, for no gain.

## 5. Strings: in-core vs host-side — options analysis + recommendation

The question the round exists to settle: does the Tier-3 workload demand `Str` back in the core (dropped in v0.3, §3), or do host handles + capabilities suffice? Two variants are on the record.

- **Variant A — `Value::Str` + minimal string primitives.** Reintroduce the `Str` leaf and admit `str-cons`, `str-uncons`, and (for RLE) an in-core integer→decimal `num-to-str`. In-core character compute.
- **Variant B — host-side opaque handles (B2 shape).** Strings are host-owned values; the host marshals strings ↔ codepoint `Quote`s at the I/O boundary and owns rendering/tokenizing as capabilities. The core stays `Value = Int | Quote` and does the `Int`/`Quote` compute with the *existing* primitives.

### 5.1 The measured verdict

Reproduce: `python3 design-v0.4/strings/measure.py`. See [`strings/`](../../bench/design-v0.4/strings/) and [`MEASUREMENTS.md`](../../bench/design-v0.4/MEASUREMENTS.md) §1.

| program | o200k | cl100k | new core primitives |
|---|---:|---:|---|
| reverse — Python idiomatic | 10 | 10 | — |
| reverse — MTL **Var A** (fold-gen + `str-cons`) | 5 | 5 | `Str` value + `str-cons` + fold-generalisation |
| reverse — MTL **Var B2** (host codepoint-list) | **4** | **4** | **none** (= corpus `reverse_list` `[][~;](`) |
| rle — Python idiomatic | 85 | 84 | — |
| rle — MTL **Var A** (`str-cons`+`num-to-str`+concat) | 28 | 28 | `Str` value + 3 str prims |
| rle — MTL **Var B2** (host codepoint-list + host render) | **31** | **30** | **none**; rendering is a host capability |

*(reverse rows are exact hand-traces; Var A/B2 RLE are design estimates, ±4 band.)*

- **reverse**: Python 10 → Var A 5, but **host-side B2 is 4 with ZERO new primitives** — it is literally the existing `reverse_list` solution `[][~;](`. B2 wins on tokens *and* proof cost. Reversal is a structural permutation of a sequence; once a string is a codepoint `Quote`, there is nothing to add.
- **rle**: Python 85 → Var A 28 vs host-side B2 31/30 — **token-neutral, within the ±4 estimate band**, so tokens do not decide it. RLE's only genuinely algorithmic half (run-detection) needs nothing but `Int` compare over a sequence; its expensive half (rendering `"a3b2c1"`) is exactly what a host formatter already does. Split at that seam and the core stays `Int | Quote`.
- **agentic suite**: **0 of 8 tasks need in-core strings** (§8). Every task pipes opaque string *handles* through capabilities; the one "stringy" task, `word_count`, is a `fold` length once `tokenize` is a capability.

### 5.2 Recommendation: keep strings host-side

**Adopt Variant B (B2 shape). Add no `Value::Str`, no string primitives.** The tie-breaker is **proof-surface cost, which is real and one-directional** (briefing §E.4/E.5): a `Str` leaf touches both `Clone` impls + their view-preservation `ensures`, every `spec_step_prim` fault-precedence arm, the parser (`StringUnsupported` rejection), and P2 lockstep for each Str-consuming primitive — for a token payoff the benchmark says is ≈0. Contrast this explicitly with `Invoke` (§4.1): `Invoke` does **not** reopen deep-view or Clone because it adds no value constructor; `Str` *would*, because it adds a leaf across the whole value model. The direction of the proof cost points the same way as the token evidence: do not add the constructor.

**Reserve `Str`-in-core as a future option**, gated on a *measured* Tier-3 task that genuinely needs in-core string compute — admitted the way `uncons` was, on an explicit **coverage** rationale, not a compression one. `word_count` is the only near-candidate, and only under Framing 2 (if a `tokenize` capability is *refused*); the contingency glyphs (§9) are ready if that trigger ever fires.

## 6. Resource metering

Fuel is the only limit today and it is **host-blind and O(n)-blind**. Review §19 enumerates the missing model (fuel, heap quota, max quotation size, max stack depth, max output bytes, per-capability call budget, host timeout); the effect boundary makes two urgent — per-capability budgets and output-byte caps.

### 6.1 The candidates

- **(a) Per-capability call budgets.** Max `N` calls per capability per run. **Host-side.** The host runner keeps `budget : name -> u64`; each `Invoke(name, …)` debits it; at 0 the host returns `HostFault(BudgetExhausted)` instead of servicing. The pure core is oblivious.
- **(b) Output-byte caps.** Max total bytes emitted. **Host-side.** The host keeps `bytes_remaining : u64`; emit-style capabilities debit by the size they are about to write; on underflow the host returns `HostFault(OutputCapExceeded)` and writes nothing.
- **(c) The fuel-honesty gap.** `PERF-BASELINE.md` §"Fuel interaction": fuel is a **pure step counter** — `run` increments one unit per `exec_step` — but **one step can perform O(n) work**. Measured: PrimRec `sum_to(10000)` = **60,004 fuel but 223 ms** (~3,723 ns/step, ~130× the 29 ns steady-state), versus an equal-fuel steady loop at ~1.7 ms. "Fuel budgets therefore under-price quadratic programs." An `Invoke` step is the sharpest instance: it charges **1 fuel** for a host call that may emit `n` bytes or run arbitrarily long — exactly review §19's "one host call may perform arbitrary work unless separately metered."

### 6.2 Recommendation: a separate host-side meter, not step-weighted fuel

Two fixes were weighed:

- **Option A — step-cost weighting.** Make `Invoke` cost fuel proportional to output size (e.g. `1 + ceil(bytes/k)`). *Rejected*: the pure core cannot see actual host output (it arises *after* the yield, host-side), so it could only charge the *declared* cap, not the real cost; and folding host cost into fuel makes fuel depend on host results — breaking the "fuel is a closed in-core quantity" invariant and muddying `spec_run`'s `decreases fuel`.
- **Option B — separate bytes/host-work meter alongside fuel.** **Recommended.** Fuel stays a **pure in-core step counter, unchanged** (`spec_run(s, fuel)` still `decreases fuel`; the whole P2/P3 surface stays byte-untouched). A **second meter**, debited *only host-side at `Invoke` boundaries*, bounds host work (bytes, calls, time). Two orthogonal budgets: fuel bounds in-core instructions; the host meter bounds host effects.

Reasons for B: (1) it keeps `fuel` a pure core quantity, so `spec_run`'s `decreases fuel` and the P2/P3 surface stay untouched; (2) in-core steps and host bytes/time are genuinely different dimensions — a single scalar cannot price both without an arbitrary exchange rate; (3) it puts the meter *where the effect lives* (host runner), matching the trust boundary the two-machine split already draws; (4) it composes cleanly with (a) and (b), which are already host-side meters debited at the yield point. At an `Invoke` boundary the host debits the budget (a) and the bytes meter (b) and services the capability **only if both pass** — otherwise it returns `HostFault(BudgetExhausted | OutputCapExceeded)` and performs no effect.

> **Honest caveat.** Option B fixes **host-side** under-pricing only. The **in-core** O(n²) fuel-honesty gap (PrimRec re-emit, Fold tail-clone, `cont.remove(0)` front-pop) is a *separate* problem, fixed not by metering but by the §12.3 continuation-representation refactor (a persistent sequence with O(1) head-pop/prepend makes fuel a faithful in-core proxy). The bytes meter prices the host; the cont refactor prices the core. Both are needed for end-to-end honesty; neither substitutes for the other.

## 7. Cancellation semantics

Review §9: "at most once in all executions; exactly once on every normal terminating path; no linear resource on the final stack; cancellation/fuel exhaustion invokes a host-defined cleanup protocol." Spec §14.2a adopts this. Recast for effects: **a side-effecting capability is invoked at-most-once per logical step (no double-execution on resume); a normally-halting program executes each effect exactly-once.**

### 7.1 The clean-yield ordering

`Invoke` is a yield point at which the core is **paused before the host acts**. Per logical effect step the order is strictly:

```
1. core reaches Invoke(name, stack, cont)     — PURE, no effect yet; core suspended
2. host.service(name, stack)                   — THE EFFECT happens here, exactly once
3. host returns Resume(result_stack) | HostFault(code)
4. core re-enters spec_run(SpecState{stack: result_stack, cont}, fuel')   — fresh run
```

Fuel is a **core-side counter checked between steps** (`spec_run`: `fuel == 0 => FuelExhausted` *before* `spec_step`). Therefore fuel exhaustion occurs **only at a step boundary** — either *before* the core emits an `Invoke` (step 1 never runs) or *after* the core has fully re-entered from a prior `Resume` (step 4 completed). It can **never** occur "mid-capability," because the core is not running while the host acts — they are separate machines and the core is suspended across step 2. So fuel/budget exhaustion between steps cancels with **zero partial effect**: no `(name, stack)` is ever handed to the host, the capability is never invoked, and at-most-once holds trivially (invocation count 0).

A host that wants to bound its *own* service time uses the review §19 host timeout — entirely host-side: it triggers the host-defined cleanup protocol (§14.2a: return ownership / close / preserve suspended VM / leak) and resolves to `HostFault(Timeout)`. The core sees only the eventual `HostResult`; its fuel accounting is untouched.

### 7.2 The ordering guarantee behind at-most-once

At-most-once per logical step holds because:

1. **The core emits a given `Invoke` at most once.** `spec_step` is deterministic (P1) and the `Invoke` carries `cont = rest` — the continuation *after* the `Call`. On resume the core continues from `cont`, which does **not** re-include the `Call`; the yielding step is never re-executed.
2. **Resume seeds a fresh run from `cont` with the result stack**, not from the pre-`Invoke` state — no re-execution of the yielding step, hence no double-execution on resume.
3. **The host services each yielded `Invoke` at most once** (host-conformance clause 5, §3.2) and resumes with exactly the carried `cont`, forbidding host-side retry of an already-serviced (effectful) `Invoke`.

Clauses 1–2 are **provable in-core** (properties of the deterministic `spec_step`/resume shape); clause 3 is an **assumed host obligation**. Together they give at-most-once *in all executions*. **Exactly-once** is the stronger property holding *only on normal terminating paths*: if the host performs the effect (step 2) then crashes before returning `Resume`, the effect happened once but the run does not complete normally — matching review §9's split precisely, and spec §14.2a's "no linear resource on the final stack" (a normally-halting program has consumed every effect and left no live handle).

Representative programs: `agent_loop` is `read-state[done?][][step][]|` (a `linrec` fixpoint, 24→10); `retry_on_fault` is `3[try-op ok?][_][][1-]|` (bounded `linrec` on an `Int` budget, 30→12); `word_count` is `read-text tokenize 0[_1+](emit-int` (a `fold` length — **ties** 11/11 because long `Call` names cost several BPE tokens each, erasing MTL's glyph-density edge).

Agentic glue is where MTL compression is *thinnest*; the MTL case for this suite rests on **capability confinement / safety** (§8.1), not on token compression. **Net v0.4 glyph recommendation: introduce ZERO or ONE new glyph** — zero if metering stays host-side/driver-level and `Str` stays host-side (the measurement-backed default); at most one, a budget marker `` ` `` / `#`, and only if v0.4 commits to in-program metering.

## 10. Open questions / future work

- **Host-conformance proof.** §3.2 states the theorem; it is a named future obligation (P9 sub-judgment / Layer F), not discharged. It cannot be proved for arbitrary Rust hosts; a Verus-written host could discharge clauses 1–2 as exec-side postconditions — out of v0.4 scope.
- **The §12.3 continuation refactor.** The in-core O(n²) fuel-honesty gap (PrimRec re-emit, Fold tail-clone, `cont.remove(0)` front-pop) is orthogonal to §6 metering and is fixed only by a persistent-sequence continuation with O(1) head-pop/prepend. Must be done spec-first without disturbing P2; P6 (tail-call space bound) is stated but not discharged.
- **`Str`-in-core reservation trigger.** Deferred until a *measured* Tier-3 task genuinely needs in-core character compute (admitted on a coverage rationale like `uncons`). `word_count` under Framing 2 (a refused `tokenize` capability) is the only near-candidate; the §9 contingency glyphs are ready.
- **Metering surface: in-program vs host-config.** §6 recommends a host-side meter; whether budgets are ever written *in the program text* (needing the §9 budget glyph) or stay driver/host configuration (as fuel is today) is unresolved and gates the one conditional glyph.
- **Implementation ordering.** `Invoke` (§2–§4) should land as an implementation PR after P2 is fully discharged, mirroring the v0.2/v0.3 discipline: (1) `SpecStep::Invoke` arm + fault precedence preserved; (2) `smoke_invoke_*` theorems (`Call` yields `Invoke` with the correct snapshot/`cont`) + re-run P2 lockstep; (3) parser (no new GLYPHS row — capabilities are bare `Call` names); (4) exec twin + proptest oracle for the `Invoke` outcome; (5) an unverified host-runner reference `drive` loop in the TCB, plus the host-side meters (§6).
