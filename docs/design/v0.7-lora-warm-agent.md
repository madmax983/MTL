# v0.7 Design Round — A LoRA-warmed MTL agent: fine-tuning away the quickref tax

- Status: **research + experiment design (v0.7).** This is the research phase of the
  *warm-agent-trial* experiment (tracking issue filed alongside this doc). It surveys the
  method/data/eval/infra space with citations, lands on one concrete recipe, and specifies a
  falsifiable eval — but **no training was run**: this environment has no GPU. Every number
  here traces to a checked-in bench artifact, a repo measurement, or a cited URL; the
  fine-tune's own numbers are stated as *hypotheses with shown arithmetic*, not results.
- Scope is deliberately narrow. The one variable under study is **where MTL competence lives:
  in context (the ~4051-token quickref preamble, "cold") or in weights (a LoRA adapter,
  "warm").** The deliverable of the eventual experiment is a *warm* MTL agent that writes MTL
  with **zero preamble in context**, deleting the fixed cost that PR #80 proved structurally
  blocks per-task economics (N\* unreachable). This doc does **not** train a model, author the
  sealed set (#53), or pursue the white-box/soft-prompt endgame (#75). It is the black-box,
  tokenizer-facing, near-term step toward #75, and it consumes — does not re-measure — the
  published cold results.
- Related work in this repo: **#73** (ICL-preamble ablation — shrinks the quickref tax by
  *amortizing/trimming* it; this doc *deletes* it), **#53** (sealed held-out eval — supplies the
  contamination-safe measurement surface the warm headline must land on), **#75**
  (learned-tokenization endgame — white-box, blocked; the LoRA route is the black-box step that
  can run today), **#45/PR #80** (the N\* negative result this doc attacks), **#55** (glyph-side
  token optimizer, orthogonal).
- Centerpiece evidence (read it; every §1 number cites it): `bench/design-lora/RESULTS.md`,
  the per-tokenizer compression study committed to this branch (`bench/design-lora/`,
  harness `tokcompress.py`).

## §0 Motivation / So-What

The write-side case for MTL is **measured and positive**: a cold agent handed the 4051-token
quickref writes correct MTL at 100% within the N=5 repair loop on both tier-2 and tier-3, at
higher token-density than idiomatic Python — cspm ratio **1.274** on tier-2 (median tok→correct
MTL 10 vs Python 17; `bench/agent-trial/results/REPORT.md`) and **1.96** on tier-3 (median 12 vs
20; `bench/agent-trial/tier3/REPORT.md`). On the emitted-program token proxy, MTL wins.

The total-token case is **measured and negative, and structurally so.** PR #80
(`bench/agent-trial/sessions/REPORT.md`, closing #45) models the full session cost (input +
output + cache-write + cache-read, priced separately) over N ∈ {1,2,3,5,8,16} tasks and finds
**N\* = none, within or beyond the tested range, for either MTL arm.** The target N\*≤3 is not
"just past 16" — it is unreachable. The reason is a single term: the cached quickref read tax,

```
Q · cache_read_mult · p_in = 4051 tok · 0.10 · $15/M = 0.6077 ¢/task,
```

which **alone more than doubles** Python's entire per-correct cost (~0.28 ¢/task). To break
even MTL must recover that tax purely through output savings — `session_econ.py` puts the
requirement at **≈81 output tokens saved/task**, but the 18-task battery delivers only **6.96
tokens/task saved** while MTL simultaneously pays **+8.39 input tokens/task**. The gap is
>10× (11.88× `shortfall_factor_vs_measured`), every replicate has `frac C<A = 0.000`, and the
result is invariant to price scale. As PR #80 states plainly: the ~4051-token quickref tax
*structurally blocks per-task economics* at these program sizes.

**The thesis of this doc: a warm model pays zero quickref tax, so the blocker disappears.** Every
agent-trial report already scopes itself "cold-only … a warm MTL arm would pay no quickref cost
and would likely raise pass@1, so the cold numbers are a lower bound." This is the natural next
experiment, and it attacks exactly the term (Q) that drives the entire negative result. If
fine-tuning moves MTL fluency from context into weights without losing density, the arithmetic
inverts (§4): warm MTL clears Python from task #1, and N\* collapses to ~1. That is the whole
so-what.

## §1 Base model + the per-tokenizer compression table (CENTERPIECE)

A LoRA fine-tune trains on a *specific* open-weights base model, and **that model's tokenizer
decides whether MTL's compression thesis survives.** MTL's edge is that BPE tokenizers *merge*
adjacent ASCII-punctuation glyphs, so a whitespace-free program can cost **< 1 token per
primitive glyph** — a property whitespace-separated Python cannot match. That was only ever
measured on OpenAI tiktoken. `bench/design-lora/tokcompress.py` measures it directly across
**9 tokenizers** over the same 35 paired MTL↔Python programs (19 glyph-dense `core` +
16 `tier3`) that `bench/tokcount/tasks.json` encodes.

### Base-model recommendation

**Primary: `Qwen/Qwen2.5-Coder-7B-Instruct` (Apache-2.0, 32K ctx).** It **preserves MTL's edge**
(3.94× core, **0.856 tok/glyph**, 1.39 bytes/tok, 16% frag — a mid-pack, safely sub-1 tokenizer),
is the current best-in-class open 7B code model (Qwen2.5-Coder family trained on 5.5T code
tokens; 32B-Instruct is SOTA-open on EvalPlus/LiveCodeBench/BigCodeBench, 73.7 Aider), and its
Apache-2.0 license permits research **and** derivative-model release with no naming/attribution
strings — ideal for shipping an MTL-specialized checkpoint. **Fallback:
`Qwen/Qwen2.5-Coder-14B-Instruct`** (also Apache-2.0) if 7B under-*reasons* (not under-*syntaxes*)
on harder algorithmic content; second fallback `Qwen/Qwen3-14B` (Apache-2.0) buys general-reasoning
headroom (Qwen team: Qwen3-14B-Base ≈ Qwen2.5-32B-Base) at the same footprint.

**Size-tier reasoning.** For a small formal DSL, *syntax acquisition is the easy part* — SFT/PEFT
reliably teach niche/low-resource languages (YAML CI, PLC Structured Text, FOL, niche SQL). The
**binding constraint is the underlying algorithmic reasoning**, because MTL encodes algorithmic
problems. So: pick the smallest tier that still *solves* the task, then teach MTL syntax on top.
1.5B/3B can learn the surface glyphs but risk under-reasoning; 7B is the sweet spot (strong code
reasoning, QLoRA fits ~10 GB → a single RTX 4090, cheapest iteration); 14B is the reasoning
fallback (~2× train/serve cost). Run a 1.5B pipeline smoke test first to validate the data format
cheaply.

**Licensing gotchas (why Qwen 7B/14B and not the alternatives).**
- **Qwen2.5-Coder-3B is "Qwen-Research" (non-commercial)** — all *other* sizes (0.5/1.5/7/14/32B)
  are Apache-2.0. Skip 3B specifically.
- **Codestral-22B → Mistral MNPL: non-production / no commercial use** without a paid agreement.
  Do not build a shippable model on it. (Codestral-Mamba-7B *is* Apache-2.0 but Mamba/SSM tooling
  in Unsloth/vLLM is thin.) Note the compression table also flags the Mistral family as the one
  tokenizer that *fragments* MTL glyphs (1.044 tok/glyph) — a double reason to avoid it.
- **Llama 3.x → Llama Community License** (not OSI): derivative names must be prefixed "Llama",
  "Built with Llama" notice, 700M-MAU special-license trigger, EU carve-out on 3.2 multimodal;
  and it is a weaker *code* model than Qwen2.5-Coder-7B.
- **Gemma/CodeGemma → custom Gemma Terms**: redistribution notice + "modified files" notice
  required, Google reserves a remote-usage-restriction clause. (CodeGemma *tops* the compression
  table at 4.75×, but the license friction rules it out as primary.)
- **StarCoder2 → OpenRAIL-M** behavioral use-restrictions propagate to derivatives.
- **Cleanest for shipping: Qwen (Apache-2.0) and DeepSeek (permissive).**

The full study is reproducible: `bench/design-lora/` (`python3 bench/design-lora/tokcompress.py`
writes `RESULTS.md`; tokenizer files only, no weights, `torch` not required).

## §2 Method

The load-bearing fact: **MTL is genuinely new surface syntax** the base model has never seen —
the regime where PEFT is weakest — so the recipe is built around that, not around style-nudging
defaults.

**LoRA vs QLoRA vs full-FT.** LoRA (Hu et al. 2021) trains a low-rank update ΔW=(α/r)·BA on
frozen weights; QLoRA (Dettmers et al. 2023) quantizes the frozen base to 4-bit NF4 +
double-quant and trains bf16 adapters on top, reported to match 16-bit full-FT quality on its
evals while fitting a 7B finetune in ~16 GB. **The caveat that matters:** "LoRA Learns Less and
Forgets Less" (Biderman et al., TMLR 2024) shows that **on code and math** — the closest analogue
to "learn a new formal system" — LoRA *substantially underperforms* full fine-tuning, because
acquiring new capability is intrinsically **high-rank** (full-FT perturbations are 10–100× higher
rank than typical LoRA r=16–256). The flip side (LoRA *forgets less*) is a benefit for us (§ below);
the *learning gap* is the risk. **The rebuttal:** rsLoRA (Kalajdzievski 2023) shows standard
LoRA's α/r scaling causes gradient collapse at high rank, so "high rank doesn't help" is partly a
scaling artifact — with α/√r scaling LoRA keeps improving to very high ranks. So the gap is
*narrowable* by (a) high rank, (b) rsLoRA, (c) all-linear targets. **Decision:** start QLoRA
(cheapest sweep vehicle, portable adapter), but budget high-rank + all-linear + rsLoRA to counter
the "learns less" effect; keep full-FT as the escalation path (§6 ladder) if MTL pass-rate stalls.

**Hyperparameters (v1 starts).**
- **Rank r = 64** (sweep 32–128). New syntax needs high effective rank; r=8–16 is for style
  adaptation, not capability acquisition (Biderman; rsLoRA).
- **α = r (=64) with `use_rslora=True`** — α/√r scaling is what makes r=64–128 actually pay off.
- **Target modules = all-linear** (`q,k,v,o,gate,up,down`). Attention-only leaves the MLP — where
  much token-to-token computation lives — untouched, capping effective rank; all-linear converges
  faster and scores higher at any rank (Unsloth; THaLLE).
- **LR 2e-4** (Unsloth SFT default), **warmup 5–10%**, weight decay ~0.01, linear/cosine schedule,
  LoRA dropout 0–0.1.
- **Epochs 2–3** (>3 = diminishing returns + forgetting), watching a held-out MTL set *and* a
  general-code eval (HumanEval) every checkpoint.

**Vocab-extension decision: NO in v1.** MTL glyphs will otherwise be shredded by the base BPE into
sub-byte pieces — but leaving the tokenizer as-is has zero embedding-init risk, no training
instability, and keeps the adapter drop-in portable. Extending the tokenizer introduces *fresh,
untrained embeddings*, and **LoRA alone will not train the new rows** — you must additionally train
`embed_tokens` + `lm_head` (via `modules_to_save` or `trainable_token_indices`) at ~1/10th LR, and
init each new token as the **mean of its BPE sub-token pieces** (never random-init — that spikes
early loss). v1 deliberately isolates the single variable "*can LoRA/QLoRA learn MTL surface form
at all*." **The v2 middle step (toward #75):** if v1 produces valid MTL but sequences are long or
glyph errors cluster at BPE seams, add **whole-glyph-sequence tokens** — merge frequent MTL glyph
n-grams into single vocab entries, mean-init from their pieces, train embed/lm_head at reduced LR.
This is the small, auditable, reversible discrete step between raw BPE (v1) and the learned-token/
VAE endgame (#75), which is white-box and out of reach here.

**Catastrophic-forgetting mitigations.** SFT-ing a narrow terse DSL onto a general code model is a
classic forgetting setup. Mitigation stack: (1) **LoRA is inherently forgetting-resistant** — the
same Biderman paper shows it preserves base-domain performance better than full-FT (a first-class
reason to prefer it beyond cost); (2) **~10–20% general (code+instruction) replay** mixed into the
MTL SFT set (replay is ratio-sensitive; treat as a tunable, watch HumanEval); (3) **completion-only
loss** (mask the NL prompt, compute loss only on the MTL completion — `completion_only_loss=True`
in TRL), which concentrates gradient on generating MTL; (4) lower LR + ≤3 epochs. Use the base
model's native chat template so the warm agent's inference-time prompt distribution matches
training, and enable packing for throughput on many short MTL examples. Do **not** include the full
spec in training prompts (the whole point is to drop it); optionally train with a short fixed
system marker ("emit MTL") so inference matches.

**v1 recipe at a glance:** QLoRA · r=64 (sweep 32–128) · α=64 + rsLoRA · all-linear · LR 2e-4,
warmup 5–10%, 2–3 epochs · **no vocab extension** · ~10–20% general replay · completion-only loss ·
native chat template · packing · validated-canonical MTL targets.

## §3 Training data

The repo's unfair advantage is a **verified oracle** (`mtlrun`,
`bench/validate/src/bin/mtlrun.rs`; `tier3run`, `crates/mtl-host/src/bin/tier3run.rs`) that
decides correctness deterministically over a machine-checked interpreter (`mtl_core::interp`,
whose `spec_step` is Verus-proven). So we can manufacture task→program pairs at scale and admit
**only** programs the oracle printed `HALT`/`PASS` for — **correctness is ~100% by construction,
not by sampling audit.** This is execution-verified rejection sampling, the workhorse of modern
code/reasoning self-improvement (RFT, STaR, ReST/ReST-EM), except MTL's verifier is a proven
interpreter rather than a gameable test suite, so label noise is ~zero.

**The pipeline.** `generate task spec → generate candidate MTL → run the oracle → keep iff correct
→ canonicalize → dedup`. Task specs come from four sources (descending volume): (1) **procedural/
templated generators** per category in `bench/tokcount/tasks.json` (arithmetic-pipeline,
stack-shuffle — a *finite* space, enumerate it exhaustively — predicate, recursion, sequence/fold),
each emitting thousands of parametric instances; (2) **property-based structural generation**
(sample a short primitive sequence, run it under the oracle, record behavior as the io_contract —
"the program *is* the spec, discovered by execution"); (3) **mining the stack-expressible subset of
HumanEval/MBPP/LeetCode-easy** re-expressed as MTL specs (transplants human-authored phrasing);
(4) **LLM-authored task specs** (Self-/Evol-Instruct style for diversity + corner coverage — the
truncation trap `-7%3=-1`, `i64::MIN` overflow, empty-list `[]>`). Candidate *programs* come from
(a) the **strong teacher cold+quickref** — the repo already proves 100% cold pass at tiers 2/3
within N=5, so yield-per-task is high; oversample k≈8–16 to harvest multiple distinct correct
programs *and* the failed→passed repair pairs for free (RFT draws k=100 but keeps only ~13 distinct
paths/question after dedup — **dedup, not pass-rate, is the binding constraint on kept size**, and
gain is log-linear in *distinct* verified programs); (b) **mutation** of known-correct programs;
(c) **bottom-up enumeration** for ≤6–8-primitive programs, pruned by observational equivalence
(PROBE), which exhaustively covers the short-program regime with the canonical *shortest* solution
per behavior and zero teacher cost.

**Verification gate.** Tier-3 uses `tier3run <task>`, whose fixture/grants/budget/expected-output
are built internally (`mtl_host::caps::task_setup`) — the capability oracle *is* the ground truth.
Tiers 0–2 compare `mtlrun` output against a **Python twin** — but the twin **must honor MTL's
truncate-toward-zero `/`,`%`** (`-7%3 = -1`, not Python's floored `2`) or every division task
silently mislabels; bake a differential `twin(x)==mtlrun(mtl(x))` test into the generator. Verify
each program on a **grid of inputs with adversarial boundaries** (0, 1, negatives, empty list,
`i64::MIN`), not one input — this is the AlphaCode false-positive control (mutation testing cut
wrong-accept from ~62% to ~4%). **Dedup/canonicalize** with `mtl_syntax::print` — the byte-identical
production mirror of the machine-proven `exec_print` (P4) — then SHA-256 the canonical bytes;
dedup by canonical-hash *and* by io-contract behavior-vector.

**Repair-trace harvesting (~15–25% of corpus).** The interpreter's verbatim fault text is *free
supervision*: pair each failing candidate with the next candidate the oracle accepted for the same
task → `(broken, fault_message, stack_state) → fixed`. The core fault kinds (Underflow,
TypeMismatch, Overflow, DivByZero) + host codes (InputClosed, OutputCapExceeded, BudgetExhausted,
ToolError, Timeout, NotGranted) are a small closed set, so repair examples cluster into learnable
fault→fix patterns; **balance across all kinds** (over-sample rare Overflow/DivByZero). This
teaches the exact N=5 self-repair the warm agent is scored on — the mechanism behind Self-Debug,
Self-Edit (+48% rel HumanEval), Reflexion, RLEF — against a *proven* interpreter, so feedback is
better than human (Olausson et al.: repair value is gated by feedback quality). Gains concentrate
in the first 1–2 iterations, ceiling ~5 — matching the repo's N=5 loop.

**Curriculum (mostly-mixed, easy-first warmup).** T_v0 arithmetic/stack/predicate first (lock the
core glyphs + canonical whitespace), then fully shuffled tier-mixed data. Tier weighting by pair
count:

| Tier | Content | Share |
|---|---|---:|
| T_v0 | arithmetic/stack/predicate/recursion (pure core) | **40%** |
| tier-2 | sequence/fold/xor/list (compression sweet spot, 3.87×/3.92×) | **35%** |
| tier-3 | agentic/effects/capabilities (cspm 1.96; bounded by 16 `tier3run` templates + variants) | **25%** |

Repair traces layer across all three tiers (~20%), not a fourth bucket. Enforce **measured
coverage**: a meter (the repo precedent is `bench/tokcount/drift.py` + `token_profile.json`
failing loud on drift) counts all 23 glyphs and tags control-flow patterns (quotation-nesting
depth, each recursion combinator `& . |`, the null-test idiom `>[;0][[]1]?`, cons-accumulator,
dip-under-tail), and **CI fails** if any primitive/pattern falls below a floor — with dedicated
generator families for rare primitives (`| ( $ ; , '`) so they aren't starved.

**Dataset size — recommended v1 ~30k pairs.** The strongest anchor is **MultiPL-T** (execution-
verified rejection sampling for *low-resource* languages: ~40k validated examples/language roughly
*doubled* pass@1 on StarCoderBase-15B) — nearly our exact recipe. Code Alpaca (20k) and RTLCoder
(27k Verilog) bracket the low end; OpenCodeInstruct warns the marginal 10k past ~50k buys ~5 MBPP
points. MTL's low intrinsic entropy (23 single-glyph primitives, no identifiers, no strings) pulls
the estimate *down*; new-syntax/high-rank + near-zero pretraining presence (concatenative languages
sit ≪0.01% of any code corpus, so synthetic generation is *mandatory*, not mining) pull it *up*.
Net: **~10k minimum-viable / ~30k recommended v1 / ~75–100k stretch.** At **30k × ~90 tokens ≈
2.7M SFT tokens** (+10–20% replay → ~3.2–3.4M/epoch), 2–3 epochs is well under 10M tokens —
*trivially cheap to train* (§5). The expensive line item is one-time teacher inference during
generation, which is CPU-parallelizable at the verify stage.

**Contamination discipline (feeds #53).** Every current headline number is dev-split, in-sample; a
*fine-tuned* corpus is a sharper contamination risk (the model can memorize a sealed task). So:
(1) **split at the task-spec level first** — partition specs into `train`/`sealed-eval` (the
`split` field already in `tasks.json`) *before* any program generation, sealed tasks authored
independently of glyph tuning per #53; (2) **two-key mechanical dedup CI gate** — program
canonical-form SHA-256 (the proven printer defeats formatting-variant leaks) **and** task-spec
behavior-vector hash over the probe grid (catches a differently-worded train task computing the
*same function*); (3) **n-gram + MinHash backstop** (8-gram glyph overlap + 256-perm Jaccard≥0.85);
(4) a CI workflow (rhyming with the `tokreport`/`drift --check` jobs) that recomputes both hash
sets and **fails the build** if `train ∩ sealed ≠ ∅`, emitting the disjointness proof as a
checked-in artifact — "no leakage" as a machine-checked claim, not a promise. Behavior-hash is
essential because a rephraser can evade n-gram detection entirely (Yang et al.); MTL's behavior
hash is computed from *execution output*, so it catches semantic duplicates a rephrase slips past.

## §4 Eval plan

Three arms, one controlled swap (competence in context vs in weights), run through the **existing**
batteries/oracles/repair-loop/token-proxy so the delta is directly attributable to fine-tuning:

| Arm | Model | Reference in context | Status |
|---|---|---|---|
| **(A) Python** | base | none (Python is warm for the base) | published baseline |
| **(B) MTL-cold** | base + **4051-tok quickref** every task | full `docs/mtl-quickref.md` | published MTL arm |
| **(C) MTL-warm** | **LoRA'd model via vLLM** | **NONE — 0 preamble** | **new arm this doc designs** |

Arms A/B are reused verbatim from the frozen published runs. All three go through the write-side
tier-2 battery (`bench/agent-trial/`, 10 tasks, 3 trials, oracle `mtlrun`, validated by
`validate_one.py`), the write-side tier-3 battery (`bench/agent-trial/tier3/`, 8 tasks, 2 trials,
oracle `tier3run`), the **N=5 repair loop** fed the verbatim fault text, the read-side readtax
ladder (`readtax/round2/report2.py`, 73 items × A→B→C→D), and the **#80 economic model**
(`sessions/session_econ.py`, 5 price configs). Token proxy = visible emitted-program tokens under
`tokcount` o200k_base (primary) / cl100k_base. **The only thing that changes for arm C is the
inference call: the MTL system prompt carries no quickref.**

### The predicted crossover (the load-bearing arithmetic)

The #80 negative result is driven entirely by the quickref term. **The warm arm deletes it
(Q=0).** Recompute cost/correct from the same measured per-task tokens (`mean_P`/`mean_O`), default
config $15/M in, $75/M out, success ≈ 1.0:

```
Python (A):    P=88.61, O=20.06  → (88.61·15 + 20.06·75)/1e6 = 0.2834 ¢/task
MTL-warm (C):  P=97.0 (task prompt, 0 preamble), O=13.10
                              → (97.0·15 + 13.10·75)/1e6 = 0.2438 ¢/task
```

**Predicted: MTL-warm 0.244 ¢/task vs Python 0.283 ¢/task — cheaper by ~14% on task #1, with no
fixed cost to amortize ⇒ N\* → 1.** It clears Python even though MTL still pays +8.39 input
tokens/task, because at the 5:1 output:input price ratio the 6.96-token output saving is worth
~34.8 input-equiv tokens, beating the +8.39 penalty by +26.4/task. The cold arm lost this fight
*only* because the 0.6077 ¢ quickref tax (~40.5 input-equiv tokens) swamped that +26.4 margin every
task. Remove it and the margin flips positive permanently: cold-cached asymptotes ~0.85 ¢/task
(> Python forever), cold-uncached is flat ~6.3 ¢/task; **warm sits below both and below Python from
N=1.** This inversion must be *produced by the harness* (`session_econ.py` rerun → read
`results/summary.json`), not asserted.

### Acceptance checkboxes (falsifiable, instrument-named)

Write-side (primary):
- ☐ MTL-warm pass@1 tier-2 ≥ MTL-cold pass@1 (`bench/agent-trial/`, `mtlrun`, 3 trials).
- ☐ MTL-warm P(correct≤5) = 100% tier-2 and tier-3 (`validate_one.py` / `tier3run` PASS lines).
- ☐ MTL-warm median tok→correct within ±2 tokens of MTL-cold on both tiers (`tokcount o200k`).
- ☐ Warm input tokens/task = task-prompt only, **0 preamble** (assert system prompt contains no
  quickref bytes).
- ☐ Warm cspm > Python cspm on ≥1 tier — **the #80 N\* result INVERTS** (`session_econ.py` rerun).
- ☐ Warm **N\* ≤ 2** under ≥3 of 5 price configs (`sessions/pricing.json` → `summary.json`).
- ☐ Tier-3 security invariant preserved: **0 ungranted-capability-call attempts** in the warm arm
  (`tier3run` `FAIL: NotGranted` count = 0), matching cold.

Read-side (secondary):
- ☐ Warm read accuracy ≥ cold at tiers A–D for comprehension/recall/mutation (`report2.py`).
- ☐ Warm read-side input-token tax collapses to ≤ 2× the Python twin (from cold's 11–16×).

**KILL criteria (a measured miss is a publishable result, per house style, never a silent
re-tune):**
- ☒ **Fluency collapse:** MTL-warm pass@1 tier-2 < 0.7 ⇒ model cannot hold MTL without the
  reference; report the `mtlrun` fault-kind histogram.
- ☒ **Density loss:** warm tok→correct > 1.5× cold ⇒ LoRA bought correctness with verbose MTL;
  the cspm edge over Python evaporates.
- ☒ **Economic non-inversion:** warm N\* still unreachable ≤16 under default config ⇒ #80 stands
  even warm.
- ☒ **Catastrophic forgetting:** warm model's general-reasoning proxy (re-run 5 Python-arm tasks
  with the warm model) regresses >10% pass@1 vs base ⇒ the fine-tune damaged the base.

### Contamination guard (feeds #53)

Warm **headline** numbers (pass@1, cspm, N\* inversion) **must land on the sealed set** (#53:
N≥12 tier-balanced held-out tasks, hashed/no-peek, `BASELINE-SEALED.md`), with the **dev−sealed
delta** reported for every metric — a large positive delta is the signature of the LoRA having
memorized dev tasks. The training corpus must be **SHA-256 hash-disjoint** from both the sealed set
and the dev battery; a warm number on a task whose solution appears (even normalized) in training
is void. This is the single most important guard: a warm arm is *exactly* the configuration where
train/eval leakage inflates results, because competence now lives in weights where the prompt
cannot be audited. Dev-split warm numbers may appear as a secondary in-sample line, never as the
headline.

### Harness reuse (the only new code)

Reused verbatim: `bench/agent-trial/` write-side, `mtlrun`/`tier3run` oracles,
`session_econ.py` + `pricing.json`, `readtax/round2/report2.py`, `bench/tokcount/tokcount.py`. Net-
new surface is **one vLLM inference adapter + three small report patches**: (1)
`bench/agent-trial/adapters/warm_vllm.py` (same `generate(prompt)→program` contract as the cold
adapter, routes to the LoRA'd model; **asserts the assembled prompt contains no quickref substring**,
failing loud to protect the zero-preamble invariant); (2) a ~1-function `session_econ.py` patch
adding arm (D) MTL-warm with **Q=0**; (3) a ~1-branch `report2.py` warm column. **No changes to any
Rust crate, the oracles, the token counter, or the frozen cold `results.jsonl`.**

**Stack: Unsloth + QLoRA** for a solo researcher on one rented GPU (custom Triton kernels, ~2–5×

**Axolotl** YAML in reserve only if you later shard 14B across 2 GPUs (DeepSpeed/FSDP); reach for
**TRL/GRPO** only if moving from SFT to RL-from-execution-feedback. Unsloth's headline speedups are
vendor-published — treat as a strong prior, measure your own.

- **Cheap path — RTX 4090** @ ~1,500 tok/s: 75M/1500 = ~13.9 h → ~$5.6 (@$0.40) to ~$9.6 (@$0.69);

- **Fast path — H100** @ ~5,000 tok/s: 75M/5000 = ~4.2 h → ~$8.3 (RunPod) to ~$14 (Lambda) →

## §6 Recommendation

**One concrete v1 recipe:**

```
Model      Qwen/Qwen2.5-Coder-7B-Instruct (Apache-2.0)  — preserves MTL's edge: 3.94×, 0.856 tok/glyph
Method     QLoRA (NF4 + double-quant, bf16 compute), Unsloth
Vocab      NO extension in v1 (raw BPE; glyph-sequence tokens deferred to v2)
Data       ~30k oracle-validated pairs, tier mix 40/35/25, ~20% repair traces, ~10–20% general replay
Hypers     r=64 (sweep 32–128) · α=64 + rsLoRA · all-linear · LR 2e-4, warmup 5–10%, 2–3 epochs · completion-only loss
Eval gate  warm pass@1 ≥ cold, tok→correct within ±2, 0 preamble, N* ≤ 2 on ≥3/5 price configs, on the #53 sealed set
```

**Fallback escalation ladder (only on a measured miss, per the KILL criteria):**
1. **r=128 + confirm rsLoRA** (+ raise replay if HumanEval drifts).
2. **v2 glyph-sequence vocab:** add whole-glyph-sequence tokens, mean-init from BPE pieces, train
   `embed_tokens`+`lm_head` via `modules_to_save` at ~1/10th LR — the discrete middle step toward
   #75.
3. **Full-FT + replay**, accepting more forgetting, per the Biderman capability gap.

**Buildable today without a GPU (recommended immediate follow-up).** The entire *data factory* is
pure Rust/Python + the existing verified interpreter — **zero GPU, zero model access.** File a
**`bench/dataset/` (`datagen`) workspace crate**: `src/generators/` (one module per category
emitting parametric specs + Python-twin/behavior io_contracts), `src/enumerate.rs` (bottom-up
enumerator for ≤8-primitive programs, calling `mtl_core::interp::run` directly), `src/verify.rs`
(in-process oracle gate — tiers 0–2 via `interp::run` FUEL=100_000, tier-3 via
`mtl_host::driver::drive`/`task_setup`), `src/canon.rs` (`mtl_syntax::print` + SHA-256),
`src/coverage.rs` (the 23-primitive + idiom meter, failing on any floor like `drift.py --check`),
`src/bin/gen.rs` (writes `train.jsonl`/`sealed.jsonl`/`coverage.json`/`repair.jsonl` +
`contamination_report.json`). **The correctness proof is a re-validation test in the 275-test
workspace suite** (`bench/dataset/tests/`, the discipline of `bench/validate/tests/corpus.rs`):
reload every emitted `mtl_canonical`, re-run it through the oracle, assert 100% `HALT==ref`/`PASS` —
if a single row fails, `cargo test` fails, so **correctness is a machine-checked invariant of the
artifact, ~100% by construction.** A first milestone (generator + enumerate + verify + canon +
re-validation test producing ~2–5k validated pairs with a coverage report) is a self-contained PR
that **de-risks the entire fine-tune before a single training step**, and its contamination gate is
exactly the hash-disjoint machinery #53 and §4 require.

## §7 Open falsifiable questions

- **Can QLoRA r=64 all-linear reach acceptable MTL validity/pass-rate without vocab extension?**
  Falsify vs a full-FT control on held-out MTL; a large rank-saturated gap ⇒ MTL-as-new-syntax is
  high-rank-bound (predicted by Biderman) and vocab-extension / full-FT is required.
- **Does the base BPE fragment MTL glyphs badly enough to matter?** Measure tokens/glyph on the
  *trained* model and whether errors cluster at BPE seams. (The Qwen2.5-Coder tokenizer sits at
  0.856 tok/glyph *statically*, §1 — but generation-time behavior is the open question.) If yes ⇒
  v2 glyph-sequence tokens justified.
- **Optimal replay ratio.** Sweep 0/10/20/33% general data; falsify "10–20% is enough" by tracking
  HumanEval degradation vs MTL gain.
- **Does mean-init stabilize new glyph embeddings** (which may have weak BPE-piece priors)? If loss
  spikes persist, freeze base + warmup-only embedding training first.
- **Is rsLoRA's high-rank benefit real on this model/task,** or does plain α=2r suffice? Cheap A/B.
- **Does removing the in-context reference cost *read*-side accuracy at tier C/D** (H6), or did the
  LoRA teach reading as well as writing? A warm accuracy drop at C/D is the read-side signature of
  a model that leaned on the crutch.
- **Does the predicted N\*→1 inversion actually appear** when `session_econ.py` is rerun with real
  warm `mean_P`/`mean_O`, or does a success-rate loss / larger-than-modeled input penalty eat the
  +26.4 input-equiv margin?
