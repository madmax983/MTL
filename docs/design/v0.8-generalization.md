# v0.8 Design Round — Generalization: closing (and honestly re-sizing) the out-of-sample compression gap

- Status: **measurement + admission decision (v0.8).** This round takes the first
  out-of-sample MTL result (PR #91, `bench/BASELINE-SEALED.md`) as its starting fact and
  asks one question: **can the ≥3× static-compression gate be recovered on unseen task
  mixes, with proper train/dev/seal hygiene — and if not, what is the honest ceiling?**
  Every number here traces to a committed bench artifact. Nothing in the frozen language
  (`mtl-core` / `mtl-syntax` / `mtl-check` proven modules) was changed; the diff is
  confined to `bench/` and this doc.
- The deliverable is a decision, not a bar cleared: **admit nothing; restate the thesis
  at the measured out-of-sample ceiling ≈ 1.7×**, with a single conditional fallback (a
  verified windowed-fold aperture) reserved for a re-scoped scan-heavy niche. The round
  also **specifies** (does not execute) the fresh re-seal protocol the next round runs.
- Related work: **#53 / PR #91** (the sealed held-out eval this round diagnoses),
  **#41** (the "juggling-bound not access-bound" wall, tested here against windowed
  scans), **#50** (literal-operand gate for parameterized primitives), **#55/#80**
  (glyph-side and economic accounting), **v0.6 checker** (`docs/design/v0.6-checker.md`,
  the proven-fragment typeability standard any candidate must meet).
- Centerpiece artifacts (read them; every §number cites them):
  `bench/design-v0.8/BROAD-DISTRIBUTION.md` (the generator-based train/dev baseline),
  `bench/design-v0.8/candidates/CANDIDATES.md` (the measured admission analysis),
  `bench/BASELINE-SEALED.md` (the PR #91 held-out result being generalized).

## §0 Framing — the round is GENERALIZATION

The motivating fact is a measured miss. Static compression is **3.87×** in-sample
(dev tier-2, `bench/BASELINE-TIER2.md`) but **1.67× (o200k) / 1.72× (cl100k)** on the
blind sealed set (PR #91, `bench/BASELINE-SEALED.md` §3, the 14 algorithmically-correct
tasks, dev-parity constructed-stack validation). The ≥3× Abrash gate **fails
out-of-sample** — and it fails structurally, not by noise: the dev corpora co-evolved
with primitive admission (the cheap dev wins lean on `(`/`$`, admitted under dev-corpus
token pressure), so the 3.87× was substantially a benchmark-fitting artifact the blind
set was built to expose.

Two things did **not** fall out-of-sample, and they anchor the restated thesis:

- **Agent writability holds.** Cold-agent success is **100% pass@5 (MTL) = 100% pass@5
  (Python)** on the 7 text-feedable sealed tasks — equal to dev
  (`bench/BASELINE-SEALED.md` §4).
- **Marginal per-solution economics hold — and widen.** The CSPM ratio (correct
  solutions per million output tokens, MTL/Python, charging failed repairs, excluding
  the one-time quickref) *rose* out-of-sample from **1.274 (dev) to 2.124 (sealed)**.
  MTL's median output is 19 tok/solution vs Python's 57.

So the picture out-of-sample is: **per-solution economics widen, static compression
collapses.** This round closes the static gap for real — measures where it actually
sits on a disciplined train/dev distribution, decides what (if anything) to admit, and
re-seals. The thesis it lands on is not "the gate was almost met"; it is "the gate was
the wrong claim, and here is the right one, measured."

## §1 The sink taxonomy (headline) — where the 3× budget is spent

Source: `bench/design-v0.8/candidates/CANDIDATES.md` and the per-task diagnosis it
builds on. All token counts o200k, **independently re-derived with tiktoken** (one
trailing newline stripped) and matching `bench/sealed/results/static_tokens.json`
exactly.

To clear the 3× gate at the fixed idiomatic-Python sum (533 tok over the 14 correct
tasks), MTL must be ≤ 177.7 tok. It is **319**. The **excess over the 3× budget is
141.3 tokens (py=533 / mtl=319, o200k)** — this is the quantity the taxonomy
apportions, attributing each task's excess to its dominant sink.

| sink category | tasks | excess tok | % of +excess |
|---|---|--:|--:|
| **(a) stack-juggling tax** (sequence/state scans) | count_local_maxima 23.0, max_adjacent_diff 27.0, rle_flatten 27.7, dedup_adjacent 7.7, ½ min_running_balance 4.2 | **89.6** | **≈59%** |
| **(b) missing-idiom / anti-niche** (terse Python builtin) | count_set_bits 16.7, digit_product 18.3 | **35.0** | **≈23%** |
| **(c) accumulator-shape** (multi-acc threading) | digit_sum_base 10.7, ½ min_running_balance 4.1 | **14.8** | **≈10%** |
| **(d) residual** (honest near-3× loop tasks) | collatz 5.7, num_divisors 3.3, int_sqrt 2.0, triangular 0.3 | **11.3** | **≈8%** |
| credits (beat 3×) | alternating_sum −4.0, xor_reduce −5.3 | −9.3 | — |
| **net excess over 3× budget** | | **≈141.3** | 100% |

**Stack-juggling dominates (≈59%).** The five sequence/state-machine tasks
(local_maxima, max_adjacent_diff, rle_flatten, dedup, min_running_balance) hold **78 of
the 131 total juggle glyphs (60%)** — independently the same ≈60% share the
excess-attribution assigns. A concatenative stack has no sliding window and no named
running state, so every window is rebuilt each step with `~^_@'` chains (20+ glyphs on
local_maxima and rle_flatten) that Python names for free.

**Missing-idiom / anti-niche (≈23%)** is `count_set_bits` (0.73× — the one task MTL
loses outright, vs `bin(abs(n)).count("1")`) and `digit_product` (1.04×, vs
`math.prod(map(int,str(abs(n))))`). This is structural for a minimal concatenative core.

### Correcting the post-mortem's input-encoding premise

`bench/BASELINE-SEALED.md` §6/§7 flags negative-integer input encoding as a gap. **It is
not a compression sink.** Precisely:

- The committed **1.67× is already computed under constructed-stack validation**
  (`bench/validate/tests/sealed.rs`), which feeds real `Value::Int` / `int_list`
  negatives directly, bypassing the text lexer. Negatives are already presented fairly.
- `input_encoding(scalar|list)` is a **text-feedability LABEL** — whether the text
  `mtlrun` harness can *lex* a task's inputs (`-5` lexes as the `Sub` primitive, there
  being no negative literal). It touches **zero solution tokens.** **Fixing the
  `-5`-lexes-as-`Sub` lexer changes the static ratio by 0.**
- Checking every solution for input-encoding workaround tokens: the only candidate is
  the `0~-` abs idiom, but each such task's **spec demands `abs`** and Python pays for
  it too (`abs(n)`). Zero solution tokens are input-encoding artifacts.

What fixing the lexer *actually* does is move the 7 currently-non-feedable correct tasks
into the feedable set, whose sub-aggregate (py 259 / mtl 188 = **1.38×**) is *lower*
than the 7 already-feedable (274/131 = **2.09×**). So a corrected text harness would
**drop the text-feedable number from 2.09× to 1.67×** — it removes a favorable selection
bias; it does not help MTL.

### The fair-terse-Python comparison

The single largest lever on the headline is not on the MTL side: it is the choice of
*verbose idiomatic* Python (a `def` header + explicit loops + named accumulators) as the
reference. Against **minified-but-fair** Python — lambda one-liners using stock builtins
(`math.isqrt`, `math.prod`, `bin().count`, `functools.reduce`), all readable idioms, not
code-golf — the same 14 solutions give:

> **py 330 / mtl 319 = 1.034× (o200k). MTL barely compresses at all.**

Task-level proof the "win" is largely Python verbosity: `int_sqrt` 15 vs `math.isqrt(n)`
15 = **1.00×**; `triangular` 6 vs 11 = 1.83× (was 2.83×); `num_divisors` 18 vs 22 =
1.22× (was 2.44×). The honest fair-Python band is **~1.0–1.7×** depending on how terse a
Python baseline you accept.

## §2 Broad-distribution baseline — the generator-based replacement for the spent corpus

Source: `bench/design-v0.8/BROAD-DISTRIBUTION.md`, emitted by
`bench/dataset/src/families.rs` (oracle-verified family generators) and
`bench/dataset/src/bin/broad.rs`; results in `bench/design-v0.8/broad_results.json`.

The dev corpus is *spent* (co-evolved with admission). Its replacement is a
generator-based, train/dev-split distribution of **1145 distinct deduped shapes**,
oracle-verified by construction: each instance ships an adversarial i/o contract (0,
±1, MIN/MAX where safe, empty/singleton/run/negative/alternating lists) checked
vector-by-vector against a Rust `checked_*` reference through the Verus-proven
`mtl_core::interp` gate. A candidate enters only if it HALTs to the exact reference
stack (or faults where the reference faults) on every vector.

**New families cover the sealed-flagged uncovered shapes**, seeded verbatim from working
held-out solutions and re-verified against a fresh non-sealed input grid:

- **`bitdigit`** (scalar): `popcount` (base-2 digit sum), `digit_sum_base` for
  b ∈ {2,3,4,5,8,10,12,16}, `digit_product_base` for b ∈ {8,10,16}.
- **`scan`** (list): `alt_sum`, `count_local_maxima`, `max_adjacent_diff`,
  `dedup_adjacent`, `rle_flatten`, `min_running_balance`.

All honor the decision record: **no `Value::Vec`, no `Value::Str`** — lists stay the
existing tier-2 quote-of-ints; scans use `(` fold, `|` linrec, `;` cons, `>` uncons,
`'` dip.

**Split convention.** Each distinct program goes to TRAIN or DEV by **the low bit of the
first hex nibble of its canonical SHA-256** (even → TRAIN, odd → DEV). This is
documented deliberately: raw **seed parity would mis-split** the scan/bitdigit programs,
which are **seed-invariant**, sending every one of them to a single split; sha-parity
gives both sides coverage of every family. The generator is swept across seeds 0–5 for
the seed-parameterized families; distinct-by-canonical dedup yields the 1145 shapes.

### Numbers (o200k, token-SUM)

| aggregate | TRAIN | DEV |
|---|--:|--:|
| **NEW uncovered shapes (scan + bitdigit)** | **1.70×** | **1.72×** |
| full distribution, micro token-SUM | 3.25× | 3.24× |

The **~1.7× on the newly-covered shapes reproduces the sealed 1.67×** out-of-sample —
strong cross-harness agreement. The full-distribution **3.25× is an artifact**: **968 of
1145 shapes are near-identical `a*n+b` affine one-liners**, so the arithmetic family
(which the language was shaped around) swamps the token-SUM. It is not a generalization
number.

**Per-shape ratios cross-validate against the sealed set exactly** (the scan templates
are byte-identical to the sealed `python-idiomatic` references):

| shape | v0.8 broad | sealed (BASELINE-SEALED) |
|---|--:|--:|
| alternating_sum | 4.33× | 4.33× |
| dedup_adjacent | 1.85× | 1.85× |
| min_running_balance | 1.86× | 1.86× |
| rle_flatten | 1.16× | 1.16× |
| max_adjacent_diff | 1.07× | 1.07× |
| count_local_maxima | 1.40× | 1.40× |
| popcount / count_set_bits | 0.68× | 0.73× |

**TRAIN and DEV agree to ≤0.02× on every aggregate** (1.70 vs 1.72; 3.25 vs 3.24) — a
hash-parity split over a family-balanced generator has **no distribution shift**, so DEV
is a faithful held-out check on anything optimized against TRAIN.

**Expressibility finding.** The scalar reductions max/min are covered by `fold`, but the
*windowed running-extremum that emits a list* (`out[i] = max(xs[0..=i])`) was **not
reduced to a verified program** — it needs a 3-element fold state (running extremum *and*
a growing output accumulator) threaded point-free, and no `Value::Vec` makes that
accumulator ergonomic. This is a genuine edge-of-language finding (and the sealed
`seal_running_max` hand-attempt was itself algorithmically wrong — seeded at 0, failing
all-negative input), not an omission. Deliberately not added: this round measures the
*current* language.

## §3 Candidates — measured on TRAIN, validated on DEV

Source: `bench/design-v0.8/candidates/CANDIDATES.md`. Every rewrite is tokenized with
the same o200k encoder as the harness, and the aperture rewrites are
**simulator-verified against the datagen oracle** (`verify_aperture.py`), not
hand-traced — the design record's repeated lesson is "no hand-traced number survives
contact with the interpreter."

The addressable sink is stack-juggling (≈59%). Of its five scan/state tasks, two are
**windowed** (fixed-width sliding look at consecutive elements — `count_local_maxima`
3-window, `max_adjacent_diff` 2-window) and three are **stateful/accumulator**
(history-dependent — `rle_flatten`, `min_running_balance`, `dedup_adjacent`). **Only the
windowed kind is addressable by an aperture combinator.** This is the load-bearing
finding.

### Candidate (a) — windowed-fold aperture `w`

**spec_step semantics** (`[xs] acc0 k [C] w`, k a **literal** per #50's operand gate):
`fold`'s rule generalized from a 1-element step to a k-element sliding window — for
`i` in `0 ..= len(xs)-k`, run `C` on stack `[acc, xs[i], …, xs[i+k-1]] -> [acc]`; lists
shorter than k contribute no window (result is `acc0`).

**Measured token delta (o200k / cl100k), simulator-verified, PASS the oracle grid:**

| task | split | baseline | aperture (param-k) | juggle glyphs |
|---|---|--:|--:|--:|
| `count_local_maxima` | **TRAIN** | 43 / 38 | `0 3[^<@@<*+]w` = **10 / 9** | **20 → 3** |
| `max_adjacent_diff` | **DEV** | 42 / 39 | `0 2[-:0<[0~-][]?^^<[~_][_]?]w` = **19 / 17** | **21 → 7** |

The aperture **removes** the windowing juggle (does not relocate it): juggle glyphs fall
**20 → 3** and **21 → 7**, the residual being genuine intra-window comparison Python also
pays for. **This DISPROVES #41 for windowed scans**: #41 ("juggling-bound not
access-bound") concerned random access into a buried carried state, where deep
pick-chains dominate regardless of primitive. Sequential windowed scans are structurally
different — the combinator *supplies* the consecutive elements, eliminating the
uncons-triple + re-thread. **#41's wall still binds the stateful tasks** (`rle_flatten`
history-dependent run state, `min_running_balance` twin accumulators) — these are
**unchanged**.

**Checker typeability.** Per `docs/design/v0.6-checker.md`, `w` types **exactly like
`fold`**: literal-k makes window arity statically known and `C : [acc:Int]++[Int;k] ->
[acc:Int]` checks for height/type stability; over an opaque runtime-length list the
verdict is **Guarded**, the *same class* as the existing fold/linrec scan programs. **No
proven-fragment shrink** (Guarded → Guarded). The cost is the full primitive-mirror
fan-out (#32/#41/#50): a fold-analogous effect rule, `spec_step_prim`/`exec_prim` arms,
7 mirror surfaces, manifest 23→24, P2 lemmas, a smoke theorem.

**Glyph-BPE + quickref cold-cost delta.** A new quickref line is ~39 tok (terse) to
~97 tok (worked-example style). Against the 4051-tok cold preamble (break-even ~154
tasks), it pushes break-even up ~1.5–3.7 tasks. Small in isolation — **but the
amortization is decisive**: windowed shapes are **2 of 1145 (0.17%)** of the broad mix,
saving ~60 tok total, so the cold + full proof cost is **not amortized** unless the
distribution is scan/window-heavy. Param-k `w` (1 glyph, 1 line) is cold-preferable to a
fixed-width `#`/`` ` `` pair (2 glyphs) despite +2 tok per use.

**Projected broad-distribution ratio (new families = scan + bitdigit, o200k):**

| variant | TRAIN | DEV |
|---|--:|--:|
| baseline (current language) | 380/223 = **1.70×** | 375/218 = **1.72×** |
| param-k aperture `w` | 380/190 = **2.00×** | 375/195 = **1.92×** |

Held-out validation holds: `w` was designed/verified on `local_maxima` (TRAIN) and
transfers to `max_adjacent_diff` (DEV, held out). It is a genuine combinator (any
sliding-window computation), more general than the `$`-xor overfit but **narrower than
`fold`/`linrec`**. It still lands at **~2×, not 3×** — the gate is not recovered.

### Candidate (b) — idiom / doc fix

Documenting the `abs`/accumulator/scan idioms in the quickref: **measured compression
effect = 0.** The corpus solutions already *use* the optimal idioms (they are
oracle-golfed), so no token count changes. This is **writability-only** (a live model may
reach the optimum faster) and it **grows the quickref**, so its cold-economics effect is
slightly negative. Worth doing for generation reliability; contributes 0 to the
compression thesis.

### Candidate (c) — `popcount` builtin

`popcount` costs 22 tok today vs `bin(abs(n)).count("1")` = 15 (0.68×, the one MTL loss).
A `popcount` glyph → 1 tok = 15× **on that one base-2 shape**. But it fires on **exactly
one shape**: `digit_sum_base` bases 3–16 and `digit_product_base` are base-parameterized
and unaffected. Broad impact: 22→1 on **1 of 1145 shapes** — negligible on the aggregate
— while spending a scarce glyph + quickref line + checker arm + proof fan-out.
**Textbook benchmark-fit — reject.**

## §4 Recommendation — ADMIT NOTHING, ceiling ~1.7×

**Primary recommendation: admit nothing; restate the thesis at the measured
out-of-sample ceiling ≈ 1.7×** (new families TRAIN 1.70× / DEV 1.72×, reproducing sealed
1.67×; the full-distribution ~3.2× is an arithmetic-domination artifact, not
generalization). Against fair terse Python the ceiling is **~1.0–1.3×**.

Rationale — the aperture `w` is the only real lever, and it does not pay:

1. It lifts **2 of 1145** shapes, to **~2×, still short of 3×**.
2. It addresses **~35%** of the juggling sink and **0%** of the state-machine /
   accumulator / missing-idiom sinks (`rle_flatten`, `min_running_balance`,
   `count_set_bits`, `digit_product` are structurally untouched).
3. On a broad mix its **cold-quickref + full primitive-mirror/proof cost is not
   amortized** by 0.17%-frequency windowed shapes.
4. Doc-fixes contribute **0** compression; `popcount` is benchmark-fit with ~0 broad
   impact.

**The restated thesis, precisely.** MTL's out-of-sample value is **agent WRITABILITY**
(100% pass@5, equal to Python) and **marginal per-solution economics** (CSPM ratio
2.124×, widened from dev's 1.274×) — **NOT ≥3× static compression on blind mixes.**
Static compression is a **niche property**: 2–4× on loop/fold/recursion shapes
(alternating_sum 4.33×, collatz 2.26×, int_sqrt 2.60×), ≤1× on scan/builtin shapes
(count_set_bits 0.73×, max_adjacent_diff 1.07×). The ≥3× Abrash gate is retired as a
generalization claim.

**Conditional fallback (only if the maintainer re-scopes to a scan/window-heavy
niche):** admit the **parameterized windowed-fold `w`** (literal-k, #50-gated) — the one
candidate that is verified, generalizing (held-out DEV transfer), and checker-typeable
(Guarded, fold-class, no proven-fragment shrink). It projects the new-families aggregate
to **~2.0× out-of-sample** (scan family TRAIN 1.62×→2.65×, DEV 1.42×→1.86×). Prefer
param-k (1 glyph) over fixed-width (2 glyphs) for cold economics. **Even then the result
is ~2×, not 3×.**

## §5 RE-SEAL PROTOCOL (specified, NOT executed this round)

The next round executes this; this round only specifies it. PR #91 spent its sealed set
by committing reference solutions (`bench/BASELINE-SEALED.md` §9a), so a **fresh** blind
set is required. The protocol encodes the #91 lessons:

**(i) Information barrier authored AFTER the admission set freezes.** This round freezes
to **"no admissions"** — the frozen language is byte-identical to the pre-round language
— so the barrier between admission and authorship is **trivially clean**: there is no
new primitive whose motivation could leak into task selection. State this in the fresh
`AUTHORSHIP.md`.

**(ii) NO reference solutions committed.** The #91 mistake was committing the 14
solutions, spending the set (`bench/BASELINE-SEALED.md` §9a). This time, commit **only**
the task specs, the salted contamination manifest, and the freeze record. Solutions are
authored, measured, and reported **without being committed** (or committed only to a
separate spent-marker artifact after the one-time unseal), so the set stays reusable for
a subsequent held-out check.

**(iii) Author against the broad-distribution shape catalogue with fresh, unseen
parameterizations, balanced across the sink taxonomy.** Draw shapes from the
`bench/dataset/src/families.rs` catalogue (arithmetic, quotation, predicate, recursion,
fold, **scan**, **bitdigit**) but with parameterizations that do not appear in
`bench/design-v0.8/broad_shapes.json` (fresh literals, fresh bases, fresh list grids).
**Balance across the §1 sink taxonomy** — the seal must include **scan/state-machine
tasks (the anti-niche)** so it measures the honest mix, not a loop-heavy cherry-pick that
would flatter MTL. Target composition mirrors the taxonomy: ~40% loop/fold/recursion
(MTL's niche), ~40% scan/state (the anti-niche), ~20% scalar builtin / accumulator.

**(iv) Gate re-run procedure:**
1. **Freeze** — fix the sealed specs + salted manifest at a seal commit; declare a
   measurement-freeze commit; tag it (rhyming with `sealed-freeze-issue-53`).
2. **Author sealed specs behind the barrier** — task semantics only, no spec/quickref/
   corpus/`.mtl` access, per `bench/sealed/AUTHORSHIP.md` discipline.
3. **Agent cold-trial (pass@5) + static compression measured by a party who never saw
   the solutions** — the measurement party is distinct from the solution author, so the
   held-out static ratio is computed without solution-authoring bias.
4. **Publish the held-out ratio** and compare it to the **restated ~1.7× thesis band**,
   **NOT** the retired 3× gate.

**Salted-contamination manifest step.** Reuse `bench/dataset/src/bin/mkseal.rs` to emit
the salted manifest (schema `mtl-sealed-manifest/v2`, a fresh salt string, per-entry
`content_sha256` + `io_hash`, `canonical_sha256` withheld pre-freeze), and re-run the
`bench/dataset/tests/contamination.rs` gate: `sealed_disjoint_from_dev` (extended to also
prove disjointness from the v0.8 broad-distribution shapes),
`planted_dev_task_collision_is_caught` (the gate bites), `manifest_matches_sealed_tasks`
(manifest reproducible from the specs).

**Pass/fail definition.** The round **PASSES** iff out-of-sample static compression **≥
the restated thesis band (~1.7×)** AND **pass@5 ≥ Python**. It **FAILS the old ≥3×
framing explicitly** — the 3× gate is recorded as retired, and a number in the
~1.7× band is a *pass*, not a miss, because the deliverable is the trustworthy held-out
number against the *correct* claim.

## §6 Honest read

**Is ≥3× out-of-sample compression achievable?** No — not on blind mixes of this
composition. The measured ceiling is **~1.7× vs idiomatic Python** (new families TRAIN
1.70× / DEV 1.72×, reproducing the sealed 1.67×) and **~1.0–1.3× vs terse-but-fair
Python**. The dev 3.87× did not generalize; it was arithmetic- and primitive-fit, a
co-evolution artifact the blind set exposed. The full-distribution ~3.2× is not a
counterexample — it is 968/1145 trivial affine one-liners padding the token-SUM.

**What DID generalize:** agent **writability** (100% pass@5, equal to Python) and
**marginal per-solution economics** (CSPM ratio 1.274 → 2.124, widened out-of-sample).
The defensible claim is "MTL is agent-writable and marginally tighter per solution," and
that claim held or improved on unseen tasks.

**What is structural:** stack-juggling on scan/state tasks is **intrinsic to the
concatenative paradigm**, not a missing primitive. A stack has no sliding window and no
named multi-field running state; the windowed *sub-shape* is addressable by an aperture
combinator (verified, ~2×), but the *stateful* sub-shape (`rle_flatten`,
`min_running_balance`) hits the #41 wall regardless of any single primitive, and the
single-builtin anti-niche (`count_set_bits`, `digit_product`) cannot be won without
turning the language into a builtin library. The ≥3× gate is not recoverable on a broad
mix without either narrowing the distribution to MTL's fold/loop niche or re-introducing
benchmark-fit primitives — both of which the round declines.
