# MTL v0.5 — Arena execution backend + host-layer speculation (design stage, measurement-driven)

- Status: **design stage**. This round designs an **arena execution backend** to kill the measured O(n²) hot-loop pathologies, plus a **host-layer speculative-execution** design over cloned VM states. MTL's semantics are **FROZEN**: `spec_step` and the Verus ghost model are untouched, no new primitives are introduced, and the reference interpreter (`crates/mtl-core/src/interp.rs`) remains the canonical **twin**. The arena is a **differentially-validated backend, not a proved one** this round: it is validated only by a differential oracle against `interp.rs` (**47/47** corpus programs bit-identical), an extension of the conformance crate, and a **stated (not executed)** P2-style refinement obligation. No core code is changed in this PR.
- Branch: `v05-design` (based on `origin/main`). The frozen corpus, `bench/tokcount/tasks.json`, `bench/BASELINE*.md`, and all existing `crates/` are untouched. This round adds a **non-production** spike crate `crates/mtl-arena-spike` (a measurement vehicle, not a shipping backend) and its measurements under `bench/design-v0.5/`.

## 1. Summary / motivation

The refactor is justified by **measured pathology, not intuition**. Today the reference interpreter's continuation is a `Vec<Word>` used as a queue, and two O(n) mechanisms on the hot path compound into O(n²): front-pop (`cont.remove(0)`) and prepend-splice (`cont := prefix ++ cont`). These make flat programs, PrimRec (`sum_to`), and Fold quadratic; `sum_to(10000)` costs **223 ms**. This round designs an arena backend whose persistent **segment-cursor continuation** makes front-pop and prepend O(1) — killing all three pathologies and making VM **fork O(1)** — and a host-layer speculation design that scales search *width* over cheaply-cloned VM states without touching the proved core.

Two framings run through the whole document, and we hold to both:

- **Semantics stay frozen; `interp.rs` stays the reference twin.** The arena computes exactly the frozen semantics by a faster mechanism. It is pinned to those semantics by the same discipline as every other component — a differential oracle, a conformance extension, and a stated P2 refinement plan. Nothing here relaxes any existing invariant.
- **This is a design + measured spike, not a production backend.** The spike exists to *measure* whether the arena earns implementation. The go/no-go is the recommendation in §6, not a claim that it ships.

## 2. The measured problem

All numbers in this section are cited from `crates/mtl-perf/PERF-BASELINE.md`.

Today the continuation is a `Vec<Word>` used as a queue, with the head at `cont[0]`. Two
distinct O(n) mechanisms sit on the hot path and compound into O(n²):

- **Front-pop.** Every step does `cont.remove(0)`, which shifts the entire remaining vector
  left by one — O(n) per step. A flat program (`1 1 + _` repeated N times) pays this on every
  single step. Measured: ns/step grew **414×** across a 256× increase in step count; at 16384
  steps the machine ran at **5057 ns/step** versus the **29 ns/step** steady-state ceiling —
  a ~175× slowdown attributable purely to front-pop shifting.

- **Prepend splice.** The recursion primitives do `cont := prefix ++ cont`, an O(n) splice.
  This produces two separate O(n²) pathologies:
  - **PrimRec re-emission.** The combinator re-emits its body `C` *after* the recursive
    marker at each level, so the continuation grows by `|C|` per level. `sum_to(10000)` costs
    **223 ms** (3723 ns/step, peak cont length 10005); `sum_to(100000)` is projected at
    **~22 s**. This is THE worst case.
  - **Fold spine-clone.** Fold re-emits `PushQuote(tail)` for each element, deep-cloning the
    shrinking list spine. The cost is hidden *inside a single `Word`*, so it is invisible to a
    continuation-length metric, but it is still O(n²): a 10k-element fold costs **~32 ms**.

One correction to the frozen spec's own hypothesis: spec §12.3 named the `: !` self-application
splice as the suspected culprit. **It is not pathological.** That splice is tail-recursive in
the continuation — each `Apply` splices onto an empty tail, so the continuation stays 6 words
deep even at recursion depth 10000. It is **linear**, and must not be "fixed." The actionable
targets are exactly three: `cont.remove(0)`, PrimRec re-emission, and Fold spine-clone.

Finally, an honesty note that scopes the entire effort: **every corpus solution already runs
in microseconds**, at ≤115 steps and ≤15 continuation words. The refactor changes nothing for
them. It matters *only* for large-data workloads in the 1k–100k range — the speculation and
search use case. This backend is an investment in that workload, and we say so plainly rather
than dressing it up as a universal win.

## 3. Arena backend design

The arena backend is a **new execution backend** — `crates/mtl-arena-spike` in this design round, `crates/mtl-arena` in production — that computes exactly the frozen semantics by a faster mechanism. It is pinned to those semantics by the same discipline as every other component: a differential oracle against `interp.rs`, an extension of the conformance crate, and a stated (not yet executed) P2-style refinement plan. Nothing here relaxes any existing invariant.

This section specifies the arena representation, generational memory management, the host
boundary, the addressing limits, and how fault-carries-state survives the change. Section 5
(**Drift accounting**) closes it out, because the architecture review named semantic drift as
the #1 risk, and a new backend adds mirrors of the primitive set that must be policed rather
than allowed to rot.

### 1. Arena representation

The core idea is to replace owned, cloned, shifted `Vec`s with **arenas of `Copy` handles**.
Everything on the hot path becomes a small index; state forks become a ~12-byte struct copy.

#### QuoteArena

All quote bodies are interned **once** into a single flat tape:

```rust
struct QuoteArena { tape: Vec<Word> }

#[derive(Clone, Copy)]
enum Word {
    PushInt(i64),
    PushQuote(QuoteId),
    Prim(Prim),
    Call(u32),        // interned capability-name id, NOT a String
}

#[derive(Clone, Copy)]
struct QuoteId { start: u32, len: u32 }
```

`alloc(&[Word])` extends the tape and returns a `QuoteId` covering the new range. `cat(a, b)`
appends copies of both ranges and returns a fresh `QuoteId`. Nested quotes are interned
recursively: a `PushQuote` holds a `QuoteId` pointing into the *same* tape, so the whole quote
forest is one contiguous allocation with no per-node boxing.

Note the deliberate divergence from `interp.rs`: the interpreter uses `Call(String)`, whereas
the arena uses `Call(u32)` over an interned capability-name table. This is a representation
mirror that the conformance crate must explicitly police — it is listed in the Drift accounting
section below.

#### Value

```rust
#[derive(Clone, Copy)]
enum Value { Int(i64), Quote(QuoteId) }
```

`Value` is small and trivially `Copy`. An `Int` carries a full `i64` payload; a `Quote` is
just a 8-byte handle into the tape.

#### StackArena (persistent)

```rust
struct StackArena { nodes: Vec<StackNode> }
struct StackNode { value: Value, parent: StackPtr }   // index 0 = empty sentinel
type StackPtr = u32;
```

The stack is a persistent cons-list. `push` allocates a node and returns its `StackPtr`; `pop`
follows `parent`. Because nodes are immutable once allocated, distinct branches of a
speculative search **share structure** — forking a stack costs nothing but copying the
`StackPtr`.

#### Continuation — segment cons-list + local cursor (the crux)

The oracle sketch proposed representing the continuation as a bare `cont: QuoteId`. **That is
wrong**, and the reason is exactly the pathology we are trying to kill:

1. A bare `QuoteId` cannot represent a **partially-consumed** continuation. To "advance past
   the head" you would have to allocate a new `QuoteId` over the remaining slice every step —
   which either re-allocates or, if it just bumps `start`, still cannot express the *return*
   structure created by prepend.
2. A bare `QuoteId` cannot do **O(1) prepend**: splicing `q ++ cont` where `cont` is a flat
   slice requires copying the tail. That is the same O(n) splice we are removing.

The actual fix is an **offset cursor over a cons-list of segments**:

```rust
struct ContArena { nodes: Vec<ContNode> }             // node 0 = NIL sentinel
struct ContNode { qstart: u32, qend: u32, off: u32, parent: ContPtr }
type ContPtr = u32;                                   // 0 = NIL

#[derive(Clone, Copy)]
struct VmState { stack: StackPtr, cont: ContPtr, cursor: u32 }
```

A `ContNode` means "execute `tape[qstart..qend]`, resuming at offset `off`." The live
execution position is `(cont, cursor)`: the head node plus how far into it we've read.
`VmState` is entirely `Copy`, so **forking a machine copies ~12 bytes, O(1)** — this is what
makes wide speculative search cheap.

**read / advance (O(1) front-pop).** One step:

```
node = arena[cont];
if cursor < node.qend {
    word = tape[node.qstart + cursor];
    cursor += 1;
} else {
    // segment exhausted: pop to parent, restore its saved offset
    cont = node.parent;
    cursor = arena[cont].off;
    if cont == NIL { HALT }
}
```

A per-step `cursor += 1` replaces `cont.remove(0)`. No vector shift ever happens. **This kills
the 414× front-pop pathology.**

**prepend `q ++ cont` (O(1), no copy).** Used by `Apply`, PrimRec re-emit, `If`, `Dip`,
`Times`, and Fold re-emit. To prepend segment `q` in front of the current continuation, we
*freeze* the current head to capture its resume point, then push `q` as a child:

```
if cont != NIL {
    // freeze the current head so we can return to it at the right offset
    frozen = alloc ContNode { qstart: h.qstart, qend: h.qend, off: cursor, parent: h.parent };
} else {
    frozen = NIL;
}
child = alloc ContNode { qstart: q.start, qend: q.start + q.len, off: 0, parent: frozen };
cont = child;
cursor = 0;
```

At most 2 node allocations, no tail copy. When the child is exhausted, the pop rule above
restores `frozen`'s offset and execution resumes exactly where it left off.

#### Each pathology mapped to its kill

- **Flat front-pop** → cursor bump. `cont.remove(0)` becomes `cursor += 1`. Kills the 414×.
- **PrimRec re-emission** → one O(1) prepend of a fresh segment per level. The body `C` is
  interned once; re-emitting it is a `PushQuote`-style segment prepend, not a `|C|`-word copy
  per level. The continuation no longer grows by `|C|`/level. Kills the 223 ms → expected to
  become linear in recursion depth.
- **Fold tail** → sub-slice. The tail of an interned list is `QuoteId { start + 1, len - 1 }`
  into the *same* shared tape — O(1), no spine clone. Kills the 32 ms O(n²).
- **`: !` self-application** → unchanged; already linear, left alone.

### 2. Arena growth, generations & GC

Arenas grow monotonically within a run; reclamation is **generational**, not per-object.

A **generation** is one speculative search episode (or one driven run). At the start of a
generation the driver records high-water marks for all three arenas:
`(tape_len, stack_nodes_len, cont_nodes_len)`. To reclaim, it **truncates each arena back to
its mark**. This is wholesale reclamation with zero per-object free cost — the classic bump/reset
arena discipline.

Two consequences and one invariant:

- **Static program code survives.** Anything interned *before* the generation began sits below
  the high-water mark and is untouched by truncation. The program the machine is running is
  still there after reset.
- **Winners must be reified.** Anything a winning branch wants to keep must be **copied out**
  to owned `Vec<Value>` / `Vec<Word>` *before* the reset — because after truncation its handles
  dangle.
- **Driver invariant:** *no `QuoteId`/`StackPtr`/`ContPtr` allocated within a generation may
  escape the generation except via reification.* This is the single property that makes
  truncation safe.

This directly answers the gap the oracle sketch flagged around `cat` on a shared tape versus
generational reset. `cat` appends into the current generation's region of the tape and is
truncated away on reset. Under the invariant above, no `cat` result escapes un-reified, so
truncating it is safe — the only references to a `cat` result live inside the generation that
is being discarded, or have already been copied out.

The honest limitation is the **cross-generation** case: a long-lived program that never resets
(or resets rarely) accumulates tape growth, especially under `cat`-heavy workloads, because
interning is append-only within a generation. We bound this two ways: (1) make generation
boundaries **explicit** in the driver so resets actually happen at episode edges, and (2) if
needed, a future **compaction pass** that re-interns only the live quotes into a fresh tape.
Compaction is stated as future work — the spike uses simple truncation only, and we do not
claim otherwise.

### 3. Host boundary (the `Vec<Value>` seam)

At an `Invoke`, the core yields a snapshot of the operand stack as a `Vec<interp::Value>`.
Today `host.rs`'s `drive` does `core::mem::take(&mut vm.stack)`; `interp::Value` carries
`Quote(Vec<Word>)`. The arena must present the same seam.

**Arena → seam conversion.** Each value crossing the boundary is materialized: a `Quote(QuoteId)`
becomes an owned `Vec<Word>` by copying `tape[start..start+len]`, recursing into nested
`PushQuote`s so the owned value is fully self-contained. **Resume** converts back — the returned
stack is interned into the arena.

The cost is O(size of the values crossing), paid **once per `Invoke`**, and **not** on the hot
step loop. We are explicit about what the arena does and does not buy here: the arena
accelerates pure-core **steps**; the host-boundary marshalling is essentially unchanged and
remains O(crossing). This is acceptable precisely because `Invoke`s are **rare relative to
steps** — corpus solutions issue at most a few `Invoke`s against thousands of steps — so paying
a bounded marshalling cost at each boundary is the correct place to spend, and it does not
reintroduce an O(n²) into the loop we just cleaned up.

### 4. `u32` addressing limits (state bounds honestly)

Tape, stack, and continuation indices are `u32`, so each arena holds up to `2^32 ≈ 4.29e9`
words/nodes **per generation**. Crucially, this is an **addressing** limit, not a value limit:
the `Int` *payload* is a full `i64` and is entirely unaffected — only the handles that address
arena slots are `u32`.

The practical ceiling is memory, not the integer type: `2^32` nodes at ~16 B each is ~64 GB,
so a workload hits RAM long before it hits the `u32` boundary. Generational reset keeps live
counts far below the ceiling in normal operation.

Two commitments:

- **Allocation checks for `u32` overflow and returns a fault** — never silent wraparound. The
  review flagged silent-corruption fallbacks (e.g. `unwrap_or('\u{FFFD}')`) as an anti-pattern;
  we do not repeat it. Exceeding the addressing range is a clean, observable fault.
- **The `u32` choice is a deliberate tradeoff**, stated as such: it keeps `Value` and `VmState`
  small and `Copy`, which is the whole point of the arena. A program that needs more than
  ~4.29e9 live interned words *in a single generation* is out of scope for this backend and
  faults cleanly rather than corrupting.

### 5. Faults carry state under arena refs (the LLM-repair feature)

`interp`'s `FaultInfo` carries `{ fault, stack, cont }`, and the host uses that snapshot for
LLM-driven repair. Under the arena, `stack` and `cont` are references (`StackPtr` / `ContPtr` /
`cursor`) rather than owned data. The feature is preserved by **reifying at the fault**, which
is a terminal event:

- Walk the stack cons-list from `StackPtr` up through `parent` into an owned `Vec<Value>`.
- Walk the continuation segment cons-list — starting from the head node at its `cursor`, then
  each `parent` at its saved `off` — splicing `tape[qstart+off .. qend]` of each segment into
  the owned remaining `Vec<Word>`.

The cost is O(state size), paid **once** at the terminal fault, with **zero hot-loop cost**.
The reified snapshot has the **identical shape** to today's `FaultInfo`, so the host and the
LLM repair layer see exactly what they see now. The arena references are an internal
implementation detail; the boundary artifact is unchanged. This is why the arena is
*transparent* to the fault-repair layer — the acceleration is invisible above the seam.

## 4. Speculation (host layer)

### 1. Speculation is host-layer, not a primitive (why)

MTL's determinism (P1) is a *proved theorem*: for any program `p` and input stack `s`, `exec` is a pure total function of `(p, s, fuel)` with no ambient state, no clock, no nondeterministic choice point. Every downstream guarantee — the differential oracle, reproducible fault snapshots, the #26/#27 total-metering result — rests on P1 holding exactly. So the design rule is non-negotiable: **nothing added for speculation may weaken P1.**

That immediately rejects the oracle sketch's `fork`-as-primitive framing. A `fork` word — one that, mid-execution, splits the current continuation into two divergent futures the core must reconcile — introduces a choice point *inside* the evaluation relation. The moment the core can branch, "the result of running `p` on `s`" is no longer a function; it is a set, or it depends on a scheduler, and P1 is false. Re-proving a weaker theorem ("deterministic modulo a fork oracle") would invalidate the existing proof corpus and every artifact keyed to it. We do not pay that price.

Instead, **speculation is a driver that operates *over* cloned `VmState`s using the existing `drive`/`Host` seam.** The core language, its opcodes, and its proofs are untouched — bit-for-bit. Each speculative branch is an *ordinary deterministic core execution*: a `VmState` plus a fuel share, stepped by the same `exec_step` every non-speculative run uses. The driver's only jobs are (a) owning many such branches at once, (b) deciding which to step next, and (c) deciding which to keep. None of those decisions reach into the core; they are scheduling over a pure function.

The driver is **trusted (TCB), exactly like the `Host` is trusted** — and, like the host, it is *not* formally verified and does not need to be. Verification buys us the guarantee that *each branch computes the right thing*. The driver cannot corrupt that: it can only choose *which* correct computations to run and *which* to discard. A buggy driver can pick a bad branch, waste fuel, or fail to find a solution that exists — a *completeness/quality* defect, never a *soundness* defect. It can never make a branch report a result the core wouldn't. This is the load-bearing separation for the architecture review: the multiverse lives entirely in untrusted scheduling code, and the trusted core never learns it is being speculated over.

### 2. Driver API sketch (over `VmState` clones)

The arena backend makes this cheap. A branch *is* a `VmState { stack: StackPtr, cont: ContPtr, cursor: u32 }` — all `Copy`, ~12 bytes. Spawning a branch is copying those three words; the pointed-to stack (StackArena delta nodes), continuation (segment-cursor cons-list in ContArena), and quotation tape (interned QuoteArena) are persistent and structurally shared. Only the *divergent suffix* a branch actually writes allocates new nodes.

```rust
// ---- ILLUSTRATIVE, not compiled ----

/// A live speculative branch: a plain core state + its remaining fuel share.
struct Branch {
    id: BranchId,
    vm: VmState,        // 12 bytes, Copy; shares arenas with all siblings
    budget: u64,        // this branch's slice of the global fuel B
    score: i64,         // optional heuristic priority (policy input)
}

enum BranchOutcome {
    Live,                       // stepped, still running
    Halted(Vec<Value>),         // reached Done; result stack reified out of arena
    Dead,                       // Faulted / HostFaulted -> cull
    Deferred(InvokeReq),        // yielded a capability Invoke (see §3: buffered)
}

struct SpecDriver<'a> {
    arena:    &'a mut Generation,   // shared StackArena/ContArena/QuoteArena for this search
    frontier: Vec<Branch>,          // all live branches
    spent:    u64,                  // fuel already consumed across the whole search
    total_B:  u64,                  // the #26/#27 global budget (never exceeded)
    policy:   StepPolicy,           // RoundRobin | Priority(fn(&Branch)->i64)
}

impl<'a> SpecDriver<'a> {
    /// O(1): copy 12 bytes; child shares all arena structure with parent.
    fn spawn(&mut self, parent: &Branch, budget: u64, score: i64) -> BranchId { /* ... */ }

    /// Advance one branch by up to `k` core steps, drawing from its own slice.
    /// Reuses exec_step exactly as `drive` does — no core changes.
    fn step_with_quota(&mut self, b: BranchId, k: u64) -> BranchOutcome {
        // for _ in 0..min(k, branch.budget) { exec_step(&mut branch.vm, self.arena)?;
        //     branch.budget -= 1; self.spent += 1; }  // see §3 for the shared-budget invariant
    }

    fn cull(&mut self, b: BranchId);                    // drop VmState; reclaim budget to pool
    fn collect_halts(&mut self) -> Vec<(BranchId, Vec<Value>)>;
}
```

Main loop shape:

```rust
fn search(mut d: SpecDriver, stop: StopRule) -> SearchResult {
    while !d.frontier.is_empty() && d.spent < d.total_B {
        let b = d.policy.pick(&d.frontier);        // round-robin or best-score
        let quota = d.policy.quota_for(b);         // k steps this turn
        match d.step_with_quota(b, quota) {
            BranchOutcome::Halted(stack) => {
                if stop == StopRule::FirstHalt { return SearchResult::Found(stack); }
                d.record_halt(b, stack);           // else keep collecting
            }
            BranchOutcome::Dead      => d.cull(b), // reclaim its remaining budget to the pool
            BranchOutcome::Deferred(req) => d.buffer_effect(b, req), // §3: dry-run
            BranchOutcome::Live      => { /* stays in frontier */ }
        }
    }
    d.resolve()   // best halt, or Exhausted / NoSolution
}
```

Each branch is a plain deterministic core run; the driver just owns a *frontier* of them plus one shared arena generation. When a search resolves, the generation's arenas reset wholesale — culled branches were only ever dropped `VmState`s, so there is nothing to free individually. Memory scales with *divergence*, not branch count: `S` shared structure + `Σ divergent suffixes`, not `N × full state`.

### 3. Metering per branch (budget splits — tie to #26/#27)

This is the part that must survive review, because it is where a naive design silently breaks a *proved* property.

The #26/#27 lineage guarantees **total metering**: fuel is a single global *decreasing* budget `B` across *all* resumptions, never reset per Invoke, and exhaustion at a step boundary yields `Cancelled`. A program that yields a capability inside a non-terminating loop still gets cancelled, because the global budget runs out. Speculation must not become an escape hatch from this.

The rule: **`B` is not multiplied, it is *split*.** On spawn, the parent's remaining budget is partitioned among children — equal shares by default, or policy-weighted (proportional to `score`). The driver maintains, as an invariant checked at every step boundary:

```
   Σ_{b ∈ live} budget(b)  +  spent  ≤  B      (always)
```

`step_with_quota` decrements the *stepped branch's* slice and increments the shared `spent`; it may never let the left side exceed `B`. Therefore **N speculative branches can never collectively out-run what a single sequential `drive(B)` was allowed.** The whole multiverse consumes at most `B` core steps, total, exactly as if the fuel had been spent on one line. The #26/#27 total-metering property survives the multiverse — the search is not a fuel amplifier.

Reclamation: when a branch is culled (Faulted/HostFaulted) or halts, its *unspent* slice returns to a shared pool that the driver may re-distribute to surviving branches. This keeps the search efficient (a dead branch's fuel isn't wasted) *without* ever raising the ceiling — the pool is drawn from `B`, never added to it. The invariant above is preserved because reclaimed fuel was already inside `B`.

Degenerate and boundary cases, stated honestly:

- **One branch** = ordinary `drive(vm, B, host)`. The whole apparatus collapses to the existing seam; this is the sanity anchor and should be an equality test in the corpus.
- **Rounding**: integer splits leave a remainder (`B mod N`); assign it to the pool, never mint it. `Σ` must hold with `≤`, not `=`.
- **A branch that yields an `Invoke` (capability effect).** This is the key design constraint. Effects are *irreversible* — an Invoke that sends a byte, mutates a store, or reads a nonce cannot be un-done if the branch is later discarded. So **speculative branches run in a "dry-run" effect mode: real Invokes are buffered/deferred, not committed.** The driver captures the `InvokeReq` (name + stack snapshot), parks the branch, and does *not* service it against the real host. Only when the search *collapses* onto a winning branch are that branch's buffered Invokes replayed against the real host, in order, exactly as a non-speculative `drive` would have issued them. Every losing branch's buffered effects are dropped with the branch. Consequences the review must accept: (a) a branch that *cannot make progress without* a real effect result (e.g. it needs the host's reply to decide its next step) is *not speculatable past that point* — the driver must either stub the reply from a declared pure model or block the branch until it is chosen; (b) speculation is therefore most valuable for *effect-light search* (pure computation over parameters/programs) and degrades gracefully — into ordinary sequential execution — for effect-heavy work. We do not claim to speculate through arbitrary side effects; we claim to speculate through *pure core computation* and to *defer* effects to the winner.

### 4. Admission benchmark design (the deliverable)

**Question.** Does VM-level speculative search earn its keep against the current approach — an LLM attempt-loop — on search-shaped tasks? We design (not run) an experiment that can return a clear *yes* or a clear *no*.

**Tasks (5, concrete).** Each gives start state, target, and search-space size.

- **(a) Halting constant — 1-D search.** Given a fixed quotation `q` with one free integer literal `k`, find `k` such that `[k] q` halts `Done` with target output `[42]`. Search space: `k ∈ [0, 2^16)`, ~65k candidates, cheaply enumerable; solution verified by running `q` to `Done` under a per-candidate fuel cap.
- **(b) Operator-sequence synthesis — combinatorial.** Start stack `[3, 5]`, target `[64]`, alphabet `{add, sub, mul, dup, swap}` (5 prims), programs of length ≤ 6. Space: `Σ_{l≤6} 5^l ≈ 19,530` programs; prune on stack-underflow fault and on partial-value bounds.
- **(c) Loop coefficients — numeric target.** A fixed loop skeleton `[a,b] -> repeat n times: x := a*x + b`, find `(a, b, n)` with `a ∈ [1,8]`, `b ∈ [-8,8]`, `n ∈ [1,16]` reaching target `x = 1000` from `x0 = 1`. Space: `8 × 17 × 16 = 2,176`.
- **(d) Reachability puzzle.** Reach `N = 100` from `1` using ops `{+3, ×2, −1}` in ≤ `k = 12` moves. Space: branching factor 3, depth 12 → `3^12 ≈ 531k` paths, heavily prunable by dominance/visited-state pruning; solution = op sequence, verified by replay.
- **(e) Fifth task — target-stack routing (own choice).** Given start `[1,2,3,4]` and target `[4,3,2,1]` (full reversal), find a shortest program over `{swap, rot, over, drop, dup}` of length ≤ 8. Space: `Σ_{l≤8} 5^l ≈ 488k`; chosen because the LLM has a *strong prior* on stack-shuffle idioms, so it is a task where Arm A is expected to be competitive — deliberately included to give the experiment a chance to produce a **negative** result for speculation.

**The two arms.**

- **Arm A — LLM attempt-loop (current approach).** The LLM proposes an MTL program; the host runs/validates it; on failure the LLM receives the fault-carries-state snapshot (the reified `Fault` + stack at the failure point) and retries. Iterate to first validated solution. Count LLM tokens (input + output, summed across all retries) and wall-clock.
- **Arm B — VM-level speculative search.** The LLM frames the search space *once* (it declares the alphabet, bounds, start, and target — one prompt, one response). Then the `SpecDriver` enumerates/prunes candidate branches at hardware speed under a single global fuel budget `B`, with **no LLM in the inner loop**. Culling is by fault (Arm B gets the same fault signal, but consumes it in-process at step speed rather than round-tripping to a model).

**Metric.** Primary, per task, per arm: **(i) LLM tokens** (input + output) and **(ii) wall-clock**, both measured to *first validated solution* (validated by the same host validator in both arms, to keep the bar identical). Secondary: solutions-explored-per-second (Arm B's inner throughput; for Arm A, candidates-per-minute), and memory high-water (arena high-water for Arm B; process RSS delta as a rough proxy for Arm A).

**Decision rule (admission threshold).** The refactor + speculation earns implementation iff **both**:

1. On **≥ 3 of the 5 tasks**, Arm B reaches first validated solution using **≥ 10× fewer LLM tokens OR ≥ 5× less wall-clock** than Arm A; **and**
2. Arm B's arena backend passes the **differential oracle on 100% of the corpus** (every branch's result agrees bit-for-bit with the reference tree-backed interpreter — speculation must not have introduced an evaluation discrepancy).

Both conditions are hard gates. Condition 2 is non-negotiable regardless of the speedup: a fast search that ever disagrees with the oracle is a *correctness* failure and disqualifies the backend outright. This is a **design for an experiment to run later** — there are no results here, and none should be inferred.

**Where speculation loses (must be observable).** The experiment is built to be able to *fail* for Arm B, and we name where:

- **Non-enumerable / large spaces.** Tasks whose search space is not cheaply enumerable under `B` (real-valued, high-dimensional, or combinatorially explosive past pruning) will exhaust the budget without a halt. Arm B returns `Exhausted`; Arm A, guided by the model's prior, may still find a solution. Task (a)–(d) are deliberately sized to be enumerable; a *stretch* variant of each (widen bounds 100×) should be included precisely to locate Arm B's cliff.
- **Strong-prior tasks.** Where the LLM's prior drastically prunes the space — idiomatic stack shuffles (task e), well-known numeric identities — the model may reach a solution in one or two attempts, and Arm B's brute enumeration, even at hardware speed, does more total work. If Arm A wins tasks like (e), that is a **valid negative result** to report, not a bug to tune away.

A run that yields "Arm B wins the 1-D and small-combinatorial tasks by >10× tokens but loses the strong-prior and non-enumerable tasks" is a *legitimate and expected* outcome, and the decision rule (≥3 of 5) is calibrated to admit speculation only if it clears a majority — not to launder a marginal result.

### 5. Prototype status

The `SpecDriver` is **designed here, not built.** The subsections above are prose, illustrative (non-compiled) API sketches, and an experiment specification. A prototype `SpecDriver` over the arena backend is built **only if the v0.5 spike leaves time after the arena backend itself lands and passes the differential oracle** — and even then, a prototype exists to *run* the Arm-B side of the benchmark, not to ship.

This matches the project's design-round pattern: **measure and design first, implement after review.** The deliverable this round is the design and the experiment spec; the go/no-go on implementation is the decision rule in §4, evaluated against results gathered in a later round. Nothing in this section should be read as a claim that speculation *is* faster — only as a claim about *how we will find out*, under a threshold strict enough that the honest answer might be "no."

## 5. Drift accounting

Semantic drift is the architecture review's #1 named risk. The primitive set and the
`Word`/`Value` shapes are already mirrored in **six** places, kept in lockstep today by
`crates/mtl-conformance` comparing against `crates/mtl-syntax/src/manifest.rs`. The existing
six mirrors are: `syntax::Prim`, `mtl_core::SpecPrim`, `interp::Prim`, the `GLYPHS` table, and
the two conversion functions between representations. A new backend that re-declares the
primitive set is, by construction, a drift risk. This section states exactly what the arena
adds and exactly how it is policed, and refuses to overclaim.

### Mirrors the arena backend adds

The arena introduces new declarations that must stay in lockstep with the existing six:

1. **Arena `Prim` enum** — a new declaration of the primitive set (its variants, order, and
   arity).
2. **Arena `Word` enum** — a new opcode shape (`PushInt` / `PushQuote` / `Prim` / `Call`).
3. **Arena `Value` enum** — a new value shape (`Int` / `Quote`).
4. **The `Call(u32)`-vs-`Call(String)` representation mirror** — the arena interns capability
   names to `u32` while `interp` carries `String`. The name table and its mapping is itself a
   mirror: an id must denote the same capability on both sides.

These are enumerated explicitly so none can hide. Left unpoliced, they would make the arena a
**seventh un-policed mirror** — precisely the outcome the review warns against.

### How the conformance crate extends to police them

The arena enters the *same* manifest-policed regime as `interp`, or it does not ship. Concretely,
the conformance crate is extended with:

- **A manifest-match test for the arena `Prim` enum** asserting its variant **names, order, and
  arity** match `manifest.rs` — exactly as the existing `interp_prim_names_match_manifest` and
  `arity_matches_interp_underflow` tests do for the interpreter. The arena `Prim` is compared
  to the single source of truth, not to `interp` transitively.
- **An arena↔interp differential-oracle test, promoted from the spike into the conformance
  crate**, so it runs in **CI on the corpus** — every corpus program is executed on both
  backends and their observable results (final stack, fault, `Invoke` sequence) are asserted
  equal. This is the day-to-day guard against behavioral drift.
- **A fault-order check** asserting the arena raises faults in the same order as `interp` —
  arity-before-type, divzero-before-overflow — so the two backends agree not just on *whether*
  a program faults but on *which* fault wins.
- **A name-interning round-trip check** for the `Call(u32)`/`Call(String)` mirror: interning an
  interp `Call(String)` to a `u32` and back yields the original capability, and the id space is
  consistent across the corpus.

The rule is stated flatly: **the arena must not become a seventh un-policed mirror.** It is
governed by the manifest exactly as the other mirrors are, and CI enforces it.

### The eventual P2-style proof obligation (statement only)

The production arena machine (`arena_step` over `VmState`) carries a Verus **refinement
obligation** that is *stated here but not executed in this round*:

> There exists an abstraction function **α** from an arena `VmState` — dereferenced through the
> `QuoteArena`, `StackArena`, and `ContArena` — to the ghost VM state, such that for all
> reachable states `s`:
>
> ```
> α(arena_step(s)) = spec_step(α(s))
> ```

Under this α, the **segment-cursor continuation** abstracts to the ghost `Seq<Word>`
continuation (α flattens the segment cons-list, honoring each node's offset, into the ghost
sequence), and the **persistent stack** abstracts to the ghost stack (α walks the `parent`
chain into the ghost `Seq`/list). A simulation/refinement theorem of this shape is what would
mechanically pin `arena_step` to the frozen `spec_step`.

Until that theorem exists, the arena is validated **only** by differential oracle plus
conformance — which is *exactly* the status the review already assigns to the production Rust
interpreter today: a **twin validated by differential testing, not mechanically proved**. We
make no stronger claim. In this round the arena is a **differentially-validated backend, not a
proved one.** The refinement theorem is the honest next step, named and scoped, deliberately
left unexecuted here.

## 6. Spike admission summary + recommendation

### Headline measured wins

The spike answers the admission question on all three pathologies, and on fork:

- **PrimRec `sum_to(10k)`: 253 ms → 1.76 ms (144×)**, arena ns/step flat (~29 → ~28) as n
  grows 10× — O(1)/level, not the interp's `|C|`-per-level tail copy. At n=100k the arena runs
  in 16.8 ms where interp is projected ~25 s.
- **Flat front-pop: 754× at 16k steps.** Interp ns/step balloons 281× (20 → 5712 ns) over the
  flat scale; arena stays flat at ~7.6 ns (O(1) cursor bump).
- **Fold: 40× at 10k** (36.3 ms → 0.91 ms) — the tail becomes an O(1) shared sub-slice, no
  spine clone.
- **VM fork: O(1), ~1.09 ns flat** across stack depths 1…10000, versus an interp `Vm.clone()`
  that scales linearly to ~23 µs at depth 10k (a 452× rise).
- **Differential oracle: 47/47** corpus programs bit-identical across both backends.

### Honest caveats

- **The arena never frees in the spike.** Tape, stack-arena, and cont-arena grow monotonically
  for the life of a run. The Fold ns/step rise (12.8 → 46 ns) is **cache pressure from that
  unbounded growth, not algorithmic** — per-element work is O(1). A production backend needs
  compaction / generational recycling of dead nodes.
- **`u32` addressing bounds.** Handles are `u32`, capping each arena at ~4.29e9 entries per
  generation; allocation faults cleanly on overflow rather than wrapping. The `i64` value
  payload is unaffected.
- **Differentially-validated, not proved.** The arena is a *separate* implementation validated
  only by the oracle + conformance — not the Verus-verified twin, and it carries no proof
  obligation this round. The P2 refinement theorem (§5) is stated, not discharged.
- **Single shared-container run.** Numbers are single-run and shared-container; they vary
  run-to-run and machine-to-machine. **The ratios and the flat-vs-growing ns/step shapes are
  the durable result, not the absolute ns.**

### Recommendation — a gated yes

The arena **decisively clears the admission bar for large-data and search workloads**: the
O(n²) is genuinely killed (144× on the primary case, with a flat ns/step confirming O(1)/level)
and fork is genuinely O(1) (~1.09 ns flat). On the merits, the full REFACTOR earns production
implementation **if a large-data or speculative-search use case is real** — the speculation
admission benchmark (§4.4) is the instrument that decides that.

It is **not urgent for the current corpus.** Every corpus solution today runs in ≤115 steps /
microseconds — three orders of magnitude below where the O(n²) bites. For the shipping language
as it stands, the arena changes nothing a user would feel.

Production entry is therefore **GATED**, not open: the arena must enter the *same*
manifest-policed regime as every other mirror — the conformance extensions of §5 (manifest-match
on the arena `Prim`, the promoted differential oracle in CI, the fault-order and
name-interning checks) **and** the stated P2 refinement obligation. It must **not ship as a
seventh un-policed mirror**. The recommendation is a **gated yes**: implement when a real
large-data/search workload justifies it, and only through the drift-accounting gate — not
before, and never around it.

## 7. Status / open obligations (post-implementation)

The gated production backend has now landed as the crate `crates/mtl-arena` (an interned,
persistent, O(1)-fork continuation engine faithful to the spike), exposing the opt-in entry
points `mtl_arena::run_arena` (pure execution) and `mtl_arena::host::arena_drive` (the
host-capability seam, mirroring `mtl_core::host::drive` outcome-for-outcome under the same
global-fuel/cancellation semantics). It is **opt-in and never silently substituted**; the
`mtl-core` interpreter (`crate::interp`) remains the default reference twin and normative
oracle. Semantics stay **frozen** — no new primitives, no new outcome, no change to `spec_step`
or the language spec.

**Done this round:** the arena is **differentially validated** against the interpreter —
bit-identical across the 47-case differential oracle, fault-corpus `FaultInfo` parity, and
host-driver parity against `host::drive`; the conformance/manifest policing of §5 (arena
`Prim`/`Word`/`Value` mirror, promoted oracle, fault-order and name-interning checks) is the
drift-accounting gate it enters under.

**Open — the P2-style refinement obligation (intentionally deferred, NOT done this round):**
the arena backend ships **differentially validated but WITHOUT a machine-checked refinement
proof.** The open obligation is exactly the one stated in §5: there is a Verus refinement
theorem — an abstraction function **α** from an arena `VmState` (dereferenced through the
`QuoteArena`, `StackArena`, and `ContArena`) to the ghost VM state such that
`α(arena_step(s)) = spec_step(α(s))` for all reachable `s` — i.e. that **`arena_step` refines
`spec_step`** (equivalently, that the arena backend refines the reference interpreter / spec),
directly analogous to the **P2** refinement obligation discharged for `exec_step` against
`spec_step`. This theorem is **not executed this round**; no Verus refinement proof for the
arena was attempted. Until it is discharged the arena is validated **only** by the differential
oracle plus conformance — the same validated-not-proved status the review assigns to the
production Rust interpreter twin. This is a named, scoped, deliberately deferred open item, not
an oversight.
