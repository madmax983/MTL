# MTL v0.2 — Recursion Primitives (design stage, measurement-driven)

- Status: **design stage**. Candidate primitives below are **not yet implemented** in the parser/interpreter, so their programs are **hand-traced against the semantics sketches**, not interpreter-validated. Every token count is real (measured with `bench/tokcount`); every correctness claim is a hand-trace, explicitly marked.
- Branch: `v02-recursion-design`. Frozen `T_v0` corpus, `bench/tokcount/tasks.json`, and `bench/BASELINE.md` are untouched. All new work lives under `bench/design-v0.2/`.
- Tokenizers: `tiktoken` `o200k_base` and `cl100k_base` at 0.8.0 (the pinned bench set). Counts below are identical under both encodings unless a cell says otherwise.

## 1. The problem, quantified

The interpreter-validated `T_v0` aggregate is **2.11×** vs idiomatic Python (`bench/BASELINE.md`), below the **≥3×** go/no-go gate. Straight-line stack code clears 3× easily (rev3 8.0×, is_even 3.5×, affine 3.25×). Real `: !` recursion is where it bleeds:

| task | idiomatic Python (tok) | MTL v0.1 (tok) | ratio |
|---|---:|---:|---:|
| factorial `1~[^0=[__][[:@*~1-]':!]?]:!` | 26 | 19 | 1.37× |
| gcd `[^0=[__][[~^%]':!]?]:!` | 24 | 15 | 1.60× |

The `: !` self-application idiom forces the author to hand-thread the retained self-copy `L` through the stack, reach *under* it with `^`/`@` to touch the loop variable, clean it up in the base case (`[__]`), and re-establish it with `' : !` on every recursive branch. That scaffolding — not the actual arithmetic — is the token cost, and it is also the dominant LLM failure mode (the original factorial/gcd sketches faulted `TypeMismatch`; see `bench/BASELINE.md` caveats).

**Goal:** primitives that internalise the self-application harness so the body expresses only the recurrence, admitted strictly by corpus-level token accounting (spec §5, §10.2, §11).

## 2. Candidate set

| candidate | shape it captures | stack effect | glyph (measured, §5) |
|---|---|---|---|
| `primrec` | primitive recursion on a natural count | `( n [I] [C] -- r )` | `&` |
| `times` | bounded iteration | `( n [Q] -- … )` | `.` |
| `linrec` | general linear / tail recursion | `( [P] [T] [R1] [R2] -- … )` | `\|` |
| `uncons` | quotation deconstruction (structural) | `( [w …] -- w [ … ] 1 )` \| `( [] -- 0 )` | `>` (provisional) |
| `pick` / `roll` | indexed deep-stack access | `( … n -- … v )` / `( … n -- … )` | deferred |

`primrec`, `times`, `linrec` are admitted on token grounds (§6). `uncons` is admitted on a *different* rationale — it is the direct enabler of the honest Turing-completeness proof (spec §6.2) and of `T_v0.2` list/string tasks — not because it moves the recursion-corpus aggregate. `pick`/`roll` are the review's writability bet (spec §10.6); on this corpus the bodies are ≤3 deep and measurement shows **no** stage-1 token win, so they are **deferred to the warm/cold agent trial** rather than admitted on intuition (§8).

## 3. Small-step semantics (spec_step extension sketches)

Written to match `spec_step_prim` in `crates/mtl-core/src/mtl_core.rs`: dispatch on `p`, arity check first (`Fault(Underflow)`), then operand types (`Fault(TypeMismatch)`), then semantic checks — the normative fault precedence (spec §4.4). Stack is `Seq<SpecValue>`, `n = stk.len() as int`, top at `stk[n-1]`. New `SpecPrim` variants: `PrimRec, Times, LinRec, Uncons`.

### 3.1 `primrec` — `( n [I] [C] -- r )`

Primitive recursion on a natural count. `n<=0` runs `I` on the base stack; `n>0` yields `C(n, primrec(n-1, I, C))`, keeping `n` available to `C`. **Total and terminating for every integer `n`** (the count strictly decreases toward 0), which is a stronger guarantee than the partial `: !` idiom.

```rust
SpecPrim::PrimRec => {
    if n < 3 { SpecStep::Fault(Error::Underflow) }
    else {
        match (stk[n - 3], stk[n - 2], stk[n - 1]) {
            (SpecValue::Int(k), SpecValue::Quote(qi), SpecValue::Quote(qc)) => {
                let base = stk.subrange(0, n - 3);
                if k <= 0 {
                    // base: discard the count, run I on the base stack
                    SpecStep::Next(SpecState { stack: base, cont: qi + rest })
                } else {
                    // else: keep n, recurse on (n-1), then run C(n, sub).
                    // k>0 && k<=i64::MAX  =>  0 <= k-1 < k, so in_i64(k-1) holds:
                    // no Overflow arm is reachable here.
                    let recur = seq![
                        SpecWord::PushInt(k),
                        SpecWord::PushInt(k - 1),
                        SpecWord::PushQuote(qi),
                        SpecWord::PushQuote(qc),
                        SpecWord::Prim(SpecPrim::PrimRec)
                    ] + qc;
                    SpecStep::Next(SpecState { stack: base, cont: recur + rest })
                }
            }
            _ => SpecStep::Fault(Error::TypeMismatch),
        }
    }
}
```

### 3.2 `times` — `( n [Q] -- … )`

Run `Q` exactly `max(n,0)` times, left to right. Total and terminating (count decreases; `n<=0` is a no-op).

```rust
SpecPrim::Times => {
    if n < 2 { SpecStep::Fault(Error::Underflow) }
    else {
        match (stk[n - 2], stk[n - 1]) {
            (SpecValue::Int(k), SpecValue::Quote(q)) => {
                let base = stk.subrange(0, n - 2);
                if k <= 0 {
                    SpecStep::Next(SpecState { stack: base, cont: rest })
                } else {
                    // run Q once, then times(n-1) Q
                    let recur = q + seq![
                        SpecWord::PushInt(k - 1),
                        SpecWord::PushQuote(q),
                        SpecWord::Prim(SpecPrim::Times)
                    ];
                    SpecStep::Next(SpecState { stack: base, cont: recur + rest })
                }
            }
            _ => SpecStep::Fault(Error::TypeMismatch),
        }
    }
}
```

### 3.3 `linrec` — `( [P] [T] [R1] [R2] -- … )`

General linear recursion (Joy's `linrec`), whole-stack: run `P` (leaves a flag); if nonzero run `T`; else run `R1`, recurse with the same four quotes, run `R2`. It **desugars into the existing `If` primitive** — this is the whole trick, and it means `linrec` inherits `If`'s verified branch semantics rather than adding a new control operator. **Partial**, exactly like `!`: termination depends on `P`/`R1` and is bounded by fuel.

```rust
SpecPrim::LinRec => {
    if n < 4 { SpecStep::Fault(Error::Underflow) }
    else {
        match (stk[n - 4], stk[n - 3], stk[n - 2], stk[n - 1]) {
            (SpecValue::Quote(qp), SpecValue::Quote(qt),
             SpecValue::Quote(qr1), SpecValue::Quote(qr2)) => {
                let base = stk.subrange(0, n - 4);
                // else-branch: R1 ; (re-push the four quotes) linrec ; R2
                let else_q = qr1 + seq![
                    SpecWord::PushQuote(qp),
                    SpecWord::PushQuote(qt),
                    SpecWord::PushQuote(qr1),
                    SpecWord::PushQuote(qr2),
                    SpecWord::Prim(SpecPrim::LinRec)
                ] + qr2;
                // continuation: P ; push T-quote ; push else-quote ; If
                let spliced = qp + seq![
                    SpecWord::PushQuote(qt),
                    SpecWord::PushQuote(else_q),
                    SpecWord::Prim(SpecPrim::If)
                ];
                SpecStep::Next(SpecState { stack: base, cont: spliced + rest })
            }
            _ => SpecStep::Fault(Error::TypeMismatch),
        }
    }
}
```

Tail recursion (e.g. gcd) is `linrec` with `R2 = []`; a dedicated 3-quote `tailrec` would save exactly 1 token on gcd (9 vs 10) at the cost of a fourth new primitive — not worth it (§6).

### 3.4 `uncons` — `( [w …] -- w [ … ] 1 )` | `( [] -- 0 )`

Deconstruct a quotation: on non-empty, push the head **as a value**, the tail quotation, and flag `1`; on empty, push flag `0`. Structural and **affine** — the input quotation is consumed once and split, never duplicated. A head word that is not itself a value (a bare `Prim`/`Call`, not `PushInt`/`PushQuote`) faults `TypeMismatch`; whether to instead wrap the head as a singleton quote `[w]` is the one open design decision here (flagged for the implementer).

```rust
SpecPrim::Uncons => {
    if n < 1 { SpecStep::Fault(Error::Underflow) }
    else {
        match stk[n - 1] {
            SpecValue::Quote(q) => {
                let base = stk.subrange(0, n - 1);
                if q.len() == 0 {
                    SpecStep::Next(SpecState {
                        stack: base.push(SpecValue::Int(0int)),
                        cont: rest,
                    })
                } else {
                    let tail = SpecValue::Quote(q.subrange(1, q.len() as int));
                    match q[0] {
                        SpecWord::PushInt(i) => SpecStep::Next(SpecState {
                            stack: base.push(SpecValue::Int(i)).push(tail).push(SpecValue::Int(1int)),
                            cont: rest,
                        }),
                        SpecWord::PushQuote(s) => SpecStep::Next(SpecState {
                            stack: base.push(SpecValue::Quote(s)).push(tail).push(SpecValue::Int(1int)),
                            cont: rest,
                        }),
                        _ => SpecStep::Fault(Error::TypeMismatch),
                    }
                }
            }
            _ => SpecStep::Fault(Error::TypeMismatch),
        }
    }
}
```

## 4. Multiplicity / affine story

The distinction that matters for the substructural reading (spec §14.4): does a primitive **consume its quote argument once** (affine) or **replicate it along the recursion spine** (multiplicative)?

| primitive | count arg | quote arg(s) | reading |
|---|---|---|---|
| `primrec` | `n` linear (decremented copies are fresh values) | `[I]`,`[C]` **replicated** n times | like `!` under recursion — the quotes are shared/copied down the spine |
| `times` | `n` linear | `[Q]` **replicated** n times | same |
| `linrec` | — | all four quotes **replicated** along the spine | same as `: !`; the whole point is the author no longer replicates them *by hand* |
| `uncons` | — | input quote **consumed once, split** | genuinely **affine/linear** — no duplication; the clean case, easy to reason about |

The replicated-quote primitives don't add a *new* substructural wrinkle over `!`/`:!`; they *remove* the hand-written replication (`' : !`, `[__]`) that the author currently performs. `uncons` is the one that's structurally linear.

## 5. Glyph assignment (measured, spec §11)

Per §11 the glyph is chosen by rendering the real corpus and counting tokens — BPE merges are context-dependent, so this is measured, not chosen for looks. Full sweep in `bench/design-v0.2/measurements.md`; the deciding rows:

| primitive | glyph | key merge fact | factorial/gcd/fib/power cost |
|---|---|---|---|
| `primrec` | `&` | `]&` is a **single token** in both encodings (probe `]&` → 1 tok) | `[1][*]&` = 5, `[0][+]&` = 5 |
| `times` | `.` | best fib merge; `01@[~^+]._` beats the `&`/`\|`/`>` variants | fib 6, power `1~[^*].~_` = 7 |
| `linrec` | `\|` | `]\|` merges (probe `[*]\|` → 2 tok); the glyph barely moves the 4-quote count | gcd `[:0=][_][~^%][]\|` = 10 |

Alternatives tested and rejected: `$` costs +1 under o200k on several forms (BPE divergence); a letter glyph (`r`, `w`) costs +1 (no punctuation merge). `.`, `&`, `\|` are also common code punctuation, so they ride **strong learned priors** — which is the rare case where the tokenizer-optimal choice is *also* the generatable one (§8, spec §11.8). `uncons` glyph `>` is provisional pending list-task measurement (no list task exists yet in the corpus).

Note for the implementer: `.` is unambiguous next to digits because integer literals are `[0-9]+` with no decimal point (spec §2.3), so `3.` lexes as `Int(3) Times`, never a float.

## 7. Hand-traces (design stage — NOT interpreter-validated)

Traced against §3. Stack shown bottom→top.

**factorial `[1][*]&`, input `[3]`** (primrec, §3.1):
`[3,[1],[*]]` →`&`(k=3)→ push `3 2 [1][*]& *`, base `[]` → … recursion retains the counts `3,2,1` on the stack and appends one `*` per level; base (k=0) runs `[1]` → then the three `*` fold: `[3,2,1,1] *→[3,2,1] *→[3,2] *→[6]`. Result `[6]` = 3!. ✓ Verified n=0,1,2,3 by hand.

**gcd `[:0=][_][~^%][]|`, input `[12,8]`** (linrec, §3.3):
`P=:0=` tests top nondestructively; `If` picks `R1=~^%` = `[a,b]→[b, a mod b]`, recurse; base `T=_` drops the 0. `[12,8]→[8,4]→[4,0]→(base)[4]`. Result `[4]` = gcd(12,8). ✓ (`R2=[]` so the unwind is identity — tail recursion.)

**fib `01@[~^+]._`, input `[3]`** (times, §3.2):
`0 1` seed accumulators, `@` rotates the count `n` to the top, `[~^+]` is the Fibonacci step `[a,b]→[b,a+b]`, `.` runs it n times, `_` drops `b`. `[0,1]→[1,1]→[1,2]→[2,3]`, drop → `[2]` = fib(3). ✓ Verified n=0,1,2,3.

**sum_to `[0][+]&`** (primrec): base `[0]`, combine `[+]` → n + (n-1 + … + 0). **power `1~[^*].~_`** (times): seed acc `1`, step `[^*]` multiplies acc by b, run e times, clean up. Both hand-verified on small inputs; see `bench/design-v0.2/candidates/`.

## 8. Writability (E[tokens × attempts])

The review warns the most *compressible* alphabet may not be the most *generatable* (spec §10.6, §11.8). Per candidate:

- **`primrec` — strongest writability win.** The model writes a named, pattern-matchable shape: "primitive recursion = base quote + combine quote," `[base][combine]&`. It **eliminates** the dominant `: !` failure mode: no retained self-copy `L` to thread, no `[__]` base cleanup, no `'` dip, no `^`/`@` reaching under `L`. The only remaining obligation is the combine's local 2-item stack effect (`n`, subresult) — shallow, not the 3-deep `[acc,n,L]` juggling of v0.1. `&` rides a strong code-punctuation prior. Net: removes a failure mode, adds none material.
- **`times` — strong.** `n [step] .` is a `for`-loop the model already knows. Removes the manual counter + termination test. New (small, bounded) cost: laying out the accumulator *under* the count before the call (the `@`/`~` in fib/power) — a shallow routing step, not open-ended juggling.
- **`linrec` — better than `: !`, not as clean as primrec/times.** Removes self-copy threading, but the author still supplies four quotes and must get `P` nondestructive and `R1`/`R2` stack-neutral. It's a known Joy idiom (some prior) but heavier; use it only where `primrec`/`times` don't fit (genuine multi-arg tail/linear recursion like gcd).
- **`uncons` — pure writability gain, no new footgun.** Replaces ad-hoc quote surgery (`,`/`;` gymnastics) with one structural, affine step; single consumption is easy for the model to track. Its real value is enabling structure-driven recursion and the honest TC proof.
- **`pick` / `roll` — writability bet, unmeasurable at stage 1.** The claim (Gemini review) is they cut the stack-juggling tax for *deep* access, which LLMs track poorly. On this corpus the bodies are ≤3 deep, so stage-1 tokens don't move — admitting them now would be intuition, not measurement. **Deferred to the warm/cold agent trial (spec §10.5)**, which is the only instrument that can adjudicate a writability-only claim.

## 9. Recommended admission set

**Admit on token grounds: `primrec` (`&`), `times` (`.`), `linrec` (`|`).** Minimal set that clears the gate with margin (3.72× frozen / 4.37× recursion-weighted), each primitive independently justified (§6.4), each a net writability improvement (§8).

**Admit on the TC-proof / list-enablement rationale (not corpus tokens): `uncons` (`>`, provisional glyph).**

**Defer pending the agent trial: `pick` / `roll`, and a dedicated 3-quote `tailrec`** (saves 1 token on gcd — not worth a fourth combinator).

## 10. Implementation plan

1. **spec_step** (`crates/mtl-core/src/mtl_core.rs`): add `SpecPrim::{PrimRec, Times, LinRec, Uncons}` and the four arms from §3. Fault precedence (§4.4) is preserved — each arm checks arity before types. `primrec`/`times` need no `Overflow` arm (`k-1` is provably in-range).
2. **Proofs:**
   - Preserve the P1-style totality of `spec_step` (the arms are total functions).
   - **New:** termination lemmas for `primrec`/`times` (the count strictly decreases → halts within fuel ∝ n). These are *stronger* than anything provable about `: !`/`linrec`, and worth stating as smoke theorems à la `smoke_dup_apply` (e.g. `primrec` at k=0 steps to the base; k>0 steps to the recur expansion; `times` likewise).
   - `linrec`: no termination claim (partial, like `!`); prove only the one-step desugaring (`linrec` == `P` ++ `If` ++ recur), which reduces it to the verified `If`.
   - Extend the exec-side twins (`exec_prim` etc. in `crates/mtl-core/src/interp.rs`) and re-run the P2 refinement lockstep.
3. **Parser** (`crates/mtl-syntax/src/ast.rs`): add `Prim::{PrimRec, Times, LinRec, Uncons}` and the `GLYPHS` rows `('&', PrimRec), ('.', Times), ('|', LinRec), ('>', Uncons)`. Confirm `print.rs` `needs_separator` treats them as self-delimiting (they are). Add lexer test vectors, incl. `3.` → `Int(3) Times`.
4. **Interpreter** (`crates/mtl-syntax` → `bench/validate` `conv` map, and `crates/mtl-core/src/interp.rs`): add the exec variants mirroring §3; extend `conv`/`conv_program` in `bench/validate/src/lib.rs`.
5. **Corpus / validate:** promote the `bench/design-v0.2/candidates/*.mtl` solutions into `bench/corpus/<task>/mtl/solution.mtl`, add I/O vectors as `#[test]` fns in `bench/validate/tests/corpus.rs` (fib, sum_to, power as new tasks; factorial/gcd rewritten), add `tasks.json` entries, and regenerate `bench/BASELINE.md` via `report.py`. Only after that are the numbers in §6 interpreter-validated rather than hand-traced.
