# v0.6 Design Round — The Static Multiplicity / Stack-Effect Checker (§14, Layer C)

Status: **design + prototype**. This round designs and prototypes the static
stack-effect checker frozen as future work in spec §14 since v0.1.1, and reports
the corpus acceptance measurement that decides whether the Layer-C proof campaign
is worth opening. Implementation-hardening and the Verus proof fan-out are
subsequent rounds, per the project pattern.

Scope is deliberately narrow and honest, following adversarial-review §20:
**Layer C — static stack typing over literal quotations only.** Programs whose
control flow is built from runtime-composed quotations (dynamic `,`/`;`/`!` on
non-literal values) are Layer D and are explicitly deferred; the checker either
`Guarded`s them behind runtime obligations or `Reject`s them with a clear reason.

---

## 1. The three judgments

The checker is a total function `check : Program -> Verdict` with

```
Verdict =
  | Static (Effect)                          -- fully inferred, no residual runtime risk
  | Guarded(Effect, obligations : [Guard])   -- inferred modulo a finite set of runtime guards
  | Reject { reason, at_word, expected, found }   -- provably faults, or outside Layer C
```

- `Static(eff)` — every reachable state is proven free of `Underflow` and
  `TypeMismatch`; the program's row-polymorphic stack effect is `eff`. This is
  the judgment the P9 static-soundness theorem attaches to.
- `Guarded(eff, obls)` — the effect is inferred, but soundness depends on a
  finite set of runtime guards discharging residual obligations (an opaque
  quotation whose arity is only known at run time, a host `Call` result of
  unknown kind, an `uncons` on an unknown-length quote reconciled at the
  following `?`). This is the P9 guard-insertion judgment (Layer D contract).
- `Reject{...}` — the program either provably faults (`[1]+` -> `TypeMismatch`) or
  falls outside Layer C's decidable fragment (self-applicative `:!` open
  recursion). The reason carries the failing word and the expected/found stack
  shapes, formatted for LLM repair (§7).

Verdict aggregation over a program: `Reject` dominates; otherwise any recorded
guard obligation yields `Guarded`; otherwise `Static`.

### Effects are row-polymorphic

An `Effect` is inferred as `forall p. p ++ pre -> p ++ post` where `pre`/`post`
are lists of abstract cells and `p` is an unknown-but-sufficient tail. A program
that consumes one integer and leaves its factorial has `pre = [Int]`,
`post = [Int]`. "Static" is a statement about *inference succeeding without
residual risk*, not about `pre` being empty — this matches the corpus, where
reusable words (`[1][*]&`) are stored without their inputs and run with inputs
prepended.

### The abstract value lattice

```
AbsVal =
  | Int              -- definitely an integer
  | Lit([Word])      -- definitely a quotation whose literal body is known (constant-folded)
  | OpaqueQuote      -- definitely a quotation, body unknown (built from Any / borrowed / host)
  | Any              -- kind unknown: host Call result, or a join of Int with a quote
```

The crucial refinement over a naive stack-height checker: **`Lit` tracks the
known body of a literal quotation and the checker constant-folds quotation
construction.** `Cat` of two `Lit`s is a `Lit`; `Cons` of a value onto a `Lit`
is a `Lit`; the tail produced by `uncons` on a `Lit` is a `Lit`; a `Lit` applied
by `!` is inlined. Opacity (`OpaqueQuote`/`Any`) only arises when a value's
identity is *genuinely* unknown — a borrowed input applied as code, a host
result, or a quote assembled from an `Any`. This is what lets Layer C accept
real combinator-style programs instead of collapsing at the first `,`.

(Implementation note: the prototype carries an internal `Base(i)` provenance
handle for borrowed base cells so that a later typed use — e.g. a `+` after a
`swap`/`over` — refines the borrowed input's required type in place, and `Dup`'d
copies share the slot. It is resolved away before any `Effect` is reported. This
is a mechanism under the same design, not a separate rule.)

## 2. How each primitive types (the 23 + literals)

Rules are derived to match `crates/mtl-core/src/interp.rs` `exec_prim` exactly —
soundness is relative to that reference semantics. `join` is the per-cell type
join (`Int join Int = Int`, `quote join quote = OpaqueQuote`, otherwise `Any`).

### Literals and value shuffles — always Static

| word | glyph | rule |
|---|---|---|
| PushInt | — | push `Int` |
| PushQuote | `[...]` | push `Lit(body)` |
| Dup | `:` | pop `x`; push `x`, push `x` |
| Drop | `_` | pop 1 |
| Swap | `~` | pop `a b`; push `b a` |
| Rot | `@` | pop `a b c`; push `b c a` |
| Over | `^` | pop `a b`; push `a b a` |

Pops borrow from the polymorphic base `p` when `cells` is empty, recording the
required cell in `pre`. Shuffles impose no type constraint, so borrowed cells
stay polymorphic.

### Arithmetic / comparison — Static, or TypeMismatch-Reject, or Any-Guarded

`Add Sub Mul Div Mod Xor Eq Lt` (`+ - * / % $ = <`) pop two operands and push
`Int`. Each operand:

- proven `Int` (or a base-borrow, required `Int`) -> fine, **Static**;
- a `Lit`/`OpaqueQuote` -> the run-time interpreter faults `TypeMismatch`, so the
  checker **Rejects** (provable fault) with expected `[Int, Int]`, found the
  quote;
- `Any` (e.g. a host result) -> cannot *prove* `Int`, so **Guarded** with a
  runtime type-guard obligation; result `Int` is still pushed.

Arithmetic-value faults (`Overflow`, `DivByZero`) are **out of scope for Layer C
progress** by review §20 — the static theorem excludes them; they remain runtime
faults.

### Combinators over literal quotations

- **Apply `!`** — pop the top quote. `Lit(body)` => **inline** `interp(body)` in
  the current context (this is how the effect is obtained in situ). Opaque/`Any`
  => **Guarded** with an arity obligation and an unknown resulting shape (which
  usually forces a later `Reject` if a concrete downstream shape is needed).
- **Dip `'`** — pop quote `q`, save cell `x`, run `q` on the rest, restore `x` on
  top. `Lit` body => Static; opaque => Guarded.
- **If `?`** — pop `[f] [t] c`. `c` must be `Int` (a quote => TypeMismatch
  Reject). With both branches `Lit`, infer each branch's effect on a copy of the
  stack and require **equal net height delta** with joinable per-cell outputs
  (branch-stack compatibility). Compatible => apply the joined effect, Static.
  Unequal deltas => **Reject** `branch-stack incompatibility: then Δ=X, else Δ=Y`.
  An opaque branch => Guarded.

### Quotation construction — constant-folded

- **Cat `,`** — both `Lit` => push `Lit(a++b)` (folded, order matching
  interp.rs); else `OpaqueQuote` + Guarded.
- **Cons `;`** — `Lit` quote + a value cell => push the folded `Lit`; else
  `OpaqueQuote` + Guarded.

### `uncons >` — the §14.5 branch-dependent shape

`uncons` pops one quote and is a sum-typed deconstructor. On a **`Lit`** operand
the length is statically known, so it is *deterministic*, not branch-dependent:

- empty body -> push `Int(0)` (Static);
- non-empty body -> the first word must be a **value word** (`PushInt`->`Int`,
  `PushQuote`->`Lit`); a bare `Prim`/`Call` head => **Reject `TypeMismatch`**
  (malformed head, matching interp.rs); otherwise push `head`, `Lit(tail)`,
  `Int(1)` (Static).

On an **`OpaqueQuote`/`Any`** operand the two success branches have different
heights (`head [tail] 1` vs `0`) — the genuine §14.5 case. The checker models
this as a **tagged sum of shapes** and relies on the *following `?`* (whose flag
is exactly the discriminant) to reconcile: the empty-handler is checked against
the empty shape, the non-empty-handler against `head [tail] 1`, and their outputs
joined (equal net delta required). If it reconciles => **Guarded** (runtime
length still discriminates). If an opaque `uncons` is *not* immediately
reconciled by a matching `?` => **Reject** `branch-dependent stack shape at
uncons; opaque quote length unknown`.

> Measurement note (§4): the corpus does exercise the opaque path — 4 programs
> (`max_list`/`min_list` in v0.2 and v0.3) `uncons` the input list at top level
> with `>_` (uncons then drop the flag), a genuine unknown-length branch-dependent
> shape, and Reject at this §14.5 boundary because `>_` is not the
> `>[emptyH][nonemptyH]?` idiom the peephole recognizes. Every other `uncons` is
> either over a literal quote (deterministic, Static) or encapsulated inside a
> `linrec`/`fold` body (Guarded, not unrolled), so the peephole sum-shape
> reconciliation was never the deciding rule on a real program.

### Recursion combinators — effect rules (no unrolling)

Combinators are checked by an **effect rule on the literal body**, never by
unrolling `n` times — this is exactly why they stay decidable where raw `:!`
recursion does not.

- **Times `.`** `n [Q] .` — require `Q` `Lit`; infer its effect on a fresh
  polymorphic stack; require **net-zero height and per-cell type-stability**
  (each touched cell's out-type equals its in-type) so iteration is a fixpoint.
  Stable => pop `n` and `[Q]`, leave the touched region unchanged, Static.
  Non-neutral => **Reject** `times body must be stack-neutral for a static shape;
  found Δ=...` (its post-height is runtime-`n`-dependent).
- **PrimRec `&`** `n [I] [C] &` — require `I`,`C` `Lit`. In the reference
  semantics (`spec_step`) the runtime counters pile up **BELOW** the accumulator,
  not on top of it, so the stability conditions are **frame-polymorphic**:
  - `I` analyzed on the **EMPTY** stack yields the accumulator region `acc`
    (purely productive — it appends `acc` to any stack); and
  - `C` analyzed on `[counter:Int] ++ acc` — i.e. one counter cell sitting
    **below** the accumulator — yields `acc` again, height- and type-stable.

  So `C`'s effect is `[counter:Int] ++ acc -> acc` (counter below), **NOT**
  `(acc, counter) -> acc` with the counter on top. Stable => run `I` once for the
  output shape (leaving `base ++ acc`), Static; unstable => Reject with the
  instability. (This is exactly how `check_m1`'s `PrimRec` arm and
  `lemma_primrec_splice` are mechanized in `checker_verus.rs`.)
- **Fold `(`** `[seq] init [C] (` — require `C` `Lit` and `(acc elem -> acc)`
  stable (`acc` is the single `init` cell; per `spec_step`, at each step the
  element is pushed **above** `acc`, so `C : [acc] ++ [elem] -> [acc]`). A `Lit`
  `seq` yields **Static only when its elements are HOMOGENEOUSLY typed** — e.g.
  every element a `PushInt`, so each iteration pushes the *same* `Int` element
  cell that `C`'s stability was checked against. A `Lit` `seq` with
  **heterogeneous or opaque element kinds is Guarded** (the element cell is
  modeled as `Any`, so a `C` that type-constrains it — e.g. `[+]` — carries a
  runtime type-guard) or needs an element-type join before it can be Static.
  `seq` opaque => Guarded (runtime length); `C` unstable => Reject. Fold
  *encapsulates* the branch-dependent element deconstruction that raw `uncons`+`?`
  exposes — this is the combinator's static-checkability payoff. (The
  homogeneous-`PushInt` literal-seq slice is the fragment proven Static in
  `checker_verus.rs`; heterogeneous/opaque folds are outside the mechanized
  Static fragment.)
- **LinRec `|`** `[P][T][R1][R2] |` — require all four `Lit`; the terminal path
  `T` and the recursive path `R1 · recurse · R2` must produce the **same output
  shape**. Provably shape-stable => Static; an opaque path => Guarded;
  incompatible => Reject. Conservative: prefer Guard/Reject over an unsound
  Static.

### `Call` (host words, tier-3)

`Call(name)` never executes in-core — it yields `Invoke`. The checker looks up
the capability's declared effect `CapabilitySig { consumes, produces }`: pop
`consumes`, push `produces` cells as `Any`, and record a host-seam **Guarded**
obligation (the result kinds and host fault contract are only known at run time).
Host programs are therefore `Guarded` by construction under Layer C — a truthful
boundary, since end-to-end soundness needs the host contract (P9's "host
conformance", Layer F).

### Self-application `: !` — outside Layer C, Rejected

`: !` (Dup then Apply) applied to a literal quote inlines that quote's body; when
the body ends in `: !` referring to the value it left on the stack, inlining
recurses without bound — the open-recursion / Turing-completeness idiom (Layer B
universality). The checker **bounds inline depth and detects re-entrant
application of a body already being inlined**, and on exceeding it returns

```
at word 4: expected bounded, non-self-applicative program; found re-entrant
application of a quote already being applied — self-applicative / unbounded
recursion — outside Layer C (use a recursion combinator)
```

This is the honest Layer-C answer, and it is a *feature of the argument*: the
v0.1 `:!`-recursion idioms Reject, while the v0.2/v0.3 combinator idioms that
replace them are statically checkable (§4). The recursion combinators exist
precisely to move programs from the Rejected fragment into the Static one.

## 3. Theorem statements (the proof-campaign contract)

Stated against `spec_step` (`crates/mtl-core/src/mtl_core.rs`). These are
*statements*, not proofs; the proof fan-out is a later round. Each is tagged
Layer-C-provable (this campaign) or deferred.

**T-Static — static soundness (P9 first bullet; Layer C, PROVABLE).**
If `check(p) = Static(eff)`, then for every initial `SpecState` whose stack
satisfies `eff.pre` (depth and cell-kind requirements), no `spec_step`-reachable
state is `Fault(Underflow)` or `Fault(TypeMismatch)`; and if execution halts, the
final stack satisfies `eff.post`. (`Overflow`/`DivByZero` excluded.)

**T-Progress — preservation + progress (Layer C, PROVABLE).**
Under `check(p)=Static`, every non-halted reachable state either steps or is an
arithmetic fault; it is never *stuck* at an `Underflow`/`TypeMismatch` precedence
point. (The standard preservation/progress decomposition of T-Static; the useful
form for the mechanized proof.)

**T-Branch — branch-stack compatibility (Layer C, PROVABLE).**
For every `?` reached under `check(p)=Static`, the then/else branch effects have
equal net height delta and joinable per-cell output types, so the post-`?` stack
shape is well-defined independent of the flag.

**T-Guard — guard-insertion soundness (P9 second bullet; Layer D, DEFERRED).**
If `check(p) = Guarded(eff, obls)`, then a run with the inserted runtime guards
either satisfies `eff` or halts at a *defined* guard fault — never at an
`Underflow`/`TypeMismatch` inside a primitive. Depends on effect-carrying dynamic
quotation values or runtime guards (Layer D); not attempted this campaign.

**T-HostConf — host conformance (P9 third bullet; Layer F, DEFERRED).**
End-to-end soundness of `Guarded` host programs *given* the §8.2 host contract.

**T-Resource — normal-exit resource closure (P9 fourth bullet = P8c; Layer F,
DEFERRED).** A statically checked program that halts normally leaves no linear
resource unconsumed. Requires the heap/resource semantics (Layers E/F).

Layer-C-provable this campaign: **T-Static, T-Progress, T-Branch**, over the
literal-quotation fragment. The bundling review §12 warned against ("a proof
obligation shaped like a small moon") is avoided: each theorem is a separate
milestone, and the deferred three are explicitly not on the Layer-C critical
path.

## 4. Corpus acceptance table (the design verdict)

The checker was run over every corpus program: the v0.1 / v0.2 / v0.3 solution
sets under `bench/corpus/*`, the tier-3 host solutions under `bench/tier3`, and
the LLM-generated agent-trial programs under `bench/agent-trial`.

| Tier | Static | Guarded | Reject | Total |
|------|-------:|--------:|-------:|------:|
| v0.1 | 3 | 0 | 2 | 5 |
| v0.2 | 8 | 8 | 2 | 18 |
| v0.3 | 1 | 8 | 2 | 11 |
| tier-3 | 0 | 8 | 0 | 8 |
| agent-trial | 15 | 15 | 7 | 37 |
| **overall** | **27** | **39** | **13** | **79** |

Overall: Static 27/79 (34%), Guarded 39/79 (49%), Reject 13/79 (16%).

Reading the verdict: the checker handles 66/79 (84%) of the corpus — 27 Static
outright, 39 Guarded behind runtime obligations — and Rejects 13 (16%). The
Rejects are principled, not coverage failures: 6 are malformed LLM output (parse
errors, agent-trial); 1 is a real `TypeMismatch` bug the checker caught before
execution (`1[*]&`, a malformed factorial where `primrec` got an `Int` where
`[I]` must be a quote — the static verdict matches the trial's recorded runtime
`error_type: TypeMismatch` exactly); 2 are the `:!` open-recursion idiom (v0.1
factorial and gcd, outside Layer C by construction); and 4 are the §14.5
opaque-`uncons` boundary (`max_list`/`min_list` in v0.2 and v0.3). Tier-3's 8/8
Guarded is the host seam behaving exactly as designed.

Critically, the split tracks the language's evolution: the combinator-based
idioms (v0.2/v0.3, tier-3) are what the checker can type; the raw `:!` recursion
of v0.1 is what it Rejects. That is a direct measurement that the recursion
combinators moved real programs out of the undecidable fragment and into the
checkable one — the "genuine can't-handle-a-valid-program" rejects total 6/79
(≈8%: two `:!`, four opaque-`uncons`), each a principled boundary.

Full per-program verdicts: `bench/design-v0.6/corpus-verdicts.md`.

### Soundness smoke test (fault corpus)

There is no standalone fault-corpus directory (negative behavior lives in
`crates/mtl-core/tests/interpreter.rs`), so the smoke test uses 12 canonical
faulting one-liners split into 7 `TypeMismatch` cases (must `Reject` regardless
of inputs) and 5 `Underflow` cases (must not check `Static`-on-empty), plus a
universal invariant: for **any** `Static` verdict, running the reference
interpreter on a `pre`-shaped stack must not fault `Underflow`/`TypeMismatch`.
All pass (`tests/soundness.rs`, 2 suites green). This is the negative half of the
soundness argument: the checker never blesses a program the interpreter faults on
for a Layer-C reason.

## 5. Zero-program-token-cost claim — confirmed, with one nuance

§14's original pitch was that the checker costs no program tokens because it reads
existing structure. For this design:

- **`check_static`** reads the AST and adds nothing to the source — **zero
  program tokens**, confirmed.
- **`check_guarded`** inserts guards, but the guards are a **runtime mechanism**
  (a side-table of obligations consulted by the VM at recorded guard points), not
  source rewrites. The LLM never writes or reads a guard, so the *program*-token
  cost is still **zero**; the cost is runtime instructions, invisible to the
  token budget.
- **`Reject`** emits a repair message (§7), which costs *response* tokens on the
  repair turn but zero *program* tokens.

So the claim holds as stated for program tokens, with the clarification that
"guard insertion" means runtime side-table entries, not injected source. The only
token cost anywhere is the repair message on a rejection — which is the point:
it is the same expected/found stack-state feedback the interpreter already gives
on a runtime fault, just delivered before execution.

## 6. Layer-D sketch (deferred — the next design round's seed)

Layer D lifts the literal-quotation restriction: quotation values carry (or are
guarded against) their stack effects at run time.

- **Effect-annotated quote values.** `Value::Quote` gains an inferred effect
  slot; `Cat`/`Cons`/`Apply` compose/consume effects dynamically. `Static`
  extends to programs where composition is effect-monotone.
- **Runtime arity guards.** Where the static effect is unknown, the VM inserts an
  arity/kind check at the guard point (from the `Guarded` obligation table); a
  failed guard is a *defined* `GuardFault`, not an `Underflow` mid-primitive.
  T-Guard is the soundness statement for this.
- **Gradual guarantee.** A fully-literal program checks `Static` and runs without
  guards; adding dynamic composition only ever *adds* guards, never changes a
  `Static` program's behavior (the review §20 Layer-D gradual-guarantee target).
- **The `uncons`/`?` sum-shape** generalizes from the peephole (§2) to a
  first-class shape-sum in the abstract domain, so branch-dependent shapes
  compose through dynamic code — this is what would convert the four §14.5
  opaque-`uncons` Rejects into checkable programs.

## 7. Error messages for LLM repair

The checker's rejection/guard messages meet the same bar as the interpreter's
runtime stack-state faults: they name the **failing word (glyph + index)** and
show the **expected vs found stack shape** at that point, so an LLM can repair in
one turn without executing.

Format: `at word <i> '<glyph>' (<Prim>): expected <shape>, found <shape> — <fault>`

Real messages emitted by the prototype:

```
[1]2+       at word 2: expected [Int, Int] on top for + (Add); found [Quote, Int] — TypeMismatch
1!          at word 1: expected [Quote] on top for ! (Apply); found [Int] — TypeMismatch
[+]>        at word 1: expected quote whose head is a value word for > (Uncons); found quote whose head is a bare Prim/Call — TypeMismatch
1 1[:][_]?  at word 4: expected both branches to have equal net stack height delta; found then Δ=1 vs else Δ=-1 — branch-stack incompatibility
```

Each message is actionable without running the program: the first says "you put a
quote where two integers were needed"; the branch-incompatibility message pins the
exact height mismatch between the two arms of a `?`.

## 8. Implementation plan (subsequent rounds)

1. **Harden the prototype** (`crates/mtl-check`): promote from prototype to a
   first-class crate; property-test the checker against the interpreter (any
   program checking `Static` must never fault `Underflow`/`TypeMismatch` under
   `run` on inputs matching `pre`) — executable validation of T-Static before the
   mechanized proof.
2. **Mechanize T-Static / T-Progress / T-Branch** in Verus over `spec_step`,
   literal-quotation fragment (Layer C). Six milestones per review §20, not one.
3. **Layer D** (§6) as its own design round once Layer C is proven.
4. **Wire into the agent loop**: surface `Reject` messages to the LLM as
   pre-execution repair hints; measure repair-rate lift vs runtime-fault-only
   feedback (MTL's actual purpose).

## 9. Recommendation

**Yes — Layer C earns the proof campaign.** T-Static covers 34% of the corpus
outright, and the checker's judgment space (Static + Guarded) covers 84%; the 16%
Reject is dominated by malformed LLM input and two principled boundaries (`:!`
open recursion, opaque `uncons`), plus one real bug the checker caught
pre-execution. A checker that rejected half the working corpus would signal a
broken design; this one Rejects ≈8% for principled reasons and blesses or guards
the rest.

Recommend proceeding to mechanize **T-Static, T-Progress, T-Branch** in Verus
over the literal-quotation fragment (six milestones per review §20), and opening
**Layer D** (§6) as the next design round — its first target is converting the 39
Guarded programs, especially the four §14.5 opaque-`uncons` cases, into
statically checkable ones.
