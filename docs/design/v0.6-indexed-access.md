# MTL v0.6 — Indexed / random access into sequences (measurement-driven decision, supersedes v0.3 §9)

- Status: **design stage — architecturally-significant decision record for the maintainer.** This is the authoritative disposition of indexed access; it supersedes the v0.3 three-option analysis (`docs/design/v0.3-sequences.md` §9, full workings in `bench/design-v0.3/indexed-access/`), which chose (c) "accept the niche" with an (a) hedge and held (b) "until P2 is discharged." Two measurement spikes have since run and are committed under `bench/design-v0.6/`. No verified file (`crates/`), no frozen baseline, and no spike directory is modified by this PR — this doc is the only addition.
- Branch: `v06-indexed-design`. Tokenizers: tiktoken `o200k_base` + `cl100k_base` at 0.8.0 (the pinned bench set). These programs are punctuation-dense, so both encoders tokenize every candidate string **identically** — one number per cell below.
- Companion evidence (read these; every validated number here cites them):
  - `bench/design-v0.6/indexed-access-spike/` — option (a), a real cons-list interpreter (`spike.rs`) with the two candidate programs (`binary_search.mtl`, `two_sum.mtl`) run against honest I/O vectors (`vectors.md`, `validate.sh`).
  - `bench/design-v0.6/capability-spike/README.md` — option (d), the `invoke_host` mechanism validated end-to-end (`mtl_core::host::drive`), with the name-overhead token measurements.
  - `bench/BASELINE-TIER2.md` — the shipped scenario-B headline (3.87× / 3.92×).

## Executive summary

**The niche thesis is now measured-proven, not projected.** The v0.3 doc *estimated* that unblocking two_sum/binary_search would be compression-neutral-to-dilutive; the option-(a) spike's real interpreter now *measures* it: adding the two tasks halves the tier-2 aggregate from 3.87×/3.92× to **2.00×** (both encodings), and two_sum lands at **0.59×** — a task where MTL is strictly *dearer* than Python. **No indexed-access mechanism earns its cost, because the cost is not access-bound.** The load-bearing finding from the (a) spike is that these two tasks are *juggling-bound*: the token count is dominated by the deep point-free shuffling of a 4-mutually-dependent-value carried state, not by the index operation — so swapping the O(n) primitive (a) for a `Value::Vec` (b) or a host capability (d) barely moves the number (the (d) spike confirms +1–2 tokens on the identical skeleton). **Recommendation, up front: decline core changes — option (c) is the standing answer.** Keep the reported headline at scenario B (3.87×/3.92×, coverage 11/13). If a future array-heavy corpus ever makes coverage genuinely worth buying, reach for **capabilities (d)**, never (a) or (b): (d) delivers correct algorithmic complexity at the same token price with zero proof risk and zero glyph spend.

## 2. The measurement (this is the heart — lead with it)

### 2.1 The validated per-task table

The v0.3 numbers were **hand-traced design estimates**, explicitly marked "not interpreter-validated." The v0.6 (a) spike is a real cons-list stack machine mirroring `interp.rs::exec_prim` plus the two option-(a) primitives (`nth` = O(n) head-walk, OOB pushes `Int(0)`; `len`); it runs the candidate programs against the I/O vectors in `vectors.md`. These are the numbers that matter:

| task | python idiomatic | option (a) — **VALIDATED** | option (d) | ratio (a vs py) | true complexity under a/b/d |
|---|---:|---:|---:|---:|---|
| two_sum | 48 | **81** | ~82 | **0.59× — MTL LOSES** | O(n²) brute force under all of a/b/d |
| binary_search | 83 | **70** | ~71 | **1.19×** | (a) = O(n·log n) **FAKE** (`nth` re-walks the head each probe); (d)/(b) = true O(log n) |

- option (a): interpreter-validated (`bench/design-v0.6/indexed-access-spike/`, `binary_search.mtl` 70 tok / `two_sum.mtl` 81 tok, both encoders identical — re-derivable with `printf '%s' "$(cat two_sum.mtl)" | python3 bench/tokcount/tokcount.py`).
- option (d): mechanism-validated end-to-end (`bench/design-v0.6/capability-spike/README.md` §3 — every host-serviced program HALTs with correct output); the per-task token figure is the (a) skeleton plus the measured name/space overhead (§2.3, +1–2 tokens), i.e. juggling-bound and therefore ≈ (a).

### 2.2 The buggy-estimate correction (state this plainly)

The v0.3 doc's estimates were **~2× optimistic and unvalidated**, and worse, the candidate program it carried was **wrong**:

| task | v0.3 estimate | v0.6 VALIDATED | what went wrong |
|---|---:|---:|---|
| binary_search | 37 (with a "band 30–38") | **70** | the v0.3 reference program **faults / returns the bare `xs` quote on every vector** — the (a) spike proved it never computes the index. A *correct* program measures 70. |
| two_sum | ~34 | **81** | the ~34 figure was a schematic that never built the `[i j]` result over the real four-deep carried state; the correct validated program is 81. |

This is not a rounding disagreement. The v0.3 numbers understated the true cost by ~1.9× on binary_search and ~2.4× on two_sum, and the program that produced the optimistic binary_search figure does not solve the task. The (d) spike's own README repeats the caveat: running option (a)'s exact reference structure in the host harness "returns 0 for `bsearch([1,3,5,7,9],7)` (should be 3)." The lesson is the same one `BASELINE-TIER2.md` insists on — no hand-traced number survives contact with the interpreter. Every number in this doc is validated or explicitly flagged as a structural delta on a validated skeleton.

### 2.3 The load-bearing finding: the cost is JUGGLING-BOUND, not access-bound

This is the finding that decides the whole question. **The token cost of these two tasks is dominated by point-free stack shuffling, not by the index operation.** Both tasks carry a state of **four mutually-dependent values** — the read-only `xs` quote, the `target`, and two moving indices (`lo`/`hi` for binary_search, `i`/`j` for two_sum). The `xs` quote sits near the *bottom* of that stack because it is read-only and never consumed, so every single index read must dredge it up from underneath the live indices with a deep point-free pick chain. The idiom that recurs throughout `binary_search.mtl` and `two_sum.mtl` is:

```
[[[^]'~]'~]'~        -- reach a value buried three deep, non-destructively, and restore the stack
```

nested `dip`-quote/`swap` towers, repeated at every probe and every comparison. **That shuffling — not the `nth` primitive — is what the tokens are spent on.** Two consequences follow directly:

1. **No indexed-access mechanism rescues these tasks on tokens.** Whether `xs[mid]` is fetched by an O(n) cons-walk primitive (a), an O(1) `Value::Vec` `get` (b), or an O(1) host `nth` capability (d), the *surrounding* pick chain to get `mid`, `lo`, `hi` and the buried `xs` into position is unchanged. The (d) spike hand-traced the identical skeleton with the primitive swapped for a capability call and measured **+1–2 tokens** total (a single forced space in `len 1`). The cost is therefore **roughly invariant across (a)/(b)/(d)**.
2. **The complexity property and the token property are decoupled.** (a) gives the *cheapest* binary_search (70 tok) but only a *fake* O(n·log n) bisection (its `nth` re-walks the list head on every probe). (d)/(b) give a *true* O(log n) at essentially the same token price. So the only thing a value-model or capability change buys over (a) is **algorithmic correctness**, never tokens.

The v0.3 writability sections already gestured at this ("nested point-free loops with 3–4 carried values … still demand deep blind stack routing … `nth`/`len` shrink the tax; they do not remove it"). The v0.6 measurement confirms it quantitatively: the tax is the whole cost, and it is mechanism-independent.

## 3. Per-option analysis

### 3.a — O(n) `nth`/`len` primitives on the cons-list

- **Mechanism.** Two new primitives on the existing `Quote` value: `len` (`[xs] -- n`) and a flagged `nth` (`[xs] i -- x 1` on a hit, `-- 0` on OOB — OOB is *data*, not a fault). No value-model change; a list is still a `Quote`. Semantics sketch is in `bench/design-v0.3/indexed-access/README.md` §3.a and the mirror in `spike.rs::exec_prim`.
- **Token cost + true complexity.** binary_search **70** (validated), two_sum **81** (validated). Delivers only **O(n·log n)** "binary search" — `nth` walks the cons-list from the head on every probe, so the bisection is a linear-per-probe scan wearing a logarithmic costume. two_sum is O(n²) (as it is under every option). Must be labelled honestly as a bisection *scan*, not a true binary search.
- **Cost to the system.** Stays within the primitive-mirror regime, but that regime is not free: **7 Prim mirror surfaces** (#32) + manifest count **23 → 25** (trips every `[_; 23]` fixed-array guard and the `manifest_shape` assert) + **2 scarce ASCII glyphs** from a table that is nearly saturated (`$`/`(` already spent on `xor`/`fold`). A fault decision is required: either add a new `IndexOutOfBounds` fault to all three fault enums and slot it into the positional precedence order, *or* use the v0.3 "flagged `nth` returns data `0`, no new `Error` variant" trick (cheaper; recommended if (a) is ever built). Plus new `spec_step_prim`/`exec_prim` arms, the P2 refinement lemmas, and two smoke theorems. It touches **no value-termination proof and no arena `Copy` break** — this is its one genuine virtue, and it is exactly the machinery that shipped for `uncons`.
- **Scenario-C impact.** Drives the aggregate to 2.00× (§4); two_sum at 0.59× is a net loss.
- **Writability.** The tax-shrinking-not-removing verdict from v0.3 §8 is now confirmed by measurement. `len`/`nth` remove the fold-to-recompute-length and the hand-threaded index counter, but the four-deep carried state's pick chains (`[[[^]'~]'~]'~`, §2.3) remain — and those are precisely the "blind spatial routing" the adversarial review names as the dominant LLM failure mode. Even *with* indexing, these programs are dense point-free juggling that an LLM finds hard to write and hard to read; a fold-style solution (where MTL actually wins) stays legible by comparison. Hand-golfing a correct two_sum at design stage was itself error-prone — a writability data point *against* these tasks regardless of the primitive.

### 3.b — a `Value::Vec` value type (O(1) true random access)

- **Mechanism.** A new `Value::Vec(Seq<Value>)` variant + `get`/`set`/`len` + `{ }` array literals. True O(1) random access and O(log n) binary search.
- **Token cost + true complexity.** Token cost is **the same juggling-bound ~70/~81 neighbourhood** as (a)/(d) (§2.3 — the pick chains dominate; O(1) `get` does not shrink them). Delivers true O(1)/O(log n). So (b) buys, over (d), **nothing**: identical tokens, identical complexity, at vastly higher cost.
- **Cost to the system — the FULL DRIFT BILL.** (b) is strictly a *superset* of (a)'s surface **plus the entire verified-value-model reopening.** Enumerated from the internals recon:
  - Reopens the value model across **5** `Value`/`SpecValue` declarations, **2 of which are Verus proof roots** in `crates/mtl-core/src/mtl_core.rs`: the exec `Value` (@616) and the ghost `SpecValue` (@48) — plus the `mtl-check` `AbsVal`/`Kind` lattice.
  - Makes the currently **non-recursive** `view_value` (`mtl_core.rs:664`) **mutually recursive**, which **REOPENS the deep-view lexicographic `decreases` termination measure** (`view_word`/`view_words`/`view_value`/`view_stack`, `mtl_core.rs:643–682`) — the file's single most delicate, hard-won lemma (it is what closed the nested-quotation hole). This is the crux cost.
  - Forces new trusted `Clone` `ensures`.
  - **Breaks the arena's `Copy` 16-byte value representation.** `Value::Quote(QuoteId)` is an interned-tape id; a `Value::Vec` either needs a *second* interned `VecId` tape or loses `Copy` and forces a rewrite of the 559-line `vm.rs` stack machine.
  - Threads a new variant through **all 7 policed mirrors** (parser `ast.rs`, ghost `mtl_core.rs`, interp `interp.rs`, `GLYPHS`, the two `conv` fns in `bench/validate` + `mtl-host`, arena `types.rs`).
  - **Enlarges the P4 print/parse round-trip PROOF** (`p4_verus.rs`, 2694 lines) with new `{ }` literal delimiters — and P4 is now *closed* (level (c), 101/0), so this reopens a shipped proof.
  - **Re-runs P5** (`p5_universality.rs`, 1710 lines — it pulls `mtl_core` unmodified today).
  - Adds a new well-formedness invariant **I3′** (recursive Vec).
  - Requires a new **`IndexOutOfBounds` fault** in all **3 fault enums** (ghost `Error`@`mtl_core.rs:58`, interp `Fault`@`interp.rs:92`, arena `Fault`@`types.rs:163`), slotted into the positionally-encoded precedence order (arity → type → DivByZero → Overflow) in **both** `spec_step_prim` and `exec_prim`, re-proving every fault-order refinement `ensures` and updating the 5-case fault-parity corpus.
  - **Breaks the affine story:** `get` copies/replicates and `set` is a persistent functional update — unlike `uncons`'s clean "consumed once" reading.
  - Net: weeks, not days; reopens the two hardest closed proofs in the repo (deep-view termination, P4 round-trip) and the arena `Copy` invariant, for a result (d) achieves with zero proof risk.
- **Scenario-C impact.** Same 2.00× neighbourhood as (a)/(d) — the dilution is mechanism-independent (§4). (b) pays the most and moves the headline exactly as little.
- **Writability.** No better than (a): the carried-state depth is identical, so the pick-chain tax is unchanged, and the author must additionally learn a *second* aggregate syntax (`{ }` vs `[ ]`) and when to use which. Net writability ≈ (a); the only win is runtime complexity, which (d) also delivers.

### 3.c — decline; document the niche

- **Mechanism.** None. two_sum/binary_search remain documented as out-of-niche walls (they are *complexity*-shaped, not strictly I/O-inexpressible — see the v0.3 note that binary_search is reachable today as a linear index-of scan).
- **Token cost + complexity.** N/a — no MTL denominator; the tasks stay excluded from the aggregate, holding the headline at scenario B (3.87× / 3.92×), coverage 11/13.
- **Cost to the system.** Zero — no prim, no variant, no glyph, no proof, no fault, no host code.
- **Scenario-C impact.** None by construction (the two tasks are not admitted into the aggregate).
- **Writability.** No new surface for the model to get wrong. The corpus stays in MTL's actual niche — control-flow/recursion (`T_v0` 3.72×, tier-2 fold-set 3.87×) — where fold-style solutions are legible and short.

### 3.d — host-side sequences via capabilities

- **Mechanism.** `nth`/`len`/`slice` are provided **not** as a core value type or core primitive but as **host capability calls** over the existing v0.4 effects seam — the same "strings pattern" MTL already uses. The core stays exactly `Int | Quote`; a sequence is an opaque `i64` handle on the stack (a `Value::Int`), and the host owns the `Vec<i64>`. A capability call is a bareword `Call(name)`; the pure core suspends at every `Call` and yields `Outcome::Invoke{name, stack}`, which `HostShim::service` grant-checks, meters, runs, and resumes (`crates/mtl-core/src/host.rs`, `crates/mtl-host/src/core_bridge.rs`, `capability.rs`). **Direct precedent:** the existing `select` capability (`crates/mtl-host/src/caps/mod.rs:293-310`) is *already* host-side `nth` over a quote of ints.
- **Token cost + true complexity.** binary_search ~71, two_sum ~82 — i.e. **+1–2 tokens over (a)** on the identical skeleton (mechanism-validated, `capability-spike/README.md` §4–5). Delivers **true O(1)** access → **true O(log n)** binary search, which (a) structurally cannot.
- **Cost to the system.** **ZERO core/proof/glyph change.** No new `SpecValue` variant, no new `SpecPrim` arm, no new `Error`, no P1–P5 obligation, no glyph consumed. The affine/verification story is untouched — the core still only copies/drops/shuffles `Int` handles. All new work is in the **unverified host TCB**: a host-side sequence store (a `Vec<Vec<i64>>` mirroring `HandleTable`), 3 cap registrations (`nth`/`len`/`slice`, ~15 lines each), and a `readseq` input capability with declared effects/faults/budgets. The cost is host-runtime complexity, not proof risk.
- **Token-name caveat (measured, matters).** A host call costs the tokenization of its *name word* + any forced whitespace. Short lowercase names (`nth`, `len`) cost **1 token — the same as a glyph** — and a name abutting a *glyph* MERGES to 1 token (`:nth` = 1). The only overhead vs a single-char glyph is a **forced space when a name abuts a digit or another name** (`^len 1-` vs `^#1-` = 5 vs 4). So name choice is load-bearing: `nth`/`len` cost the same as a glyph; a 4-letter `nthc` tokenizes *worse*. Names never consume a scarce ASCII glyph — a real plus given the near-saturated table.
- **Scenario-C impact.** Same 2.00× neighbourhood (§4) — because the cost is juggling-bound, (d) dilutes exactly as much as (a)/(b).
- **Writability.** Same pick-chain tax as (a)/(b) on the carried state — no mechanism removes it. But (d)'s **named** capabilities (`nth`, `len`, `slice`) read slightly better than opaque single-char glyph prims: a bareword the model already knows conceptually is a marginally stronger prior than a scarce ASCII glyph. Still: even with capabilities these remain dense point-free programs an LLM finds hard to author; the fold-style corpus stays the legible one.

## 4. Aggregate / decision criterion

The whole decision turns on which single number the corpus is optimizing, so state both framings explicitly.

| scenario | tasks | encoding | py total | mtl total | aggregate | coverage |
|---|---:|---|---:|---:|---:|---:|
| **B (shipped headline)** | 11 (fold/xor real) | o200k | 352 | 91 | **3.87×** | 11/13 |
| **B (shipped headline)** | 11 | cl100k | 349 | 89 | **3.92×** | 11/13 |
| **C (adds the two walls), option (a) VALIDATED** | 13 | o200k | 483 | 242 | **2.00×** | 13/13 |
| **C (adds the two walls), option (a) VALIDATED** | 13 | cl100k | 480 | 240 | **2.00×** | 13/13 |

- Scenario C arithmetic: o200k mtl = 91 + 70 + 81 = 242, py = 352 + 48 + 83 = 483 → 483/242 = **2.00×**; cl100k 89 + 70 + 81 = 240, 349 + 48 + 83 = 480 → 480/240 = **2.00×**.
- **Dilution: −1.87× (o200k) / −1.92× (cl100k), ≈ −48–49% of the aggregate.** Because the cost is juggling-bound (§2.3), options (b) and (d) land in the *same* ~2.00× neighbourhood — **the dilution is mechanism-independent.** You cannot buy your way out of it with a better access mechanism.
- **Headline framing.** If the single reported number is the goal, the standing **anti-tarpit rule** decides it: a feature is admitted iff it pays for itself in corpus tokens. Indexed access does the *opposite* — it nearly halves the aggregate and makes two_sum (0.59×) a task where MTL is dearer than Python. Keep the headline at scenario B (3.87× / 3.92×), coverage 11/13 → **decline (option c)**.
- **Coverage framing.** If 13/13 coverage is the explicit goal, any of a/b/d reaches it, but *all three* roughly halve the aggregate to ~2.00×. These two tasks are array/complexity-shaped — exactly the **anti-niche**. MTL's structural edge is control-flow/recursion (`T_v0` 3.72×, tier-2 fold-set 3.87×), not positional iteration. Chasing coverage here trades the headline for two tasks the language was never going to win.

**Recommend optimizing the headline.** The anti-tarpit rule is the project's standing admission criterion, and by it indexed access does not clear the bar the way `fold` did.

## 5. Recommendation

**Decline core changes. Option (c) — document the niche — is the standing answer, and it is now measured-proven rather than projected.**

One-line reason: **the cost of these two tasks is juggling-bound, not access-bound, so no indexed-access mechanism rescues them on tokens — they halve the aggregate to 2.00× and leave two_sum a net loss (0.59×) under every option, so admitting any of them fails the anti-tarpit rule.**

**If coverage is ever genuinely wanted** (a future array-heavy corpus that shifts MTL's target niche), reach for **capabilities (d)** — never (a) or (b). (d) delivers correct algorithmic complexity (true O(1)/O(log n)) at the same token price as (a), with **zero proof risk and zero glyph spend**, moving the entire cost to the replaceable host TCB. (b) is the single most expensive option in the repo and buys **nothing (d) does not** — since the token cost is access-independent, (b)'s O(1) `get` is redundant with (d)'s O(1) capability, and (b) additionally reopens the two hardest closed proofs and the arena `Copy` invariant. (a) is proof-cheap but delivers only a *fake* O(n·log n) binary search and still spends two scarce glyphs and two manifest slots. So the only defensible coverage move is (d).

## 6. If admitted, what implementation fans out to

Recorded so the maintainer can cost each path; none of this is done in this PR.

### If (a) is admitted (proof-safe, coverage-only rationale)

Stays inside the primitive-mirror regime; mirrors the v0.3 §10 plan:
1. **7 Prim mirror surfaces** (#32): parser `ast.rs`, ghost `mtl_core.rs`, interp `interp.rs`, `GLYPHS`, the two `conv` fns (`bench/validate`, `mtl-host`), arena `types.rs` — new `Len`/`Nth` in each.
2. **Manifest count 23 → 25** — update every `[_; 23]` fixed-array guard and the `manifest_shape` assert.
3. **2 glyphs** — `('#', Len)`, `` ('`', Nth) ``; confirm `print.rs::needs_separator` treats them as self-delimiting.
4. **Fault decision** — either the flagged-`nth`-returns-`0` trick (no new `Error` variant; recommended) or a new `IndexOutOfBounds` in all 3 fault enums slotted into the precedence order.
5. **`spec_step_prim` / `exec_prim` arms** + the **P2 refinement lemmas** for both exec twins.
6. **2 smoke theorems** (`smoke_len`, `smoke_nth_*`), near-copies of the `uncons` smoke theorems. No deep-view measure change, no arena `Copy` break.
7. **Corpus/validate/baseline** — promote two_sum + binary_search (labelled as a bisection *scan*, O(n·log n)) with I/O vectors, flip `expressible:true`, regenerate `BASELINE-TIER2.md`.

### If (d) is admitted (the coverage route — all non-proof, host TCB)

Reuses the `Invoke`/`HostShim`/`Registry`/metering seam verbatim (`select` is the precedent):
1. **Host-side sequence store** — a `Vec<Vec<i64>>` / handle table for sequences, mirroring `HandleTable`.
2. **3 capability registrations** — `nth (s i -- s x)`, `len (s -- s n)`, `slice (s lo hi -- s')`, ~15 lines each, each indexing the host `Vec<i64>` in O(1) (see `capability-spike/src/main.rs`).
3. **A `readseq` input capability** (like `readlines`) to hand a sequence handle to the program.
4. **Declared stack-effects / fault contracts / meter budgets** for each.
