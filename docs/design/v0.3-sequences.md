## 1. The problem, quantified

The interpreter-validated `T_v0` aggregate is **3.72√ó** (`bench/BASELINE.md`) ‚Äî comfortably over the ‚â•3√ó gate. But that corpus is recursion/control-heavy, which is exactly where MTL wins. The **tier-2 probe** (`bench/BASELINE-TIER2.md`, 10 solvable list/numeric tasks) was built to surface the *other* half of the story, and it sits at **1.91√ó (o200k) / 1.89√ó (cl100k)** ‚Äî barely over the gate and far below `T_v0`:

| encoding | sum(idiomatic Py) | sum(MTL v0.2) | aggregate |
|---|---:|---:|---:|
| o200k_base | 327 | 171 | **1.91√ó** |

Two structural gaps drag it down.

**(a) The stack-juggling tax.** Five of the ten solved tasks are "solved-but-ugly": they hand-roll a left-threaded accumulator that the language does not give them, costing **113 / 113 tokens** between them:

| task | v0.2 source | o200k | cl100k |
|---|---|---:|---:|
| max_list | `>_[>[;0][[]1]?][_][>_[^^<[~_][_]?]'][]\|` | 22 | 22 |
| min_list | `>_[>[;0][[]1]?][_][>_[^^<[_][~_]?]'][]\|` | 23 | 23 |
| reverse_list | `[]~[>[;0][[]1]?][_][>_[~;]'][]\|` | 19 | 19 |
| contains | `0~@[>[;0][[]1]?][__][>_[^=@~+0~<~]'][]\|` | 26 | 26 |
| count_occurrences | `0~@[>[;0][[]1]?][__][>_[^=@~+~]'][]\|` | 23 | 23 |
| **total** | | **113** | **113** |

`linrec`'s natural shape is a **right fold**, `combine(head, fold(tail))` with a *constant* init ‚Äî which is why the three *clean* folds (`sum_list [>0=][0][][+]|`, `product_list`, `length_list`) work. It has **no left-threaded accumulator slot** and **no place to carry an extra scalar**. max/min must seed the accumulator from the first element (there is no writable `-‚àû`); reverse must prepend into a growing accumulator; contains/count thread both an accumulator and the search scalar `x`. Lacking those slots, the v0.2 solutions emulate a left fold by hand over `linrec`: a rebuild-the-list non-destructive null-test `[>[;0][[]1]?]` (9 tokens, √ó5 ‚âà a ~45-token re-derivation of `empty?`) that forces the recursive branch to re-`uncons` with `>_`, a `dip` under the accumulator, and `^`/`@`/`~` routing to keep `x` reachable. That scaffolding ‚Äî not the arithmetic ‚Äî is the token cost, and it is the dominant LLM failure mode (blind spatial routing; the incident log in `bench/validate/TIER2_NOTES.md`; adversarial review ¬ß"Stack Juggling Tax").

**(b) Three walls.** Three tier-2 tasks are inexpressible in v0.2 and excluded from the aggregate (no MTL denominator):

| task | class | idiomatic Py (o200k/cl100k) | blocker |
|---|---|---:|---|
| single_number | bit-manipulation | 25 / 25 | canonical is XOR-reduce; **no bitwise primitive** (`^`=Over, `&`=PrimRec, `\|`=LinRec are all taken) |
| two_sum | indexed-search | 48 / 48 | returning **indices**; cons-list has only sequential `uncons`, no positional access |
| binary_search | indexed-search | 83 / 83 | needs **O(1) random access** to `xs[mid]`; cons-list is strictly sequential |

**Goal:** admit the minimal primitives that internalise the left-fold harness (removing the ~73-token juggling tax) and clear the walls that pay for themselves in corpus-level token accounting (spec ¬ß5/¬ß10.2/¬ß11) ‚Äî while honestly reporting which candidates do *not* pay, and which touch the value model too deeply to admit before the P2 refinement proof lands.

## 3. Small-step semantics (spec_step extension sketches)

Written to match `spec_step_prim` in `crates/mtl-core/src/mtl_core.rs`: dispatch on `p`, arity check first (`Fault(Underflow)`), then operand types (`Fault(TypeMismatch)`), then the semantic body ‚Äî the normative fault precedence (spec ¬ß4.4). Stack is `Seq<SpecValue>`, `n = stk.len() as int`, top at `stk[n-1]`. New `SpecPrim` variants: `Fold`, `Xor`.

### 3.1 `fold` ‚Äî `( seq init [C] -- result )`

`fold` is a **LEFT** fold: pop the combine quote `[C]`, the seed `init` (any value), and `seq` (must be a `Quote`). On the empty sequence the result is `init`. On `seq = [w | tl]`, compute `acc' = C(init, w)` ‚Äî run `C` with the stack `‚Ä¶ init w` on top ‚Äî then recurse `fold(tl, acc', C)`, left to right. Left (not right) because (a) it is the natural tail/iterative form for a stack machine ‚Äî constant stack, no unwind, matching `times`/tail-`linrec`; (b) it makes `reverse` a one-liner (`[][~;](`) that a right fold cannot; (c) operand order `( seq init [C] )` reads left-to-right ("fold THIS from INIT by C") and puts `seq` ‚Äî usually already on the stack as input ‚Äî at the bottom, so clean folds are just `init [C] (`.

`fold` is a **native** step rule that recurses by re-emitting itself, exactly like `primrec`/`times`. It does **not** desugar into `linrec`: `linrec` is a *right* fold with no left-threaded accumulator slot, so desugaring through it would reproduce the very hand-juggling `fold` exists to remove (see the desugar-verification note below). The head deconstruction is the `uncons` semantics inlined ‚Äî a head that is not itself a value (a bare `Prim`/`Call`) faults `TypeMismatch`, the same rule as `SpecPrim::Uncons` (mtl_core.rs). `fold` is **partial** like `!`/`linrec` (a non-terminating `C` would loop), but on any finite list with a straight-line `C` it terminates in `len(seq)` applications; the spine strictly shrinks each step. It performs no arithmetic, so there is **no semantic-fault arm** ‚Äî `Overflow`/`DivByZero` arise only inside `C`, under `C`'s own rules.

```rust
// fold ( [w0 w1 ...] init [C] -- r ): LEFT fold. init seeds the accumulator;
// C ( acc w -- acc' ) is applied once per element, left to right. The sequence
// is deconstructed head-first -- affine, like uncons (consumed once, never
// duplicated); C is replicated along the spine -- multiplicative, like C in
// primrec. Native recursion via re-emitting Fold (does NOT desugar into linrec).
// Uses the existing value_to_word helper (mtl_core.rs) to re-push `init`.
SpecPrim::Fold => {
    if n < 3 { SpecStep::Fault(Error::Underflow) }              // (1) arity  -> Underflow
    else {
        match (stk[n - 3], stk[n - 2], stk[n - 1]) {
            // seq must be a Quote; init is ANY value; combine must be a Quote.
            (SpecValue::Quote(qs), init, SpecValue::Quote(qc)) => {
                let base = stk.subrange(0, n - 3);
                if qs.len() == 0 {
                    // empty list: the result is the seed accumulator.
                    SpecStep::Next(SpecState { stack: base.push(init), cont: rest })
                } else {
                    let tail = qs.subrange(1, qs.len() as int);    // head-first (uncons reading)
                    match qs[0] {
                        // Continuation, run on `base`:
                        //   [tail]  init  <push head>   C   [C] fold
                        // -> park [tail] at the bottom, put `init head` on top,
                        //    run C ( acc w -- acc' ), then recurse fold(tail, acc', C).
                        SpecWord::PushInt(_) | SpecWord::PushQuote(_) => {
                            let recur = seq![
                                SpecWord::PushQuote(tail),
                                value_to_word(init),
                                qs[0]
                            ] + qc + seq![
                                SpecWord::PushQuote(qc),
                                SpecWord::Prim(SpecPrim::Fold)
                            ];
                            SpecStep::Next(SpecState { stack: base, cont: recur + rest })
                        }
                        _ => SpecStep::Fault(Error::TypeMismatch),  // (2b) non-value head
                    }
                }
            }
            _ => SpecStep::Fault(Error::TypeMismatch),              // (2a) seq/C not quotes
        }
    }
}
```

**Desugar-verification note (native, NOT linrec).** The step rule was mechanically checked at the *word* level (not just the value level) on `sum 0[+](` over `[3 1 2]`. State `[3 1 2] 0 [+]`, `Fold`: `qs=[3 1 2]`, `init=0`, `qc=[+]`, `base=[]`, `tail=[1 2]`, `qs[0]=PushInt(3)`, so `recur = PushQuote([1 2]), PushInt(0), PushInt(3), Add, PushQuote([+]), Fold`. Executing on `[]`: `[1 2]` ‚Üí `[1 2] 0` ‚Üí `[1 2] 0 3` ‚Üí `Add` ‚Üí `[1 2] 3` ‚Üí `[1 2] 3 [+]` ‚Üí `Fold` = `fold([1 2], 3, [+])`. Recurse ‚Üí `fold([2], 4, [+])` ‚Üí `fold([], 6, [+])` ‚Üí empty ‚áí push init ‚áí **6** = sum([3,1,2]). ‚úì Because the accumulator lives *below* the head on the stack while `C` runs and is re-emitted directly above `[tail]`, no per-call machine swap is needed; the only routing the *author* ever writes is when they choose to seed init from the list (max/min: `>_~`, a fixed 3-token prelude).

### 3.2 `xor` ‚Äî `( a b -- a^b )`

Pop two Ints, push their bitwise XOR. MTL ints are **i64** (`Value ::= Int(i64) | Quote`, spec ¬ß3), so XOR is defined on the **64-bit two's-complement bit pattern** ‚Äî exactly Rust's `i64 ^ i64`. Structurally identical to `Eq`/`Lt` (mtl_core.rs lines 267‚Äì294): **total**, arity ‚Üí type ordering only. Unlike `Add`/`Sub`/`Mul` (which compute in `int` then check `in_i64` for `Overflow`), the XOR of two in-range i64 values is *always* in i64 range, so there is **no `Overflow` arm and no `DivByZero` arm** ‚Äî the semantic layer is trivially total. New variant: `SpecPrim::Xor`.

```rust
// dispatch line, alongside SpecPrim::Add / Sub / Mul:
SpecPrim::Xor => {
    if n < 2 { SpecStep::Fault(Error::Underflow) }        // (1) arity -> Underflow
    else {
        match (stk[n - 2], stk[n - 1]) {
            (SpecValue::Int(a), SpecValue::Int(b)) =>
                // (3) semantic: total. Bitwise XOR on the i64 two's-complement
                // representation. a,b are in_i64 by the stack invariant, and
                // (a ^ b) is always in i64 range => no Overflow arm (cf. Eq/Lt).
                SpecStep::Next(SpecState {
                    stack: stk.subrange(0, n - 2).push(
                        SpecValue::Int(i64_bitxor(a, b))),   // ((a as i64) ^ (b as i64)) as int
                    cont: rest,
                }),
            _ => SpecStep::Fault(Error::TypeMismatch),        // (2) type -> TypeMismatch
        }
    }
}
```

The exec twin in `interp.rs` is literally `a ^ b` on `i64` (no `checked_*`, since it cannot overflow), mirroring the `exec_cmp` shape rather than `exec_arith`.

## 4. Multiplicity / affine story

The distinction that matters for the substructural reading (spec ¬ß14.4): does a primitive **consume its argument once** (affine) or **replicate it along the recursion spine** (multiplicative)?

| primitive | affine arg | linear arg | multiplicative arg | reading |
|---|---|---|---|---|
| `fold` | **`seq`** ‚Äî consumed once, deconstructed head-first | `init` threaded linearly (each `C` consumes the old acc, produces the new) | **`[C]`** replicated along the spine, applied once per element | first primitive that is **affine in one argument and multiplicative in another at once**; the affine spine-split matches `uncons`, the replicated `[C]` matches `primrec`. It *removes* the hand-written replication (`'`, `[>_‚Ä¶]` re-uncons, `[__]` base cleanup) the author writes today. |
| `xor` | ‚Äî | `a`, `b` both linear Int values, consumed once | ‚Äî | plain binary op, exactly like `+`/`=`/`<`: no quote arg, no replication, no affine subtlety. Touches only the value layer's Int case. |

Neither adds a *new* substructural wrinkle over `!`/`:!`. `fold` keeps the affine single-consumption of the list that makes `uncons` easy to reason about while internalising the replication the author currently performs by hand; `xor` is the simplest kind of primitive in the substructural sense (two linear Ints in, one out).

## 5. Glyph assignment (measured, spec ¬ß11)

Per ¬ß11 the glyph is chosen by rendering the real corpus and counting tokens **corpus-level, in context** under both pinned tokenizers ‚Äî BPE merges are context-dependent (`:!` may be 1 token while `:?` is 2), so this is measured, never chosen for looks. Full sweep in `bench/design-v0.3/glyphs/README.md`. The free ASCII set left after the 21 v0.2 primitives is `` #  $  (  )  \  `  {  } `` (the "obvious" bitwise glyphs `^`/`&`/`|` are all taken).

`fold` is the final word of every fold-rewritten solution and follows `]` in 6 of 8 (`[+](`, `[*](`, `‚Ä¶?](`, `[~;](`), so its load-bearing bigram is **`]G`**. `xor` sits inside `[$]` in single_number, so its load-bearing bigram is **`[G`**.

### 5.3 The joint optimum and the picks

Both primitives want overlapping merge-friendly sets but must be **different** characters. The total is separable across the two solution files, so **any (fold, xor) with fold ‚àà {`( { \ } )`}, xor ‚àà {`# $ ( { \ ` `}, fold ‚âÝ xor hits the joint optimum 56/54 (fold) + 9/9 (xor).** The choice within the optima is on secondary criteria:

- **`fold = (`** ‚Äî token-optimal (56/54), and among the five optimal chars it spends the one with least future value: it keeps **`{ }` in reserve** for a possible future array-literal syntax (indexed-access option (b) explicitly wants `{ }` as delimiters), and avoids **`\`**, which must be escaped (`\\`) inside JSON / double-quoted prompt strings ‚Äî a real robustness/generatability hazard for a language whose whole point is being emitted by LLMs into JSON contexts.
- **`xor = $`** ‚Äî canonical; `[$` is the merge that undercuts the arithmetic fold shape. Spending `$` on xor frees nothing better for fold (`]$` misses on o200k ‚Üí 60).

**Rejected alternatives (and why):** `fold = #` (the placeholder) costs **+4 tokens** on both encodings (`]#` is not a BPE token) ‚Äî retained only as the **generatability hedge** (¬ß11.8: `#` is a strong-prior character; if the agent trial shows `(` mis-generates, fall back to `#` at a measured headline cost of 3.87√ó/3.92√ó ‚Üí 3.71√ó/3.75√ó, still clearing 3√ó). `fold = $` is encoding-inconsistent and better spent on xor. `fold = \` is escape-hostile. `fold = {`/`}` burns a reserved pair-char. `xor = }`/`)` miss the `[G` merge (+1). Collision/self-delimiting check: `(` and `$` are both free, distinct, not delimiters, self-delimiting under maximal-munch, and never digit-adjacent in the corpus (no `01`-munch hazard). ‚úì

| primitive | glyph | key merge fact | corpus token cost |
|---|:--:|---|---|
| `fold` | **`(`** | `](` = 1 token in **both** encodings | **56 o200k / 54 cl100k** over 8 solutions (vs 60/58 for `#`) |
| `xor` | **`$`** | `[$` = 1 token in **both** encodings | **9 / 9** for single_number (vs 10/10 for `+`-fold) |

This is the coverage-vs-compression tradeoff quantified ‚Äî see ¬ß9. two_sum's ~34 estimate has a sensitivity band 30‚Äì38 tok ‚Üí o200k 3.06√ó‚Ä¶2.91√ó; the dilution conclusion is robust across the whole band.

## 7. Hand-traces (design stage ‚Äî NOT interpreter-validated)

Traced against ¬ß3. Stack shown bottom ‚Üí top. The input contract is unchanged from the frozen v0.2 solutions (the input list ‚Äî and, for contains/count, the scalar `x` ‚Äî is already on the stack, exactly as `bench/validate/tests/tier2.rs` prepends it).

**The 3 clean folds** (were clean under `linrec`; `fold` still wins):
- **sum_list** `0[+](`, list `[3 1 2]`: `[3 1 2] 0 [+] (` ‚Üí acc0=0; C(0,3)=3; C(3,1)=4; C(4,2)=6 ‚Üí **6** = sum. ‚úì hand-traced (re-derived through the ¬ß3.1 desugar).
- **product_list** `1[*](`, `[3 1 2]`: acc0=1; 1¬∑3=3; 3¬∑1=3; 3¬∑2=6 ‚Üí **6**. ‚úì
- **length_list** `0[_1+](`, `[3 1 2]`: C=`[_1+]` = `(acc w -- acc+1)` (drop `w`, push 1, add): acc0=0‚Üí1‚Üí2‚Üí3 ‚Üí **3**. ‚úì

**The 5 accumulator rewrites:**
- **max_list** `>_~[^^<[~_][_]?](`, `[3 1 2]`: `>_~` seeds acc from the first element and leaves `[tail] head` for `fold` ‚Äî `[3 1 2] > ‚Üí 3 [1 2] 1`, `_` ‚Üí `3 [1 2]`, `~` ‚Üí `[1 2] 3`. C = `[^^<[~_][_]?]` = `(acc w -- max acc w)` (the same max-combine the v0.2 solution used, reused verbatim). Fold over `[1 2]` from acc=3: C(3,1): `3<1=0` ‚Üí `[_]` keeps 3; C(3,2): `3<2=0` ‚Üí `[_]` keeps 3 ‚Üí **3** = max. ‚úì (empty list ‚áí `>_~` faults Underflow ‚Äî max of empty undefined, same behaviour class as v0.2.)
- **min_list** `>_~[^^<[_][~_]?](`, `[3 1 2]`: identical shape, branches swapped. seed 3; C(3,1): `3<1=0` ‚Üí `[~_]` keeps 1; C(1,2): `1<2=1` ‚Üí `[_]` keeps 1 ‚Üí **1** = min. ‚úì
- **reverse_list** `[][~;](`, `[3 1 2]`: init `[]`, C = `[~;]` = `(acc w -- w;acc)` (swap so value is under quote, `;` prepends). C(`[]`,3)=`[3]`; C(`[3]`,1)=`[1 3]`; C(`[1 3]`,2)=`[2 1 3]` ‚Üí **`[2 1 3]`** = reverse. ‚úì (A left fold that prepends yields the reversal ‚Äî exactly why `fold` is defined LEFT.)
- **contains** `[=+0~<];0~(`, `[3 1 2]` with `x=1`: the carried scalar `x` is closed into the combine quote via the existing `;` cons ‚Äî `[=+0~<]` push, `;` ‚Üí `[1 =+0~<]` (x baked in as first word), `0~` ‚Üí seq init=0. C = `(acc w -- acc OR (w==x))`: body on `acc w` runs `x = + 0~<` (`0~<` = OR, valid because both operands are non-negative flags). C(0,3)=0; C(0,1)=1; C(1,2)=1 ‚Üí **1** = contains. On `x=5` (absent): every eq=0 ‚Üí **0**. ‚úì
- **count_occurrences** `[=+];0~(`, `[3 1 2]` with `x=1`: same closure trick, combine keeps the running sum ‚Äî `[1 =+]`, seq init=0. C = `(acc w -- acc + (w==x))`: C(0,3)=0; C(0,1)=1; C(1,2)=1 ‚Üí **1**. On `[1 1 2]`, x=1: 0‚Üí1‚Üí2‚Üí2 ‚Üí **2**. ‚úì

**single_number** `[>0=][0][][$]|` ‚Äî trace on `[4 1 2 1 2]`: clean linrec xor-fold computes `4^1^2^1^2^0 = 4^(1^1)^(2^2)^0 = 4`. ‚úì hand-traced (n=5).

**two_sum** (indexed-access option (a), design estimate) ‚Äî `two_sum([2 7 11 15], 9) ‚Üí [0 1]`: brute-force O(n¬≤) point-free double walk, `nth` to read `xs[i]`/`xs[j]`, `;` to build the `[i j]` result on first match. n=4; i=0 (val 2): j=1 ‚Üí 2+7=9 == target ‚Üí emit `[0 1]` ‚Üí **`[0 1]`**. ‚úì hand-traced at the value level; the exact minimal golf of the four-deep carried state is a design estimate (~34 tok, band 30‚Äì38), marked **not measurable to the token** at design stage.

**binary_search** (indexed-access option (a), measured 37/37) ‚Äî `^#1-0~@[^^~<0=][@@___][@:^+2/:\@@^~<[1+~][~1-]?][+]|`, `binary_search([1 3 5 7 9], 7) ‚Üí 3`: carry `xs t lo hi`; `linrec` loops while `lo<=hi`; each iter `mid=(lo+hi)/2`, `x = xs nth mid`, branch. `^#1-` ‚Üí hi=4, `0~@` seeds lo=0. Probe 1: mid=(0+4)/2=2, xs[2]=5, 5<7 ‚Üí lo=3. Probe 2: mid=(3+4)/2=3, xs[3]=7, 7==7 ‚Üí return **3**. ‚úì hand-traced. **Complexity caveat:** `nth` (`\`) walks the cons-list from the head every probe ‚Üí O(n) per probe ‚Üí **O(n¬∑log n) overall; not a true binary search** (see ¬ß9). (Program re-measured here with `len=#`, `nth=\` so it does not collide with `xor=$`; 37/37, unchanged from the indexed worker's `$`/`#` version ‚Äî free single-char glyphs all cost ~1 token.)

## 8. Writability (E[tokens √ó attempts])

The review warns the most *compressible* alphabet may not be the most *generatable* (spec ¬ß10.6, ¬ß11.8). Per candidate:

- **`fold` ‚Äî strongest writability win, in the `primrec`/`times` tier.** It replaces the corpus's worst blind-routing idiom. The author writes a named, pattern-matchable shape ‚Äî "accumulate = seed + step", `init [step] (` ‚Äî and the step's only obligation is a shallow **2-item** local effect `( acc w -- acc' )`. It **eliminates** the whole v0.2 failure cluster: the rebuild-then-re-uncons null test (the incident-log Underflow bug), the `dip` under the accumulator, and the `^`/`@`/`~` routing to keep `x` reachable ‚Äî precisely the "stack-juggling tax"/"scratchpad deficit" the adversarial review names as the dominant LLM failure mode. Residual obligations are both shallow and nameable: max/min seed init from the first element (`>_~`, a fixed 3-token prelude); contains/count close `x` into the combine via `;` ("bake the carried value into the step"). `(` rides no strong learned prior (unlike `.`/`&`/`|`), but the shape `init[‚Ä¶](` is regular and self-delimiting; the generatability question is deferred to the agent trial (with `#` as the measured +4-token hedge, ¬ß5.3).
- **`xor` ‚Äî trivial, pure gain.** `[$` is a single named binary op the model already knows conceptually; it adds no quote arg, no depth, no new footgun. It removes a WALL rather than adding a failure mode.
- **`pick` / `roll` ‚Äî writability bet that goes negative here.** The claim (Gemini review) is they cut the stack-juggling tax for *deep* access. But on this corpus every body is ‚â§3 deep, and the rewrite *loses* tokens (contains 26‚Üí33, count 23‚Üí29, max_list 22‚Üí25). Worse, they **move** the counting burden rather than removing it: named `over`/`swap`/`rot`/`dup` have strong LLM priors, whereas `n pick` demands index arithmetic and a re-count on every use (position-dynamic). They trade spatial-routing errors for depth-miscount errors ‚Äî a wash-to-negative on this corpus. The one genuine win (non-destructive depth ‚â•3 access) is unexercised by any task. **Reject/defer** (¬ß9).
- **`nth`/`len` (indexed, opt a) ‚Äî modest, real, but tax-shrinking not tax-removing.** `#`/`len` removes re-deriving length via a fold; `nth` removes hand-threading an index counter through an `uncons` walk. But nested point-free loops with 3‚Äì4 carried values (two_sum's double loop, binary_search's `lo/hi/mid`) still demand deep blind stack routing ‚Äî the exact failure mode the review flags. Golfing a correct two_sum by hand at design stage was itself error-prone: a writability data point *against* these tasks regardless of the primitive. `nth`/`len` shrink the tax; they do not remove it.

## 9. Recommended admission set

**Admit `fold` (`(`) and `xor` (`$`).**
- `fold` on **token grounds**: ‚àí69/‚àí71 on the five ugly accumulators, ‚àí89/‚àí91 across all eight list folds, **no regressions**, and it removes the corpus's dominant LLM failure mode. It is the whole headline lever: {fold} alone takes tier-2 to 3.99√ó (o200k).
- `xor` because it **clears the single_number WALL and the whole bit-manipulation class** at trivial cost: total (arity‚Üítype only), zero value-model impact, one lock-step P2 pair, one `GLYPHS` row ‚Äî and, because `[$` merges, it costs *fewer* tokens than the arithmetic analogue.

**Headline verdict: {fold + xor} = 3.87√ó (o200k) / 3.92√ó (cl100k) over 11 tier-2 tasks ‚Äî clears the ‚â•3√ó gate decisively**, more than doubling v0.2's 1.91√ó/1.89√ó and landing on par with frozen `T_v0`'s 3.72√ó.

**Defer `empty?` / `len`** (the non-fold list observers). In isolation they are real savings (`empty?` 31 tok across the 5 ugly tasks, `len` 12 tok on length_list ‚Äî honestly *below* the briefing's "~45" once BPE-in-context merges are counted). But the exact tasks they help are the ones `fold` rewrites *wholesale* ‚Äî `fold` internalises traversal + empty-check + length ‚Äî so once `fold` lands, all ~43 tokens are captured by `fold`, not additive, and `empty?`/`len` marginal value ‚Üí ~0. They retain independent value only for an emptiness-branch or a length *without* a full fold, and no current corpus task needs that. (If ever needed, `empty?` should be a **non-consuming peek** `([xs] -- [xs] b)` ‚Äî the only drop-in for `linrec`'s save/restore-free predicate slot.)

**Reject / defer `pick` / `roll`.** They are a strict **token loss** on the solved corpus (contains 26‚Üí33, count 23‚Üí29, max_list 22‚Üí25) and writability-wash-to-negative at the shallow depths every task uses (¬ß8); the one real win (non-destructive depth ‚â•3 access) is unexercised by any corpus task. They also carry a **proof problem**: `n pick`/`n roll` have **data-dependent arity** (= d+2, depending on the runtime operand value), which breaks ¬ß4.4's "static arity checked before operands are inspected" invariant ‚Äî the precedence surface P1 relies on. Defer to the warm/cold agent trial, matching v0.2's disposition.

**Indexed access ‚Äî (c)-framing primary, optional (a) hedge, hold (b) until P2.** This is the architecturally significant call, so all three options are on the record (full analysis in `bench/design-v0.3/indexed-access/`):
- **(a) `nth`/`len` on the cons-list** ‚Äî O(n) access, **no value-model change**, "uncons-sized": two `SpecPrim` arms whose typed cases already carry `_ => TypeMismatch` (P1/P3 total by construction), a flagged `nth` (`x 1 | 0`, OOB is data not a fault ‚Üí **no new `Error` variant**), deep-view termination measure **untouched** (a list is still a `Quote`), two self-delimiting glyph rows for P4. Proof-safe even with P2 still open. binary_search = 37 tok, two_sum ~34.
- **(b) a `Value::Vec` value type** + `get`/`set`/`len` + `{ }` literals ‚Äî O(1) access, but it **reopens the file's hardest closed proof** (the deep-view lexicographic termination measure, ~55 `SpecValue` audit sites, a third mutual-recursion spine), adds a new `IndexOutOfBounds` `Error` variant (P1 precedence grows), new `{ }` literal delimiters (materially larger P4), and **breaks the affine story** (`get` copies, `set` persists ‚Äî not `uncons`'s clean "consumed once"). Built on today's `admit()`-stubbed P2 it maximizes rework. Weeks, not days. **Should land only AFTER P2 is discharged.**
- **(c) do nothing + document the niche** ‚Äî two_sum/binary_search are array/complexity-shaped; MTL wins on control-flow-shaped/recursive problems (`T_v0` 3.72√ó vs list-traversal 1.91√ó). binary_search is in fact **I/O-solvable today** as a linear index-of scan (it is a *complexity* wall, not an I/O wall), and two_sum's index return is reachable by carrying a counter during an `uncons` walk ‚Äî so the WALLs overstate strict impossibility.

**Recommendation:** adopt **(c)'s framing now** ‚Äî indexed random access is where imperative languages structurally win, the token evidence says unblocking these two tasks is **compression-neutral-to-dilutive** (scenario C drops the headline ~3.9√ó ‚Üí ~3.0√ó), so by the standing anti-tarpit rule (admit iff it pays for itself in corpus tokens) indexed access does **not** clear the bar the way `fold` does. **Do not chase two_sum/binary_search for the headline metric.** If benchmark *coverage* is wanted, ship **(a) only**, admitted on a coverage/writability rationale exactly as `uncons` was admitted on a non-token rationale ‚Äî and label binary_search honestly as an O(n¬∑log n) bisection scan, not a true binary search. **Hold (b) firmly until P2 lands.**

**Defer `and`/`or`/`shl`/`shr`:** block 0 corpus tasks; `and`/`or`'s only tangential use (boolean synthesis) is already covered by `+0~<` (4 tok); `shl`/`shr` additionally carry a shift-amount out-of-range decision and an arithmetic-vs-logical choice (real spec+proof commitments) for zero payoff. Each would burn one of just 8 scarce free ASCII chars on speculation.
