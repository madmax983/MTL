The largest remaining architectural risk: SEMANTIC TRIPLICATION. At least three primitive declarations (syntax::Prim, mtl_core::SpecPrim, interp::Prim) and three executable-ish semantic surfaces (Verus exec, stable Rust production, ghost spec). The parser glyph table is a single source of truth within the syntax crate but is not literally shared by the Verus artifact or production interpreter enum. This is the most likely source of future unsoundness-by-drift — the ordinary variety: a new primitive lands in production but not P4; a fault order changes in stable Rust but not Verus; a glyph assignment changes in syntax but not benchmark tooling; an enum arm is added to one mirror; Call changes representation; a helper becomes panic-capable while comments still say total. Differential tests do real work here, but the project would benefit from generated shared artifacts — e.g. a declarative primitive!{ Dup => ':', Drop => '_', ... } manifest generating syntax Prim, production Prim, glyph conversion, benchmark names, proof-side audit expectations. Verus may still require its own declaration, but you could generate and compare a checked manifest hash or generated file.

"Panic-free" is not literally true in the production interpreter: it says exec_step and run never panic but contains .expect("checked len >= 2"), .expect("non-empty checked above"), unreachable!("matched Call above"), unreachable!("checked Int below two quotes"), and other unreachable arms. Logically guarded, but in ordinary Rust they remain panic sites. Either make the strict claim true (remove all expect/unreachable, internal invariant fault or exhaustive ownership matches) or state the practical claim ("no panic reachable from well-formed VM states under the interpreter invariants; fuzzing and the verified twin validate those invariants"). At present "never panics" is stronger than what the stable Rust source alone establishes. The Verus exec twin is different: its impossible arms are PROVED unreachable — but that proof does not mechanically transfer to the cargo implementation.

prim_to_glyph has a suspicious "total fallback": .unwrap_or('\u{FFFD}') trades a crash for silent corruption — if GLYPHS ever misses a primitive the printer emits � which the parser rejects, and round-trip fails confusingly at runtime. Prim is closed: use an exhaustive const match (compile-time exhaustiveness), or keep GLYPHS but make coverage a compile-time/generated invariant. Returning a replacement character is the worst of both worlds.

pop_quote conceals invariant violations: on a wrong caller it silently converts the mistake into an empty quotation — a valid-looking but incorrect execution rather than a fault. For a formally motivated interpreter, silent invariant repair is dangerous: it produces divergence between the stable twin and formal model while keeping the machine moving. Use Option/Result, inline ownership matches, or an explicit panic if treated as a programmer invariant.

The host boundary is now well-factored: Invoke(name, stack, rest) is exactly the right split of verified core from trusted capabilities. Small cleanup: UnknownWord now appears unreachable through normal Call execution — remove it from the pure-core error enum or document which layer can still produce it; dead semantic variants become fossil beds of confusion.

A subtle claim about Dip: "the scoped borrow reading is literal" — the operational part is literal, but avoid calling it a Rust-style borrow until the multiplicity checker establishes alias conditions. Safe formulation: "Dip creates a syntactically scoped non-access interval for one stack occurrence." The stronger ownership interpretation belongs to P9.

P5's out-of-range-PC policy is consistent but should be branded: explicit Halt OR invalid PC ⇒ halted is now part of the theorem, not an implementation accident. Name it ("finite-code Minsky machine with implicit halt outside the code domain") so readers don't interpret p5_halt_reverse under a different machine definition.
