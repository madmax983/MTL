# External review: MTL P5 universality (2026-07-13)

This fixes the central honesty problem. Unary counters stored in quotation length are a legitimate source of unbounded storage in the abstract semantics, so the previous i64 objection no longer applies. The construction also looks internally coherent. I walked the stack choreography for: Inc(c1) and Inc(c2); zero and nonzero branches of both DecJz variants; the :! re-entry loop; dispatch preservation of pc; final halt cleanup. Assuming uncons has the implied effect, the stack transformations check out. Nicely done.

What this establishes: good executable validation that (1) counter magnitude is represented structurally rather than in i64; (2) the compiler handles all three Minsky instruction forms; (3) several concrete machines agree with an independent reference interpreter; (4) the real interpreter implements the intended quotation/self-application mechanics for these examples; (5) the construction survives repeated jumps and operations on both registers. That substantially upgrades P5 from "plausible sketch" to "credible proof target."

What it does not establish: the tests do not themselves prove universality. The load-bearing result remains the Verus theorem: for every valid finite Minsky program and represented configuration, one Minsky transition corresponds to a finite, bounded sequence of MTL steps that reaches the representation of the successor configuration. The executable suite is a very good transcription check but four terminating programs cannot establish the universal quantification. Rename "//! P5 empirical validation" -> "//! P5 executable validation".

The largest remaining wording problem: "// confirming the unbounded (length-encoded) counters behave past small values" - a finite test at 57 does not confirm unboundedness. The concrete interpreter uses Vec<Word>, bounded by usize/memory; universality is a property of the idealized semantics. Better: "// Confirms that counter magnitude is carried by quotation length rather than by the i64 payload, and exercises growth beyond the tiny smoke cases." And near the top: "//! In the abstract Verus model, counter quotations are unbounded Seqs. //! In this executable validation they are finite Vecs, bounded by the host machine as all concrete implementations necessarily are."

uncons is now load-bearing: pin its exact stack effect (nonempty: [q] -> head [tail] 1; empty: [] -> 0), result ordering, empty-case shape, malformed-quotation behavior, and whether it accepts any quotation. The sum-type encoding with branch-dependent stack shapes works dynamically but should be acknowledged in section 14.5 for the future stack-effect checker.

Real bug-shape: unchecked usize-as-i64 casts (int(j as i64), int(nz as i64), int(jz as i64), int(i as i64), Value::Int(pc0 as i64)) silently wrap above i64::MAX. Use i64::try_from + a validate_prog checking prog.len() <= i64::MAX, pc0 <= i64::MAX, every jump target fits and is in range. Make the theorem's len(prog) <= i64::MAX assumption explicit - it bounds only the instruction-address space.

Dispatch: linear comparison cascade is adequate for universality; record that one simulated Minsky step costs O(program length) MTL steps, and that recursive disp construction can overflow the Rust call stack - use an iterative reverse fold.

Out-of-range PC: both implementations treat pc >= prog.len() as halt - internally consistent, but choose and state a model. Preferred: only explicit HALT is valid; jump targets must be in range; validate at compile; make disp's fallback fault-producing rather than halt if MTL has a mechanism.

"Lock-step" is really bounded-stutter: m ->M m' implies encode(m) ->MTL+ encode(m') with a computable bound K(prog, pc, instr), likely linear in pc. Name it a stuttering/multi-step simulation.

Halting correspondence needs: forward simulation; MTL boundary preservation (representation invariant at every loop boundary); halt preservation; and reflection/no-spurious-halt (encode(m) ->MTL* Halt(s) => m is halted, under valid-PC assumptions). Without the last, you prove MTL can follow transitions but not that it cannot silently terminate early.

Strengthen tests cheaply: a generated differential layer (proptest over bounded machines), and especially boundary-at-a-time comparison via run_to_next_boundary verifying every instruction form across many counter values and jump targets, plus explicit invariant checks after each simulated step.

Revised verdict: valid proof route, provided uncons is part of the formal language and the Verus multi-step simulation theorem is actually discharged. Acceptable language once the Verus proof contains those universal lemmas: "MTL has an explicit universality construction based on unary quotation-encoded counters, exercised against the production interpreter and cross-checked with a reference Minsky evaluator; a Verus proof establishes the general multi-step simulation and halting correspondence." Until then: "MTL has a credible, executable universality construction whose general proof is in progress."
